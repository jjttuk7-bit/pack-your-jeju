## 2. 문제 정의

제주 여행자는 여행 전 블로그, 지도 리뷰, 공식 관광 포털, SNS를 오가며 장소를 고른다. 정보의 양은 충분하지만, 사용자가 실제로 판단해야 하는 것은 "어디가 유명한가"보다 "지금 이 정보를 믿어도 되는가"이다.

여행 정보가 흔들리는 대표 상황은 다음과 같다.

- 공식 포털에는 장소가 있지만 운영시간, 주소, 사진, 상세 정보가 수정 요청된 경우
- 리뷰에는 좋다고 쓰여 있지만 공공데이터 기준으로는 확인되지 않는 경우
- 가족, 부모님, 혼행 등 사용자 조건에 따라 주차, 정류장, 이동 가능성이 중요해지는 경우
- 오름, 바다 산책, 노을 감상처럼 날씨와 바람의 영향을 크게 받는 경우
- 여행 후 사용자가 실제로 확인한 내용이 다음 추천에 반영되지 않는 경우

따라서 이 프로젝트의 문제 정의는 다음과 같다.

> 제주 여행자가 과장된 후기나 단편적인 추천에 의존하지 않고, 공공데이터 근거와 사용자 조건을 함께 보고 신뢰 가능한 여행 계획을 만들 수 있어야 한다.

## 3. 핵심 사용자 시나리오

1. 사용자는 여행 지역, 기간, 동행자, 여행 목적, 원하는 순간을 선택한다.
2. 서비스는 선택 지역의 데이터 커버리지와 추천 가능성을 먼저 보여준다.
3. 사용자는 공공데이터로 확인된 후보, 주의 신호가 있는 후보, 데이터가 부족한 조합을 구분해서 본다.
4. 후보 카드에서 출처, 신뢰 점수, 확인 필요 항목, 날씨 영향, 이동 신호를 확인한다.
5. 마음에 드는 후보를 내 여행플랜에 담는다.
6. 플랜에 담긴 장소는 지도 마커, 맞춤 짐, 공유 문구, 방문 후 피드백 UI로 이어진다.
7. 여행 후 사용자는 방문함, 방문하지 않음, 정보 다름, 만족, 불만족 등의 피드백을 남긴다.
8. 이 피드백은 신뢰도 업데이트 시뮬레이션과 공공데이터 수정요청 큐로 분리되어 기록된다.

## 4. 해결 전략

### 4.1 추천보다 신뢰도를 먼저 보여준다

기존 여행 추천 서비스는 장소 리스트를 많이 보여주는 데 집중한다. 제주를 담다는 장소를 보여주기 전에 "이 후보를 어떤 근거로 믿을 수 있는지"를 먼저 보여준다.

핵심 UI는 다음과 같다.

- 지역 신뢰 대시보드
- 후보 카드별 신뢰 점수
- 확인 필요 항목
- 공공데이터 후보/주의 신호/데이터 부족 구분
- 플랜 담기 후 지도 마커
- 방문 피드백과 신뢰도 변화 카드

### 4.2 gpt-5-mini는 사실 생성기가 아니라 근거 조립기로 사용한다

모델은 장소명, 주소, 운영시간, 좌표 같은 사실을 만들지 않는다. 사실 정보는 DB와 공공데이터 조회 결과에서만 가져온다.

gpt-5-mini의 역할은 다음으로 제한한다.

- 조회된 근거를 사용자 문장으로 요약
- 여러 신뢰 신호를 하나의 설명으로 조립
- 리뷰 검증 화면에서 사용자 문장을 claim 단위로 나누고 판정 설명 생성
- 하루방 에이전트가 현재 플랜 상태를 설명하고 다음 행동을 안내

이 구조는 모델 성능을 높이는 동시에 환각 위험을 줄인다.

### 4.3 데이터가 부족하면 추천하지 않는다

검색 결과가 없을 때 "좋아 보이는 곳"을 생성하지 않는다. 대신 fallback을 분리한다.

| 분류 | 의미 | 사용자에게 보여주는 원칙 |
| --- | --- | --- |
| out_of_scope | 제주 여행 범위 밖 | 범위 밖이라고 설명 |
| coverage_gap | 참조 데이터 기준 확인 불가 | "저희가 참조하는 공공데이터 기준으로 확인되지 않습니다" |
| retrieval_miss | 검색 실패 가능성 | 내부 관측 및 완화 검색 |
| contradicted | 반증 또는 변경 신호 존재 | 변경 또는 폐업 가능성 표시 |

이 방식은 "모른다"를 서비스 실패가 아니라 신뢰 기능으로 전환한다.

## 6. 신뢰도 설계

각 장소 후보는 단일 배지만 갖는 것이 아니라 점수와 세부 근거를 함께 가진다.

주요 구성 요소:

- 공공데이터 일치도
- 사용자 조건 적합도
- 날씨 민감도
- 이동 가능성
- 운영 정보 안정성
- 방문 피드백 신호
- 최근성

UI에서는 이 정보를 다음 방식으로 노출한다.

- 신뢰 점수
- 신뢰 신호/주의 신호 배지
- 확인 필요 항목
- 근거 출처
- 방문 후 신뢰도 변화 예측

## 7. 서비스 구조

```text
사용자 입력
  -> 지역/기간/동행자/목적/순간 카드
  -> FastAPI /pack
  -> DB 검색 + Trust Engine + Weather Signal
  -> gpt-5-mini 요약/조립
  -> 후보 카드 + 신뢰 점수 + 확인 필요 항목
  -> 플랜 담기
  -> 지도 마커 + 맞춤 짐 + 방문 피드백
```

주요 모듈:

- `apps/api/engine/search.py`: 장소와 교통 신호 검색
- `apps/api/engine/trust.py`: 신뢰도, fallback, 배지 산출
- `apps/api/engine/weather.py`: 기상청 예보 요약
- `apps/api/engine/visit_signals.py`: 방문 피드백 신뢰도 변화
- `apps/web/src/components/PackingDashboard.tsx`: 추천 후보, 플랜, 지도, 피드백 UI
- `apps/web/src/components/TrustMapDashboard.tsx`: 지역 신뢰 지도
- `apps/web/src/components/VerifyPage.tsx`: 리뷰 검증

## 8. 서비스 견고성

서비스 배포 환경에서 다음 상황을 고려했다.

- OpenAI API 키가 없어도 템플릿 문구로 동작
- 외부 API 실패 시 확인 필요 문구로 명확히 표시
- 네이버 지도 인증 실패 시 간이 제주 지도와 좌표 마커 fallback 제공
- 프론트와 백엔드를 Vercel/Railway로 분리 배포
- 환경변수 예시는 제공하되 실제 키는 GitHub에 커밋하지 않음
- 공공데이터와 사용자 피드백을 UI에서 분리해 표시

## 9. 기대 효과

제주를 담다는 단순한 여행 추천 앱이 아니라, 여행 정보의 신뢰도를 사용자가 이해할 수 있게 만드는 서비스다.

기대 효과:

- 여행자가 과장된 후기보다 확인된 근거를 중심으로 계획을 세울 수 있다.
- 데이터가 부족한 영역을 숨기지 않아 서비스 신뢰가 높아진다.
- 방문 후 피드백을 통해 "다녀올수록 신뢰도가 자라는" 구조를 보여준다.
- 공공데이터의 수정 필요 지점을 서비스 안에서 발견할 수 있다.
