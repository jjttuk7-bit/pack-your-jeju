# 제주를 담다 20분 발표자료 구성안

## 발표 목표

20분 발표의 목표는 기능을 많이 보여주는 것이 아니라, 다음 세 가지를 심사위원에게 설득하는 것이다.

1. 제주 여행 정보 신뢰 문제를 명확히 정의했다.
2. gpt-5-mini를 RAG와 결합해 환각을 줄이고 신뢰도를 높였다.
3. 웹 서비스로 배포 가능한 수준의 사용자 흐름과 견고성을 갖췄다.

## 슬라이드 구성

### 1. 제목

- 제목: 제주를 담다
- 부제: 공공데이터와 gpt-5-mini RAG로 만드는 제주 여행 정보 신뢰도 분석 서비스
- 발표 메시지: "우리는 더 많은 추천보다, 믿을 수 있는 추천을 만들었습니다."
- 시간: 30초

### 2. 문제 제기: 제주 여행 정보는 많지만 믿기 어렵다

- 블로그, 리뷰, 지도, 공식 포털 정보가 흩어져 있음
- 여행자는 무엇이 최신인지, 무엇이 확인된 정보인지 판단하기 어려움
- 핵심 질문: "이 장소를 지금 내 일정에 넣어도 되는가?"
- 시간: 1분 30초

### 3. 실증 근거: 공식 데이터도 계속 수정된다

- 비짓제주 콘텐츠 수정요청 1,686건
- 운영시간/휴무, 주소/도로명, 메뉴/가격, 상세정보 수정 등 실제 변화 존재
- 메시지: 공식 데이터도 시간이 지나면 낡기 때문에 신뢰도 분석이 필요하다
- 시간: 1분

### 4. 서비스 한 줄 정의

- 제주를 담다는 지역, 기간, 동행자, 여행 순간을 입력하면 공공데이터 근거로 장소 후보를 검증하고 여행팩으로 조립하는 서비스
- 핵심 흐름:
  - 고르기
  - 근거 확인
  - 플랜에 담기
  - 방문 후 피드백
- 시간: 1분

### 5. 기존 GPT 추천과의 차이

- 일반 GPT 추천: 그럴듯한 장소와 설명을 생성할 수 있음
- 제주를 담다: DB에 있는 장소, 출처, 좌표, 수정요청, 날씨, 이동 신호만 사용
- 차별 문장: "모르는 것을 채우지 않고, 모른다고 표시합니다."
- 시간: 1분

### 6. 데이터 구조

- 비짓제주 장소 데이터
- 콘텐츠 수정요청 CSV
- 주차장/정류장 데이터
- 기상청 단기예보 조회서비스
- 사용자 방문 피드백
- 시간: 1분 30초

### 7. RAG와 모델 활용 설계

```text
사용자 조건
  -> 구조화 필터
  -> DB 검색
  -> Trust Engine
  -> gpt-5-mini 요약/조립
  -> 근거 기반 UI
```

- 모델은 장소명, 주소, 운영시간을 생성하지 않음
- 모델은 조회된 근거를 설명하고 비교하는 데 사용
- 시간: 2분

### 8. 신뢰도 산출 방식

- 공공데이터 확인
- 수정요청 이력
- 사용자 조건 적합성
- 날씨 민감도
- 이동 가능성
- 방문 피드백
- 최근성
- UI에는 점수, 배지, 확인 필요 항목으로 표시
- 시간: 1분 30초

### 9. 데모 1: 지역 신뢰 대시보드

시연:

- 제주 지역 지도에서 지역 선택
- 추천 가능한 순간과 데이터 부족 조합 확인
- 오른쪽 카드에서 실제 후보와 주의 신호 확인

강조:

- 추천 전 단계에서 데이터 커버리지를 먼저 보여준다.
- 시간: 1분 30초

### 10. 데모 2: 팩 만들기

시연:

- 지역, 날짜, 동행자, 목적, 순간 카드 선택
- 내 팩 생성
- 후보 카드의 신뢰 점수, 출처, 확인 필요 항목 확인

강조:

- 후보를 많이 보여주는 것이 아니라, 후보마다 "왜 확인이 필요한지"를 보여준다.
- 시간: 2분

### 11. 데모 3: 날씨와 이동 신호

시연:

- 여행 기간별 날씨 요약
- 오름/바다 산책 등 날씨 민감 활동의 확인 필요 항목
- 주차/정류장 근접 신호

강조:

- 여행 판단은 장소 정보만이 아니라 날짜와 이동 가능성까지 포함해야 한다.
- 시간: 1분

### 12. 데모 4: 플랜 담기와 지도 마커

시연:

- 후보를 내 여행플랜에 담기
- 지도 카드에서 플랜 장소 마커 확인
- 네이버 지도 인증 실패 시 fallback 설명

강조:

- 추천 후보가 실제 여행 계획으로 전환된다.
- 배포 환경의 실패 상황도 고려했다.
- 시간: 1분 30초

### 13. 데모 5: 방문 피드백과 신뢰도 업데이트

시연:

- 방문함, 정보 다름, 만족/불만족 선택
- 방문 후 피드백 입력
- 기존 신뢰도에서 업데이트 신뢰도로 변화 표시
- 공공데이터 수정요청 큐 설명

강조:

- "다녀올수록 신뢰도가 자란다"는 제안서 핵심을 UI로 보여준다.
- 시간: 1분 30초

### 14. 데모 6: 리뷰 검증

시연:

- 후기 문장 입력
- claim 분해
- 확인됨/주의/공공데이터 기준 미확인 판정

강조:

- 모델의 감이 아니라 원본 데이터와 대조해 판단한다.
- 시간: 1분

## Q&A 예상 질문

### Q1. 그냥 GPT에게 제주 여행 추천을 물어보는 것과 무엇이 다른가?

GPT는 그럴듯한 답을 생성할 수 있지만, 제주를 담다는 DB에서 조회된 장소와 출처만 사용한다. 모르는 장소는 생성하지 않고, 공공데이터 기준으로 확인되지 않는다고 표시한다.

### Q2. 공공데이터가 틀리면 어떻게 하나?

공공데이터도 완벽하지 않기 때문에 콘텐츠 수정요청 데이터와 사용자 방문 피드백을 별도 신호로 분리했다. 공공데이터 확인됨과 사용자 신고는 같은 배지로 섞지 않고, 수정요청 큐로 분리해 보여준다.

### Q3. gpt-5-mini는 어디에 쓰였나?

장소명과 주소 생성에는 쓰지 않았다. RAG 결과 요약, 리뷰 claim 분해, 검증 설명, 하루방 에이전트 브리핑처럼 조회된 근거를 사용자에게 이해 가능한 문장으로 조립하는 데 사용했다.

### Q4. 신뢰도 점수는 얼마나 객관적인가?

절대적 진실 점수가 아니라 의사결정 보조 점수다. 공공데이터 일치, 수정요청, 이동 가능성, 날씨, 사용자 조건, 방문 피드백을 분해해서 보여주기 때문에 사용자가 점수의 이유를 확인할 수 있다.

### Q5. API 키와 보안은 어떻게 처리했나?

OpenAI, KMA, Naver 지도 키는 모두 환경변수로 관리한다. GitHub에는 `.env.example`만 제공하고 실제 키는 Vercel/Railway Secrets에 저장한다.

### Q6. 외부 API가 실패하면 서비스가 죽나?

아니다. OpenAI 키가 없어도 템플릿 폴백으로 동작하고, 네이버 지도 인증 실패 시 간이 제주 지도 fallback을 보여준다. 날씨 API 실패 시에도 확인 필요 문구로 표시한다.

### Q7. 방문 피드백이 실제 공공데이터에 바로 반영되나?

바로 원본 공공데이터를 수정하지는 않는다. 서비스 내부에서 수정요청 큐로 분리하고, 향후 공공기관 제출 또는 관리 대시보드로 확장할 수 있는 구조로 설계했다.

### Q8. 데이터가 부족한 지역은 어떻게 처리하나?

추천으로 억지로 채우지 않는다. 지역 신뢰 대시보드와 후보 카드에서 데이터 부족 조합을 먼저 보여주고, 공공데이터 기준으로 확인되지 않는다고 명시한다.
