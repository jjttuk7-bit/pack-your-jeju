# 제주를 담다 공공데이터 라이프사이클 사업계획서 개정 설계

- 작성일: 2026-07-15
- 상태: 사용자 승인
- 제출 형태: 10페이지 이내 DOCX 및 PDF
- 신청 주체: 개인 1인 창업자

## 1. 개정 목적

기존 사업계획서는 근거 기반 여행 플랜과 하루방 리서치 에이전트의 사용자 가치를 중심으로 설명했다. 개정본은 여행 플랜을 최종 목적이 아니라 제주 공공데이터를 실제 이용·현장 검증·보정·확장으로 연결하는 라이프사이클의 한 단계로 재정의한다.

핵심 메시지는 다음과 같다.

> 제주를 담다는 공공데이터로 여행 플랜을 만드는 데서 끝나지 않고, 공공데이터·하루방 검색·사용자 직접입력으로 구성된 실제 여행 결과를 검증 가능한 현장 피드백으로 전환하여 제주 공공데이터의 신뢰성과 범위를 지속적으로 강화하는 데이터 라이프사이클 모델이다.

## 2. 선택한 서사

세 가지 대안 중 `공공데이터 라이프사이클 중심형`을 채택한다.

- 여행 서비스 중심형은 사용성을 설명하기 쉽지만 공공데이터 경진대회에서 데이터 환류의 독창성이 약해질 수 있다.
- 기관용 데이터 품질관리 플랫폼 중심형은 B2G 확장성이 강하지만 현재 MVP와 사용자 행동의 연결이 멀어 보일 수 있다.
- 라이프사이클 중심형은 현재 MVP의 여행 플랜 경험과 향후 공공데이터 품질·범위 확장을 하나의 순환으로 연결한다.

## 3. 핵심 데이터 라이프사이클

```text
제주 공공데이터 원본
  -> 정규화·변경 감지·신뢰 상태 계산
  -> 세 가지 경로의 여행 플랜
       1) 공공데이터 후보
       2) 하루방 웹검색 추천 후보
       3) 사용자 직접입력 후보
  -> 실제 여행과 장소 방문
  -> 구조화 피드백·선택 증빙
  -> 반복성·독립성·출처 일치·이상 제출 검토
  -> 운영자 승인·보류·기각
  -> 원본을 보존한 버전형 보정 및 신규 근거 데이터
  -> 다음 여행 플랜과 기관용 품질 개선 후보에 반영
```

여행 플랜은 공공데이터 활용 단계이며, 실제 여행과 피드백이 다시 데이터 품질을 높이는 환류 구조가 사업의 중심이다.

## 4. 세 가지 플랜 입력의 통합 원칙

| 경로 | 역할 | 보존할 근거 | 환류 가능성 |
|---|---|---|---|
| 공공데이터 | 공식 기본 후보와 장소 정체성 제공 | 데이터셋, 원본 ID, 수집·확인 시점 | 변경·오류·누락 후보 발견 |
| 하루방 검색 | 최신 공식·플랫폼·경험 원문으로 공백 보강 | 출처 URL, 출처 역할, 확인 시점, 선택 이유 | 공공데이터와 충돌·미수록 정보 발견 |
| 사용자 직접입력 | 개인이 실제 원하는 장소를 플랜에 포함 | 입력 원문, 주소·좌표, 중복·존재 확인 상태 | 검증 후 신규 지역 데이터 후보 축적 |

세 경로는 동일한 플랜 항목 계약으로 통합하되 출처를 섞거나 지우지 않는다. 당시의 장소명·주소·좌표·출처·확인 시점·선택 이유를 스냅샷으로 보존한다.

## 5. 피드백을 신뢰 데이터로 만드는 설계

사용자 피드백이 즉시 공공데이터 원본을 덮어쓰지 않도록 근거 원장과 검토 계층을 둔다.

1. 사실 변경과 경험 평가를 분리한다.
2. 실제 플랜 포함 여부와 방문 결과를 연결한다.
3. 위치·사진 등은 선택 증빙으로 사용하며 개인정보를 보호한다.
4. 로그인 계정, 정상 활동, 독립 제보, 공식·플랫폼 원문과의 일치, 이상 제출을 검토 우선순위에 반영한다.
5. 높은 가중치만으로 폐업·이전·주소 변경을 자동 확정하지 않는다.
6. 운영자가 승인·보류·기각하고 판정과 취소 이력을 버전으로 남긴다.
7. 승인된 보정만 다음 하루방 답변과 공공데이터 교차확인 근거로 사용한다.
8. 기관에는 원본 변경이 아니라 출처·확인 시점·판정 이력이 있는 보정 후보로 전달한다.

검증된 사용자 직접입력 장소는 공공데이터 미확인 상태를 유지한 채 `커뮤니티 확인 후보`로 축적할 수 있으며, 운영 품질이 확보된 뒤 공개 범위를 확대한다.

## 6. 현재 MVP와 향후 확장의 구분

### 현재 MVP

- 제주 공공데이터 기반 후보 탐색과 여행팩
- 하루방의 웹 원문 검색·비교와 후보 플랜 추가
- 사용자 직접입력 장소의 개인 플랜 추가
- 세 출처를 구분한 플랜 항목과 근거 스냅샷
- 방문 피드백·검토 흐름의 최소 API와 화면
- 신뢰 순환을 시연할 수 있는 배포형 웹 서비스

MVP는 전체 운영 체계를 완성한 상용 서비스가 아니라 세 가지 플랜 경로와 데이터 환류 가능성을 검증하기 위한 최소 구현이다.

### 고도화·확장 계획

- 회원·권한·장기 저장·기기 간 동기화
- 선택 증빙의 비공개 저장, EXIF 제거, 자동 가림
- 기여자 신뢰와 이상 제출 탐지
- 웹 재검증과 운영자 판정 자동화 지원
- 검증된 직접입력 장소의 커뮤니티 후보화
- 지역·주장 유형별 오류·누락·갱신 우선순위 대시보드
- 공공기관 전달 및 처리 결과 재수집
- 제주에서 검증한 데이터 라이프사이클의 타 지역 확장

## 7. 10페이지 구성

1. 아이템 정의와 핵심 제안
2. 문제 정의와 공공데이터 라이프사이클 필요성
3. 전체 순환 모델과 사용자 흐름
4. 세 가지 플랜 생성 경로와 통합 근거 구조
5. 제주 공공데이터 활용 현황과 가공 방식
6. 피드백 신뢰 축적·검토·버전형 보정 설계
7. 현재 MVP 구현 범위와 기술적 우수성
8. 단계별 고도화·확장 로드맵과 위험 대응
9. B2C 활용에서 B2B·B2G 데이터 품질 사업으로 확장하는 사업화 모델
10. 개인 1인 창업자의 실행 전략, 성과지표, 기대효과

## 8. 문서 디자인

- A4 세로, 10페이지 고정
- 본문은 9.5~10.5pt, 제목은 15~22pt로 계층화한다.
- 제주 감귤색을 강조색으로 사용하고 현무암색을 본문·제목에 사용한다.
- `현재 MVP`는 진한 청록 또는 감귤색 라벨, `고도화 계획`은 연한 청회색 라벨로 명확히 구분한다.
- 3페이지에는 전체 라이프사이클 순환 도식을 크게 배치한다.
- 4페이지에는 세 가지 플랜 입력이 하나의 근거 원장으로 합류하는 구조를 시각화한다.
- 6페이지에는 피드백이 자동 수정되지 않고 검증 계층을 거치는 흐름을 단계 도식으로 표현한다.
- 문장은 짧게 유지하고 표·강조 박스·숫자 카드를 사용해 심사자가 페이지별 핵심을 빠르게 파악하게 한다.
- 표는 한 페이지에 과밀하게 배치하지 않고 핵심 열만 남긴다.

## 9. 사실 표현 원칙

- 현재 시제품에서 확인 가능한 기능만 `현재 구현`으로 표시한다.
- 인증, 장기 저장, 증빙 보안, 기여자 신뢰, 기관 연계, 유료화는 `고도화·검증 계획`으로 표시한다.
- 팀 역량 표현은 사용하지 않고 개인 1인 창업자의 실행 범위·외부 협력 계획으로 작성한다.
- 공공데이터 원본 자동 수정, 기관 협력 확정, 매출 확정 등 검증되지 않은 내용을 단정하지 않는다.
- 기존 검증 수치와 데이터 스냅샷은 기준일과 근거를 함께 표시한다.

## 10. 성공 기준

- 첫 페이지부터 여행 추천이 아니라 공공데이터 라이프사이클 사업임이 드러난다.
- 세 가지 플랜 생성 경로가 모두 설명되고 출처가 보존되는 이유가 명확하다.
- 실제 여행과 피드백이 신뢰 가능한 보정·신규 데이터 후보로 전환되는 설계가 이해된다.
- 현재 MVP와 미래 확장이 혼동되지 않는다.
- 개인 1인 제출자의 실행 가능성과 단계적 확장성이 설득력 있게 표현된다.
- 최종 DOCX와 PDF가 모두 10페이지이며 표·도식·문단의 잘림이 없다.
