# 제주를 담다 사업계획서 기술 중심 전면 개정 설계

## 1. 개정 목적

주최 측 사업계획서 목차의 평가 순서에 맞춰 `1-1 개발 서비스의 기능 및 특징(우수성)`에서 실제 핵심 기술·기능·서비스·특징을 먼저 제시한다. 기존 문서의 공공데이터 라이프사이클 철학은 유지하되, 심사위원이 도입부에서 다음 질문에 바로 답을 얻도록 재구성한다.

- 무엇을 개발했는가?
- 어떤 실제 기술로 작동하는가?
- 기술이 어떤 기능과 사용자 가치를 만드는가?
- 현재 구현과 향후 고도화의 경계는 어디인가?
- 기존 서비스와 무엇이 다른가?

## 2. 작성 원칙

- 주최 측 공식 목차인 `1-1`, `1-2`, `1-3`, `2-1`, `2-2`를 그대로 사용한다.
- 기술명은 실제 구현 용어를 사용한다. 쉬운 별칭으로 대체하지 않는다.
- 각 기술은 `입력 → 처리 → 출력 → 사용자·공공데이터 효과 → 현재 구현 근거` 순서로 설명한다.
- 전문 용어는 유지하되 바로 뒤에 비개발 심사위원도 이해할 수 있는 설명을 붙인다.
- 현재 MVP와 고도화 계획을 명확히 구분하고 미완성 기능을 완료된 것처럼 쓰지 않는다.
- 공공데이터 라이프사이클은 독립된 구호가 아니라 기술·기능·사업화 전체를 잇는 중심 구조로 사용한다.
- 개인 신청자를 `대표자`라고 부르지 않는다. 문맥에 따라 `개인 창업자`, `신청자`, `개발자`를 사용한다.

## 3. 10페이지 구성

| 페이지 | 공식 항목 | 핵심 내용 |
|---|---|---|
| 1 | 표지·요약 | 아이템 정의, 기술·서비스 한 문장, 실제 기술 스택 요약, 현재 MVP |
| 2 | 1-1 | React PWA–FastAPI–검색·신뢰 엔진–PostgreSQL 근거 원장의 전체 아키텍처 |
| 3 | 1-1 | Python ETL·정규화, PostgreSQL 검색·장소 식별, 공간·기상 결합 |
| 4 | 1-1 | LLM Function Calling·RAG 하루방, Trust Engine, 근거 원장·버전 보정, 핵심 기능 흐름 |
| 5 | 1-2 | 데이터별 출처·내용·획득·정제·결합·서비스 활용 방식과 개발 스냅샷 |
| 6 | 1-2 | 공공데이터 라이프사이클과 지속 수집·검증·환류 설계 |
| 7 | 1-3 | 지도·일반 여행 AI·공공데이터 포털·후기·신고 시스템과의 차별성 |
| 8 | 2-1 | 개인 창업자의 동기, AI 엔지니어링 역량, 60개 이상 프로젝트, 직접 구현 범위와 협업 계획 |
| 9 | 2-2 | B2C→B2B→B2G 사업화, 시장 진입과 홍보·마케팅, 검증 지표 |
| 10 | 2-2 | 단계별 고도화 로드맵, 위험 대응, 성과지표, 기대효과와 결론 |

## 4. 1-1 실제 핵심 기술

### 4.1 Python 기반 공공데이터 ETL·정규화

- 비짓제주 관광정보·콘텐츠 수정요청·공영주차장·버스정류장·기상청 데이터 수집 파이프라인
- 장소명·주소·좌표·카테고리 정규화와 원본 ID·수집 시점 보존
- 체크섬 기반 변경 감지
- 좌표 거리 계산을 통한 주차장 1km·버스정류장 500m 접근성 결합

### 4.2 PostgreSQL 기반 검색·장소 식별

- 관계형 장소 데이터와 원본 출처 관리
- Full Text Search·GIN·`pg_trgm` 검색 인덱스
- 동일 장소·별칭·중복 후보 식별
- `pgvector`는 스키마에 준비된 의미 검색 고도화 기반으로 표시하고, 현재 핵심 검색처럼 과장하지 않는다.

### 4.3 LLM Function Calling·RAG 기반 하루방 에이전트

- 멀티턴 문맥에서 지역·일정·동행자·취향·제외 조건 해석
- 구조화 DB 조회와 웹 리서치 도구 호출
- 공식·플랫폼·경험 출처 분류와 원문 URL·확인 시점 보존
- 검색 실패 시 검색어 전환 재시도, `sufficient/partial/unavailable` 상태 구분
- 확인된 근거만 비교·요약하고 미확인 사실은 생성하지 않는 답변 계약

### 4.4 규칙 기반 Trust Engine

- 공공데이터 상태, 수정요청, 사용자 조건, 기상, 주차·정류장, 운영정보, 방문 피드백, 최신성을 분리 평가
- 점수뿐 아니라 세부 산출 근거, 신뢰 배지, 추가 확인 항목 제공
- 리뷰 문장의 주장 분해와 확인·충돌·미확인 판정

### 4.5 PostgreSQL 근거 원장·비파괴 버전 보정

- `PlanItem`, `Evidence`, `VisitFeedback`, `ModerationCase`, `PublicDataCorrection` 데이터 구조
- 공공데이터 후보·하루방 웹검색·사용자 직접입력의 출처 스냅샷 보존
- idempotency key를 통한 중복 피드백 방지
- 사용자 신호와 공공데이터 원본 분리
- 운영자 승인 결과를 원본 덮어쓰기 없이 버전형 보정으로 축적

### 4.6 React PWA·FastAPI 운영 구조

- React/Vite PWA, FastAPI, SQLAlchemy, PostgreSQL
- Vercel Web, Railway API·DB 배포
- 외부 API timeout·재시도·오류 분류, 환경변수 기반 키 관리
- 지도·날씨·LLM 장애 시 제한 상태와 fallback
- 골든셋 12/12와 백엔드 자동 테스트 229건

## 5. 1-2 공공데이터 활용 적정성

각 데이터는 `출처 → 원본 내용 → 획득 방법 → 정제·결합 기술 → 현재 기능 → 지속 활용`로 설명한다.

- 비짓제주 관광정보: 장소 기준 레코드와 여행 후보
- 콘텐츠 수정요청: 변경 유형과 재확인·갱신 우선순위
- 공영주차장: 좌표 정제와 1km 접근성
- 버스정류장: 중복 제거와 500m 접근성
- 기상청 예보: 날짜·권역·위험 신호와 대체 일정 판단

공공데이터 라이프사이클은 `수집 → 정규화 → 검색·플랜 → 실제 여행 → 구조화 피드백 → 웹 재검증·운영자 판정 → 버전 보정·신규 근거 → 다음 활용`로 표현한다.

## 6. 1-3 차별성

지도·여행 플랫폼, 일반 여행 AI, 공공데이터 포털, 후기·SNS, 단순 신고 시스템을 비교한다. 차별 기술은 다음 세 축으로 압축한다.

1. 공공데이터와 웹 정보를 역할별 근거로 결합
2. 여행 당시의 출처·확인 시점·선택 이유를 스냅샷으로 보존
3. 원본을 덮어쓰지 않는 검증·운영자 판정·버전 보정

## 7. 2-1 개인 창업자 역량

확정 문안의 사실 기준은 다음과 같다.

- AI 엔지니어링 정규 교육과정 이수 완료, 공식 수료 절차 전
- Python 데이터 처리, 머신러닝, LLM·RAG 응용 학습
- 60개 이상의 웹 서비스·프로토타입 저장소 구축 경험
- 기획, 데이터 처리, AI 기능, 웹·API, 배포 전 과정 반복 경험
- 제주를 담다의 공공데이터 파이프라인, 하루방, Trust Engine, 플랜·피드백 MVP 직접 설계·구현

외부 전문성은 개인정보·보안·법률, 제주 현장 검증, 로컬 파일럿, 기관 환류, 확장 단계 UX·영업·인프라에 단계적으로 확보한다.

## 8. 2-2 사업화

- B2C: 무료 여행팩·하루방·플랜·PWA → 공유 플랜·여행 저널·고급 비교·변경 알림 가설 검증
- B2B: 로컬 사업자 운영정보 불일치·출처 충돌·현장 확인 상태 리포트
- B2G: 반복 오류·미수록 장소·갱신 우선순위·보정 후보 대시보드
- 진입 순서: 조건형 여행 콘텐츠 → PWA 설치 → 플랜 공유 → 방문 피드백 → 로컬 파일럿 → 기관 PoC
- 가격·매출·기관 계약은 확정하지 않고 행동·업무효율 지표로 검증

## 9. 시각·독자 설계

- 한 페이지에 하나의 핵심 질문만 배치한다.
- 아키텍처는 계층형 흐름도, 데이터 처리는 입력·처리·출력 표, 차별성은 비교표, 사업화는 단계형 로드맵으로 표현한다.
- 기술명은 실제 용어를 사용하고, 그 바로 아래에 효과를 쉬운 문장으로 설명한다.
- `현재 MVP`와 `고도화 계획`을 색상과 라벨로 구분한다.
- 심사위원이 기술, 공공데이터 적정성, 차별성, 실행 역량, 사업성 질문에 문서만으로 답할 수 있는지 최종 점검한다.

## 10. 검증 기준

- PDF는 10페이지 이내, 목표는 정확히 10페이지다.
- 공식 목차 5개 항목이 모두 명시된다.
- 실제 기술 6개와 구현 근거가 포함된다.
- 초기 발표자료의 폐기된 `DB에 있는 장소만 사용`, 특정 모델 고정 표현은 최신 서비스 설명으로 갱신한다.
- 현재 MVP와 고도화 계획이 혼합되지 않는다.
- `대표자` 표현을 사용하지 않는다.
- DOCX와 PDF의 텍스트·링크·페이지 수를 자동 검증한다.
- 전 페이지 렌더링으로 잘림·겹침·과도한 밀도를 확인한다.
