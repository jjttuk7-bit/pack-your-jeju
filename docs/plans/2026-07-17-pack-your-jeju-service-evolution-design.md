# 제주를 담다 실서비스 고도화 기획서

> 작성일: 2026-07-17  
> 기획 범위: 제주 자유여행자 중심 B2C 공개 베타까지의 6개월  
> 운영 전제: 1인 창업자가 핵심을 직접 운영하고, 보안·법률·전문 디자인은 필요한 시점에 외부 협력

## 1. 기획 요약

제주를 담다는 추천 장소를 많이 나열하는 서비스가 아니다. 공공데이터, 공식·플랫폼·경험 출처, 하루방의 웹 리서치, 사용자 직접 선택을 하나의 여행 플랜으로 연결하고, 실제 방문 결과를 다시 검증 가능한 데이터로 축적하는 **근거 있는 제주여행 동반자**다.

실서비스 고도화의 중심은 다음 신뢰 순환을 실제 사용자와 반복하는 것이다.

```text
여행 조건
→ 하루방 조사와 후보 비교
→ 여행 플랜 저장
→ 여행 중 사용
→ 방문·정보 정확성 피드백
→ 반복 신호와 출처 재확인
→ 운영자 검토와 보정정보 승인
→ 다음 여행의 더 나은 근거
```

첫 6개월의 주 고객은 제주 자유여행자다. 핵심 성공지표는 단순 방문자나 질문 수가 아니라 **플랜 저장부터 실제 방문과 피드백 제출까지 완료된 신뢰 순환 여행 건수**로 정한다.

## 2. 현재 위치

### 2.1 이미 구현된 기반

- 제주 여행 조건 입력과 지역·순간 선택
- 공공데이터 기반 장소 후보와 지역 커버리지
- 공식·플랫폼·경험 출처를 조사하는 하루방 웹 리서치
- 출처 역할, 불확실성, 공공데이터 교차확인 표시
- 하루방 추천 후보를 사용자 플랜에 추가하는 흐름
- 여행 플랜 편집과 여행책 형태의 PDF 다운로드
- 방문 여부·정보 정확성·메모를 받는 피드백 화면
- 피드백 원장, 증빙, 검토 사례, 보정정보를 위한 백엔드 도메인
- React PWA, Vercel 프론트엔드, Railway FastAPI 구조
- Supabase JWT 인증 경계와 사용자별 플랜 API의 서버 기반

### 2.2 실서비스 전환의 공백

- 프론트엔드는 아직 실제 회원 로그인보다 시연용 접근 상태에 의존한다.
- 서버의 인증·플랜·피드백 기반이 사용자 화면 전체와 완전히 연결되지 않았다.
- 여행 플랜이 현장에서 쓰이는 오늘 일정·방문 완료·재확인 경험으로 이어져야 한다.
- 피드백을 운영자가 검토하고 승인된 보정정보를 다음 추천에 반영하는 운영 순환이 필요하다.
- 외부 검색과 모델의 지연·부분 실패·비용을 지속적으로 관측하고 제한할 장치가 필요하다.
- 개인정보, 위치, 사진 증빙, 계정 삭제에 대한 정책과 사용자 제어가 필요하다.
- 초기 JavaScript 번들 크기와 모바일 성능을 실서비스 기준으로 개선해야 한다.

## 3. 제품 전략

### 3.1 선택한 전략

세 가지 성장 전략을 비교했다.

1. 종합 제주여행 플랫폼: 시장은 크지만 지도·예약 대형 서비스와 직접 경쟁하고 1인 운영 범위를 넘는다.
2. 근거 있는 제주여행 동반자: 현재 기술과 데이터 순환을 가장 잘 활용하며 명확한 차별성을 만든다.
3. 여행자 참여 데이터 플랫폼: 장기 진입장벽은 강하지만 초기 사용자와 검수 체계가 없으면 작동하지 않는다.

따라서 **2번을 중심으로**, 종합 플랫폼의 필수 편의 기능과 참여형 데이터 축적을 단계적으로 결합한다.

### 3.2 포지셔닝

> 제주를 담다는 여행지를 대신 결정해 주는 서비스가 아니라, 출처가 있는 후보를 비교하고 실제 여행 결과를 다음 여행의 더 믿을 수 있는 근거로 돌려주는 제주여행 동반자다.

### 3.3 하지 않을 것

- 광고비를 낸 장소를 일반 추천처럼 우선 노출
- 공공데이터에 없다는 이유로 충분히 확인된 웹 후보 제거
- 사용자 한 명의 제보로 사실정보 즉시 변경
- 모든 여행 질문에 같은 검색 깊이와 모델 사용
- 첫 6개월 안에 전국 확장, 네이티브 앱, 대규모 커뮤니티 동시 추진
- 장기 위치 추적이나 불필요한 동행자 개인정보 수집

## 4. 핵심 사용자와 해결할 일

### 4.1 1차 사용자

- 제주를 처음 방문하는 자유여행자
- 혼자 1박 2일 또는 2박 3일 여행하는 사용자
- 부모님이나 아이와 함께해 실패 비용이 큰 사용자
- 렌터카 없이 이동 가능한 현실적인 동선이 필요한 사용자
- 비·바람·휴무 등 변동 조건에 맞는 대안이 필요한 사용자

### 4.2 사용자가 해결하려는 일

- 내 조건에 맞는 후보를 과도한 검색 없이 찾고 싶다.
- 왜 추천됐는지와 어떤 정보가 불확실한지 알고 싶다.
- 여러 후보를 실제 날짜별 플랜으로 정리하고 싶다.
- 여행 당일 운영 여부와 다음 행동을 빠르게 확인하고 싶다.
- 잘못된 정보를 쉽게 알리고 다음 여행자에게 도움이 되고 싶다.

## 5. 사용자 경험 구조

### 5.1 여행 전: 믿을 수 있는 플랜 만들기

- 대화 또는 폼으로 지역·기간·동행자·목적·하고 싶은 순간 정리
- 공공데이터·공식·플랫폼·경험 출처를 종합한 후보 조사
- 추천 이유, 출처, 확인 시점, 불확실성 표시
- 후보 비교와 플랜 추가
- 날짜별 일정과 이동 권역 정리
- 과도한 이동, 휴무, 날씨, 예약 조건 경고
- 플랜 저장·수정·공유
- 모바일 여행패스와 여행책 PDF 제공

### 5.2 여행 중: 실제 행동을 돕는 여행패스

- 오늘의 일정과 다음 장소
- 지도·전화·예약·공식 페이지 바로가기
- 방문 완료·건너뛰기·일정 변경
- 운영시간·날씨·휴무 재확인
- 현장에서 발견한 장소를 내 플랜에 추가
- 핵심 일정 오프라인 확인
- 정보 불일치 간편 제보

서비스 안에 사용자를 가두기보다 지도, 전화, 예약 등 실제 행동으로 빠르게 연결한다.

### 5.3 여행 후: 10~20초 피드백

- 실제 방문 여부
- 정보가 정확했는지
- 운영시간·주소·주차·휴무 등 무엇이 달랐는지
- 동행자와 여행 목적에 잘 맞았는지
- 다시 추천할 의향
- 선택적인 한 줄 메모와 비공개 사진 증빙

### 5.4 사용자 화면

1. 여행 만들기
2. 후보 비교
3. 내 여행
4. 여행 기록
5. 마이페이지

운영자에게만 피드백과 보정정보를 처리하는 **신뢰 운영실**을 제공한다.

## 6. 데이터·AI·Trust Engine

### 6.1 핵심 데이터 객체

| 객체 | 역할 |
|---|---|
| `Place` | 출처마다 다른 장소명·주소·좌표를 하나의 장소로 통합 |
| `SourceSnapshot` | URL, 출처 역할, 확인 시점, 원문 스냅샷 보존 |
| `Claim` | 운영시간·휴무·가격 등 확인할 사실 단위 |
| `Evidence` | 특정 주장을 뒷받침하거나 반박하는 근거 |
| `PlanItem` | 사용자가 선택한 장소와 선택 당시 근거 |
| `VisitFeedback` | 실제 방문 여부와 정보 정확성 |
| `ModerationCase` | 반복 제보와 출처 충돌의 운영 검토 단위 |
| `PublicDataCorrection` | 승인된 보정정보와 버전 이력 |

### 6.2 장소 통합

장소명 유사도, 정규화 주소, 좌표 거리, 전화번호, 공식 URL, 시설 식별값을 결합한다. 자동 연결이 불확실하면 합치지 않고 `동일 장소 추정` 상태로 보존한다.

### 6.3 하루방 조사 모드

| 모드 | 예시 | 처리 원칙 | 목표 응답시간 |
|---|---|---|---:|
| 빠른 확인 | “오늘 열어?” | 공식·플랫폼 우선, 직접 답변 | 3~8초 |
| 비교 조사 | “부모님과 갈 맛집 세 곳” | 출처 병렬 조사와 후보 비교 | 8~15초 |
| 플랜 설계 | “2박 3일 가족여행” | 조건·동선·날씨·대안 종합 | 진행 단계 표시 |

### 6.4 속도와 비용

- 규칙과 DB로 처리할 수 있는 작업에는 LLM을 사용하지 않는다.
- 빠른 모델은 질문 해석·검색계획·형식 정리에 사용한다.
- 복합 일정과 출처 충돌 판단에만 고성능 모델을 사용한다.
- 검색은 출처 유형별로 병렬 실행한다.
- 느린 출처 하나가 전체 답변을 막지 않도록 부분 성공을 허용한다.
- 같은 질문·여행 조건의 성공 결과와 유효한 출처 스냅샷을 재사용한다.
- 검색 공급자별 timeout, 제한된 재시도, 오류 분류를 둔다.
- 질문당 비용과 일일 한도를 측정하고 초과 시 보호 모드로 전환한다.

### 6.5 신뢰 판단

하나의 점수만 보여주지 않고 다음 요소를 분리해 평가한다.

- 출처의 역할과 권위
- 확인 날짜와 최신성
- 독립 출처의 일치 정도
- 장소 식별 정확도
- 공공데이터와의 일치 여부
- 실제 방문 피드백의 반복성
- 운영자 검토 여부
- 정보의 변동 가능성

사용자 표시 상태는 `확인 높음`, `대체로 확인`, `재확인 필요`, `변경 의심`, `보정 승인`으로 단순화하고 이유를 함께 제시한다.

### 6.6 피드백 검증

```text
피드백 접수
→ 중복·스팸·개인정보 위험 확인
→ 장소와 주장 연결
→ 독립 사용자 반복 신호 결합
→ 공식·플랫폼·공공데이터 재확인
→ 운영자 검토
→ 보정 승인·반려·관찰·만료
→ 다음 추천 반영
```

공공데이터 원본은 덮어쓰지 않는다. 승인된 보정정보는 근거, 판단자, 판단 시점, 이전 값과 함께 별도 버전으로 보존한다.

## 7. 회원·보안·운영

### 7.1 회원 경계

| 기능 | 비회원 | 회원 | 운영자 |
|---|---:|---:|---:|
| 하루방 상담·임시 플랜 | 가능 | 가능 | 가능 |
| 기본 PDF 다운로드 | 가능 | 가능 | 가능 |
| 장기 저장·기기 동기화 | 제한 | 가능 | 가능 |
| 공유 링크 | 제한 | 가능 | 가능 |
| 방문 피드백·증빙 | 불가 | 가능 | 가능 |
| 보정정보 검토 | 불가 | 불가 | 가능 |

초기 인증은 이메일 또는 소셜 간편 로그인을 사용한다. 비회원 임시 플랜은 로그인 후 계정으로 안전하게 이전한다.

### 7.2 개인정보와 증빙

- 계정 식별값, 저장 플랜, 사용자가 제출한 피드백만 최소 수집
- 동행자 실명·생년월일·상세 주소는 수집하지 않음
- 위치는 선택적으로 일시 요청하고 원본보다 거리 판정 결과를 저장
- 증빙 사진은 기본 비공개, 메타데이터 제거, 형식·크기 검사
- 공개 활용은 별도 동의
- 계정·플랜·피드백·증빙 삭제 기능 제공

### 7.3 1인 운영 자동화

시스템이 중복 신호를 묶고 출처를 재확인해 긴급도가 높은 사례만 운영자에게 올린다.

1. 폐업·이전·접근 불가
2. 운영시간·휴무·예약 조건 반복 불일치
3. 주소·주차·대중교통
4. 분위기·혼잡도·동행자 적합성
5. 표현 수정과 낮은 영향도 제안

모든 승인·반려 작업에는 판단자, 시점, 근거, 변경 전후, 사유를 기록한다.

### 7.4 인프라 역할

| 영역 | 역할 |
|---|---|
| Vercel | React PWA, 정적 자산, 사용자 화면 |
| Railway | FastAPI, 하루방, Trust Engine, PDF, 장시간 작업 |
| Supabase Auth | 회원가입·로그인·토큰 |
| Supabase PostgreSQL | 플랜·피드백·근거·보정 이력 |
| Supabase Storage | 비공개 사진·증빙 |
| 외부 검색·LLM | 웹 조사·검색계획·답변 조립 |
| 모니터링 | 오류·속도·비용·외부 공급자 장애 |

트래픽이 실제로 증가하기 전에는 검색 워커나 별도 서버를 추가하지 않는다.

## 8. 수익모델

### 8.1 1단계: 무료 B2C 검증

첫 3~6개월은 하루방 기본 조사, 플랜 저장, 모바일 여행패스, 기본 PDF, 방문 피드백을 무료로 제공한다.

### 8.2 2단계: 여행 단위 프리미엄

- 복합 일정 정밀 설계
- 여러 플랜 비교와 버전 저장
- 공동 편집
- 고급 여행책 PDF
- 출발 전 운영정보 일괄 재확인
- 여행 중 일정 재구성

초기 가격 가설은 여행 1건 3,900~6,900원 또는 제주여행 패스 7,900~12,900원이다. 월 구독은 재방문 빈도를 확인한 뒤 판단한다.

### 8.3 3단계: 예약·이용 제휴

공식 예약 페이지와 체험·입장권·교통 등 실제 행동을 연결한다. 제휴 여부를 명확히 표시하고 추천 순서와 분리한다.

### 8.4 4단계: B2G·B2B 데이터 서비스

- 공공 관광데이터와 현장 신호 비교
- 변경 의심 장소와 반복 오류
- 지역·분류별 최신성 대시보드
- 보정 후보와 근거 자료
- 사업장 공식정보 확인·변경 요청
- 실제 여행에서 발생한 정보 불일치 분석

## 9. 시장 진입

### 9.1 초기 집중 권역

- 제주시·공항 인근
- 애월·한림
- 성산·구좌
- 서귀포 도심

### 9.2 초기 집중 상황

- 제주 첫 방문
- 혼자 떠나는 1박 2일·2박 3일
- 부모님과 함께하는 여행
- 비 오는 날의 대안 일정
- 렌터카 없는 여행

### 9.3 첫 사용자 확보

- 검색 가능한 여행 문제 해결 콘텐츠
- 1차 30명, 2차 50명의 참여형 베타
- 제주 게스트하우스·소규모 숙소·여행작가와 문제 중심 제휴
- 피드백 완료 사용자에게 프리미엄 여행책 제공
- 보정 결과가 반영되면 기여자에게 알림

## 10. 핵심 지표

### 10.1 북극성 지표

**플랜 저장 → 실제 방문 → 피드백 제출까지 완료된 신뢰 순환 여행 건수**

### 10.2 제품 퍼널

- 방문자 대비 하루방 첫 질문 비율
- 질문 사용자 대비 후보 추가율
- 후보 추가 사용자 대비 플랜 저장률
- 저장 플랜 대비 여행패스 사용률
- 여행 시작 플랜 대비 피드백 완료율
- 공유 플랜을 통한 신규 사용자 비율

### 10.3 신뢰·운영 지표

- 출처가 연결된 주요 주장 비율
- 검색 성공·부분 성공·timeout 비율
- 정보 불일치 발생률
- 반복 변경 신호 수
- 보정 승인까지 걸린 시간
- 동일 조건 답변의 품질 편차
- 질문당 검색·모델 비용

### 10.4 6개월 검증 목표

- 실제 회원 300~500명
- 저장 플랜 150건 이상
- 실제 여행이 시작된 플랜 80건 이상
- 신뢰 순환 여행 40건 이상
- 장소별 현장 피드백 200건 이상
- 반복 변경 신호 20건 이상
- 공공기관 또는 관광사업자 실증 논의 1~2건
- 하루방 핵심 질문 중앙 응답시간 10초 안팎

수치는 확정 매출 목표가 아니라 제품과 신뢰 순환을 검증하기 위한 초기 가설이다.

## 14. 1인 운영 원칙과 의사결정 게이트

- 매달 하나의 핵심 사용자 흐름만 완성한다.
- 기능은 실제 사용자 행동과 측정 지표를 함께 배포한다.
- 자동화 전에 판단 기준을 문서화한다.
- 사용자가 쓰지 않는 기능은 고도화하지 않는다.
- 데이터와 보정 판단은 근거와 버전을 남기고 되돌릴 수 있어야 한다.
- 보안·법률·전문 디자인은 위험이 커지는 시점에 외부 검토를 받는다.

각 단계는 종료 조건을 충족해야 다음 단계로 넘어간다. 핵심 지표가 나오지 않으면 새 기능보다 해당 퍼널의 문제를 먼저 해결한다.

## 15. 최종 판단

제주를 담다의 실서비스 경쟁력은 AI 추천 자체가 아니다. **여행자의 선택과 실제 현장 결과를 근거 원장으로 연결하고, 검증된 보정정보를 다음 여행에 되돌리는 구조**가 경쟁력이다.

첫 6개월에는 제주 자유여행자의 실제 사용 경험을 완성하고 신뢰 순환을 40건 이상 축적한다. 이 데이터로 B2C 프리미엄 가능성과 B2G 데이터 품질 실증 가능성을 함께 검증한다.
