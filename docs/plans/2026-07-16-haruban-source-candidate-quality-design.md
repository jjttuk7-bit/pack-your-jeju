# 하루방 출처·후보 품질 보강 설계

## 운영 관찰

GPT-4o 웹검색 분리 후 동일 질문은 약 19초에 성공했지만 다음 문제가 확인됐다.

- 티스토리 블로그가 `experience`가 아닌 일반 `web` 출처로 분류됐다.
- 후보 추출기가 답변 전체의 굵은 글자를 훑어 `방문 팁`의 가격 문구를 장소로 오인했다.
- 후기 출처를 근거로 1인분 가능, 예약, 평균 가격 같은 변동·운영 사실을 강하게 표현했다.

## 검토한 접근

### 1. 결정론적 파서·분류기·프롬프트 보강 — 채택

추가 모델 호출 없이 `추천 장소` 섹션만 파싱하고, 가격·일반 문구를 방어적으로 제외한다. 블로그 도메인을 후기 출처로 분류하고 출처 역할 금지선을 검색 프롬프트에 명시한다. 현재 약 19초 응답속도를 거의 유지할 수 있다.

### 2. 두 번째 LLM 검증 호출

후보와 주장을 다시 검사할 수 있지만 API 호출과 지연시간이 늘어난다. 현재 사용자의 속도 우선 요구와 맞지 않는다.

### 3. 공식·플랫폼 출처가 없으면 답변 전체 거부

운영 사실에는 안전하지만 제주 소규모 식당처럼 공식 웹사이트가 없는 장소를 지나치게 많이 제외할 수 있다.

## 설계

### 후보 추출 범위

- Markdown의 `## 추천 장소` 섹션부터 다음 `##` 제목 전까지만 파싱한다.
- 섹션이 없으면 후보를 만들지 않는다.
- 장소 블록에는 `특징`, `추천 이유`, `위치`, `주소` 중 하나 이상의 구조화 필드와 출처 URL이 있어야 한다.
- 금액·거리·시간·수량처럼 명백한 수치 문구는 장소명으로 허용하지 않는다.
- 원 답변 텍스트는 그대로 보여주되 플랜에 담을 구조화 후보만 보수적으로 제한한다.

### 출처 분류

- `tistory.com`, `blog.naver.com`, `brunch.co.kr`, `medium.com` 등 개인 게시 플랫폼은 `experience`로 분류한다.
- 공식·지도·예약 플랫폼 우선순위는 유지한다.
- 미분류 출처는 임의로 공식이나 플랫폼으로 승격하지 않는다.

### 검색 프롬프트

- 장소 제목의 굵은 글자에는 정확한 상호명·고유명만 쓰도록 한다.
- 가격, 설명 문구, 추천 수식어를 장소명처럼 굵게 쓰지 않도록 한다.
- 운영시간·휴무·가격·예약·1인 주문 가능 여부는 공식 또는 지도·예약 플랫폼에서 직접 확인된 경우에만 단정한다.
- 블로그·후기는 분위기와 방문 경험에만 사용한다.
- 공식·플랫폼 근거가 없으면 “후기 기반 참고 후보”로 범위를 낮추고 운영 사실은 재확인 필요로 표시한다.

### 최종 답변 출처 가드레일

- 최종 GPT 답변 직전에 실제 `source_class` 구성을 계산한다.
- 후기 또는 미분류 웹 출처가 포함됐는데 “모두 공식·플랫폼에서 확인”이라고 표현한 문장은 제거한다.
- 답변 앞에 공식·플랫폼·후기·기타 웹 출처 건수를 표시하고 운영 정보 재확인 필요를 명시한다.
- 이 검증은 문자열과 구조화 메타데이터로 처리해 추가 모델 호출과 지연을 만들지 않는다.

## 테스트

- 티스토리와 네이버 블로그가 `experience`로 분류되는지 검증한다.
- `방문 팁`의 가격 문구가 후보에서 제외되는지 검증한다.
- `추천 장소` 섹션 밖의 굵은 글자가 후보로 생성되지 않는지 검증한다.
- 구조화 필드 없는 굵은 문구가 후보로 생성되지 않는지 검증한다.
- 검색 프롬프트에 출처 역할과 정확한 장소명 규칙이 포함되는지 검증한다.

## 성공 기준

- 가격·일반 문구가 `place_candidates`에 포함되지 않는다.
- 블로그 출처가 `experience`로 표시된다.
- 운영 사실을 후기 하나로 확정하도록 유도하지 않는다.
- 추가 OpenAI 호출 없이 기존 응답시간 수준을 유지한다.
