# 하루방 지역 개요 답변 품질 설계

## 문제

`안덕은 어떤 곳이야?`처럼 인식 가능한 제주 지역의 성격을 묻는 질문이
`어떤 곳`이라는 넓은 정규식 때문에 일반 여행 계획 조언으로 분류된다.
그 결과 선택된 지역과 질문을 무시하고 제주 전역의 일정 구성법을 답한다.

웹 리서치 답변에서는 혼합 출처 안전 문구가 답변 맨 앞에 강제로 붙어
직접 결론보다 내부 품질 보고가 먼저 보이는 문제도 있다.

## 목표

- 인식된 제주 지역의 특징·성격을 묻는 질문은 웹 리서치로 보낸다.
- 일반적인 일정 수립 방법과 준비 팁은 검색 없는 조언 경로에 유지한다.
- 혼합 출처 안내는 본문을 가리지 않도록 답변 뒤에 배치한다.
- 현재 선택 지역과 순간을 웹 검색 컨텍스트에 그대로 전달한다.

## 접근안 비교

### 1. 프롬프트만 강화

일반 조언 프롬프트에 지역 질문을 우선 답하라고 추가한다.
변경은 작지만 잘못된 검색 생략 상태가 유지되고 출처 없는 지역 사실을 만들 위험이 있다.

### 2. 지역 개요 의도 전용 판별

인식 가능한 지역과 `어떤 곳`, `지역 특징`, `분위기`, `성격` 표현이 함께 있으면
지역 개요 질문으로 판별해 웹 리서치에 우선 연결한다. 일반 조언 정규식에서는
모호한 `어떤 곳` 단독 조건을 제거한다.

이 접근은 현재 신뢰 엔진의 웹 우선 원칙과 일치하고 회귀 테스트로 경계를 명확히 고정할 수 있다.

### 3. 모든 `어떤 곳` 질문을 웹 검색

구현은 간단하지만 `여행 계획은 어떤 방식이 좋아?` 같은 일반 조언까지 검색할 수 있어 과도하다.

## 선택

2번 접근을 사용한다.

## 데이터 흐름

```text
사용자 질문
  -> 지역 ID 인식
  -> 지역 개요 표현 판별
  -> web_search_jeju
  -> 선택 폼 컨텍스트 포함
  -> 결론과 지역 설명
  -> 답변 끝의 짧은 출처 범위 안내
```

## 답변 순서

웹 답변은 질문에 대한 결론과 근거 본문을 먼저 표시한다.
후기·기타 웹 출처가 포함된 경우에만 마지막에 `확인 범위` 안내를 덧붙인다.
모든 출처가 공식·플랫폼인 경우에는 추가 경고를 붙이지 않는다.

## 회귀 테스트

- `안덕은 어떤 곳이야?`는 웹 리서치로 라우팅한다.
- 검색 컨텍스트에는 선택된 `jocheon`, `andeok`과 선택 순간이 유지된다.
- `제주시는 어떤 곳을 가보면 좋은지 알려줘`도 추천성 질문이므로 웹 리서치로 라우팅한다.
- `제주는 처음인데 여행 계획은 어떻게 세워?`는 일반 조언 경로를 유지한다.
- 혼합 출처 안내는 직접 결론 뒤에 배치된다.

## 범위 밖

- 검색 공급자나 모델 교체
- 프론트 채팅 UI 재설계
- 지역별 고정 설명 템플릿 추가
- 공공데이터를 웹 검색 대신 사용하는 fallback
