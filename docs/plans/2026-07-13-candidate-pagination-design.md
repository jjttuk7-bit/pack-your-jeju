# 후보 탐색 페이지네이션 설계

## 문제

현재 순간별 후보는 검색 엔진의 `DEFAULT_LIMIT = 5`에서 잘린다. 이 다섯 곳은 대표 후보로 선정된 것이 아니며 같은 조건에서는 동일한 결과가 반복된다. 프론트에는 전체 후보 수와 추가 탐색 수단이 없어 사용자가 데이터베이스의 다른 후보에 접근할 수 없다.

## 목표

- 최초 화면은 비교 가능한 5곳을 유지한다.
- 순간별 전체 후보 수와 현재 표시 수를 공개한다.
- 사용자가 중복 없이 다음 5곳을 계속 불러올 수 있게 한다.
- 기존 `/pack`, PDF, 요일별 일정과 하루방 호출은 하위 호환을 유지한다.
- 이번 단계에서는 후보 순위 알고리즘 자체를 바꾸지 않는다.

## 선택한 구조

`POST /pack/candidates`를 추가한다. 요청은 기존 여행 조건과 단일 `moment`, 선택적인 불투명 `cursor`를 받는다. 서버는 정확 지역 후보를 우선하고 해당 후보를 소진하면 인근 권역 후보를 이어서 조회한다. 정렬은 기존 결정론적 순서를 유지하며 커서는 다음 offset을 외부에서 해석할 필요가 없는 문자열로 제공한다.

기존 `/pack`의 각 section에는 선택적 `total_count`, `shown_count`, `has_more`, `next_cursor`를 추가한다. 기존 소비자는 새 필드를 무시할 수 있다. 프론트는 순간별 섹션 아래에 `전체 N곳 중 M곳`과 `다른 후보 5곳 보기`를 표시하고, 새 페이지를 기존 목록에 중복 없이 누적한다.

## 오류와 경계

- cursor가 잘못되면 400 오류를 반환한다.
- page size는 서버에서 5로 고정해 과도한 응답을 막는다.
- 다음 페이지 요청 실패 시 기존 후보는 보존하고 해당 섹션에 재시도 가능한 오류를 표시한다.
- 후보가 모두 노출되면 버튼을 숨기고 `전체 후보를 모두 확인했습니다`를 표시한다.
- exact와 relaxed 결과가 겹칠 수 있으므로 external_id 기준으로 중복을 제거한다.

## 후속 단계

페이지네이션 안정화 뒤 후보 정렬을 사용자 조건 적합성, 근거 강도, 최신성, 차별성 기반으로 교체한다. 여행 추천은 웹 리서치를 기본 후보 원천으로 전환하고 내부 공공데이터는 마지막 교차확인으로 유지한다.

## 검증

- 첫 페이지와 다음 페이지가 겹치지 않는 검색 단위 테스트
- 전체 개수, cursor, has_more API 계약 테스트
- 잘못된 cursor와 마지막 페이지 테스트
- 프론트의 전체/표시 개수, 로딩, 오류, 완료 상태 테스트
- 기존 `/pack`, PDF, 하루방 및 TypeScript·프로덕션 빌드 회귀 검증
