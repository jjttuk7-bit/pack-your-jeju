# 여행팩 순차 안내 설계

## 문제

기존 Pack Journey는 사용자가 단계를 경험했는지가 아니라 현재 데이터만으로 진행 단계를 계산한다.

- 후보 응답이 도착하면 `후보 살펴보기`를 자동 완료해 첫 화면부터 2단계를 표시한다.
- 플랜 항목에 Day 값이 미리 있으면 장소를 담은 직후 3단계를 건너뛰고 4단계를 표시한다.

이 방식은 사용자가 보지 못한 1·3단계를 완료한 것으로 처리해 안내의 연속성을 깨뜨린다.

## 목표

Pack Journey를 `1 → 2 → 3 → 4` 순서로 경험하게 하고, 각 단계 카드에서 지금 해야 할 행동을 짧고 분명하게 설명한다.

## 결정

진행 단계는 후보 수나 미리 채워진 Day 값이 아니라 사용자의 명시적인 행동으로만 전환한다.

1. `후보 살펴보기`
   - 안내: `마음에 드는 후보를 골라보세요.`
   - 팩 대시보드에 처음 진입하면 항상 현재 단계로 표시한다.
   - 현재 단계의 `지금 후보 둘러보기`를 눌러 후보 영역으로 이동하면 2단계로 전환한다.
2. `플랜에 담기`
   - 안내: `고른 후보를 여행 플랜에 담아보세요.`
   - 플랜 항목 수가 실제로 증가하면 3단계로 전환한다.
3. `일정 정하기`
   - 안내: `담은 장소로 여행 일정을 만들어보세요.`
   - 기존 Day 값만으로 완료하지 않는다.
   - 사용자가 날짜, 시간대 또는 시간 고정 값을 직접 변경하면 4단계로 전환한다.
4. `저장·공유하기`
   - 안내: `완성한 일정을 저장하거나 함께 갈 사람에게 공유해보세요.`

예상하지 않은 이벤트나 미래 단계 버튼 클릭은 현재 단계를 건너뛰게 하지 않는다.

## 상태 구조

현재 단계 ID를 대시보드의 로컬 UI 상태로 관리한다. 순수 상태 전환 함수는 현재 단계에서 허용된 이벤트만 다음 단계로 바꾼다.

| 현재 단계 | 허용 이벤트 | 다음 단계 |
| --- | --- | --- |
| 후보 살펴보기 | 후보 영역으로 이동 | 플랜에 담기 |
| 플랜에 담기 | 플랜 항목 수 증가 | 일정 정하기 |
| 일정 정하기 | 일정 값 직접 변경 | 저장·공유하기 |
| 저장·공유하기 | 없음 | 유지 |

서버 API나 여행 플랜 데이터 계약은 변경하지 않는다. 대시보드가 유지되는 동안만 안내 상태를 유지하며, 별도의 영구 저장은 이번 범위에 포함하지 않는다.

## 화면 표현

- 네 단계 카드는 계속 2×2 그리드로 노출한다.
- 카드마다 번호, 단계명, 핵심 행동 문구, `완료·현재·다음` 상태를 함께 표시한다.
- 현재 단계는 감귤색, 완료 단계는 민트색, 다음 단계는 중립색으로 구분한다.
- 상태 텍스트를 유지해 색상만으로 의미를 전달하지 않는다.
- 현재 행동 버튼과 단계 버튼의 기존 스크롤·포커스 동작은 유지한다.

## 테스트

- 첫 팩 응답에 후보가 있어도 1단계부터 시작한다.
- 1단계 이동 행동 전에는 2단계로 넘어가지 않는다.
- 2단계에서 플랜 항목 수가 증가하면 3단계가 된다.
- 플랜 항목에 Day 값이 이미 있어도 3단계가 표시된다.
- 일정 편집 콜백이 실행된 뒤에만 4단계가 된다.
- 순서에 맞지 않는 이벤트로는 단계를 건너뛰지 않는다.
- 각 카드의 핵심 안내 문구와 접근 가능한 상태 이름을 검증한다.
