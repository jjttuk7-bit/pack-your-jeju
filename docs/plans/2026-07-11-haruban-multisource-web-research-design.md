# Haruban Multisource Web Research Design

## Status

Approved on 2026-07-11. This design implements the web-research-first constitution in the Haruban runtime.

## Goal

하루방이 제주 추천 질문과 후속 질문을 웹 우선으로 조사하고, 단일 검색 결과를 짧게 전달하는 수준을 넘어 출처 역할·최신성·사용자 조건을 반영한 충분한 답변을 제공하게 한다.

## Root Cause

- 웹 검색 라우팅은 현재 사용자 문장의 일부 키워드만 본다. 첫 질문이 웹 검색이어도 후속 `오름은?`, `맛집은?`은 `search_places`로 내려간다.
- 웹 검색은 검색어 하나와 짧은 답변 프롬프트, 최대 600 출력 토큰만 사용한다.
- 결과 출처는 URL과 제목만 보존하며 공식·플랫폼·경험 출처를 구분하지 않는다.
- 웹 검색 실패 폴백은 사용자를 다시 공공데이터 후보 흐름으로 유도한다.
- 시스템 프롬프트가 리뷰 사용 금지, 공공데이터 전용 사실 근거, 짧은 답변을 여전히 강제한다.

## Selected Approach

기존 OpenAI Responses API의 웹 검색 도구를 유지하면서, 하루방 내부에 검색 계획과 결과 병합 계층을 추가한다. 신규 검색 공급자나 크롤러는 도입하지 않는다.

이 방식은 현재 인증·오류 처리 구조를 재사용하면서 다음 품질 개선을 제공한다.

- 질문별 복수 검색 관점
- 부족한 결과에 대한 제한적 재검색
- 출처 역할 분류와 중복 제거
- 대화 기반 웹 검색 의도 유지
- 충분한 답변 계약
- 내부 공공데이터의 후순위 교차확인

## Components

### 1. Conversation Research Intent

`_should_use_web_research`는 현재 질문과 최근 대화를 함께 본다.

- 현재 질문이 추천, 탐색, 맛집, 카페, 오름, 관광지, 운영 정보라면 웹 검색한다.
- 최근 대화에서 웹 검색을 수행했고 현재 질문이 후속 카테고리·비교·추가 요청이면 웹 의도를 유지한다.
- 사용자가 `공공데이터 기준`, `내부 데이터`, `후보 수`처럼 내부 데이터 모드를 명시하면 `search_places`를 허용한다.
- 특정 장소 상세도 운영시간, 휴무, 가격, 현재 상태처럼 변동 정보가 포함되면 웹 검색을 우선한다.

### 2. Search Plan

질문과 문맥을 바탕으로 최대 3개의 검색 관점을 만든다.

1. `official`: 지역·장소·카테고리와 공식 정보 표현을 조합한다.
2. `platform`: 지도, 예약, 영업 상태, 메뉴처럼 플랫폼 정보가 유용한 표현을 조합한다.
3. `experience`: 최근 후기, 분위기, 동행자 적합성, 실제 방문 경험을 찾는 표현을 조합한다.

검색어는 중복을 제거하고 원래 질문을 항상 포함한다. 첫 결과가 모두 비어 있으면 표현을 단순화한 검색어로 한 번만 재시도한다.

### 3. Research Result

웹 연구 결과는 다음 정보를 반환한다.

```json
{
  "available": true,
  "research_status": "sufficient",
  "query": "구좌에서 가볼 만한 지역",
  "queries": ["..."],
  "answer": "...",
  "sources": [
    {
      "title": "...",
      "url": "...",
      "snippet": "...",
      "source_class": "official",
      "checked_at": "2026-07-11T...+09:00"
    }
  ],
  "reason": ""
}
```

`research_status`는 `sufficient`, `partial`, `conflicted`, `unavailable` 중 하나다. 일부 검색만 성공하면 `partial`, 모두 실패하면 `unavailable`로 처리한다.

### 4. Source Classification

도메인과 문서 단서를 사용해 출처를 보수적으로 분류한다.

- 지자체·공공기관·비짓제주·장소 운영 주체는 `official`
- 지도·예약·지역 매체는 `platform`
- 블로그·유튜브·SNS·리뷰는 `experience`
- 분류 근거가 부족하면 `web`

분류는 신뢰도를 자동 확정하지 않는다. 답변 조립에서 해당 출처가 어떤 종류의 주장을 지원할 수 있는지 판단하는 메타데이터다.

### 5. Answer Composition

웹 검색 프롬프트와 하루방 시스템 프롬프트를 새 헌법에 맞춘다.

- `짧게 답하세요`와 3~5문장 제한을 제거한다.
- 추천 질문에는 직접 결론, 후보별 이유, 비교, 주의점, 출처가 있어야 한다.
- 리뷰와 SNS는 경험 경향에 사용할 수 있으나 운영 사실의 단독 근거로 사용하지 않는다.
- “최고”, “가장”은 현재 조건과 확인한 출처 기준의 우선 후보로 해석하고 선정 기준을 밝힌다.
- 웹 검색 실패 시 공공데이터 후보로 자동 전환하지 않는다.
- 내부 공공데이터가 확인되면 마지막 `공공데이터 교차확인`에만 사용한다.

### 6. Public Data Cross-Check

첫 구현에서는 기존 `search_places`를 별도로 다시 호출해 웹 후보 전체를 정밀 매칭하는 대규모 파이프라인을 만들지 않는다. 선택 지역과 카테고리가 명확할 때 기존 내부 검색 결과를 보조 컨텍스트로 제공하고, 답변 마지막에 일치 여부를 한정해 설명한다.

내부 검색 실패나 미수록은 웹 추천을 제거하는 이유가 아니다. 웹 검색 실패 시에는 교차확인도 웹 답변을 대신하지 않는다.

## Error Handling

- 개별 검색 호출 실패는 다른 검색 관점의 성공을 무효화하지 않는다.
- 모든 호출이 실패하면 오류 내용을 사용자용 문장으로 그대로 노출하지 않고 실패 유형을 한국어로 요약한다.
- usable source가 없으면 `unavailable`이다.
- 답변만 있고 출처가 없으면 사실 추천에 충분한 것으로 처리하지 않고 `partial` 또는 `unavailable`로 낮춘다.
- 검색 예산은 최대 3개 기본 질의와 1개 재시도로 제한한다.

## Testing

- `구좌에서 가볼 만한 지역`은 `web_search_jeju`로 라우팅된다.
- 웹 검색 뒤 `구좌의 오름들은?`도 웹 검색을 유지한다.
- 웹 검색 뒤 `구좌에서 가장 맛집은?`도 웹 검색을 유지한다.
- `공공데이터 기준으로 구좌 오름 후보 수`는 `search_places`로 라우팅된다.
- 검색 계획은 공식·플랫폼·경험 관점을 만들고 중복을 제거한다.
- 일부 검색 성공은 `partial`, 전체 실패는 `unavailable`을 반환한다.
- 출처는 중복 제거되고 `source_class`, `checked_at`을 가진다.
- 웹 실패 폴백은 공공데이터 후보 선택을 유도하지 않는다.
- 시스템 프롬프트에는 리뷰 금지와 공공데이터 전용 규칙이 남지 않는다.
- 기존 하루방 테스트와 API 통합 테스트가 유지된다.

## Non-Goals

- 별도 검색 API, 자체 크롤러, 브라우저 자동화 도입
- 모든 후보의 운영시간을 완벽하게 정규화하는 범용 claim graph
- 네이버·카카오 비공개 API 우회
- 평점 하나로 자동 순위를 만드는 기능
- 기존 공공데이터 검색과 검증 기능 삭제
