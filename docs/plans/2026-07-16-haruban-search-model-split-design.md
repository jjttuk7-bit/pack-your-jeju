# 하루방 웹검색 모델 분리 설계

## 문제

운영 API에서 맛집 질문을 직접 측정한 결과 약 29.7초 뒤 웹검색 실패 안내가 반환됐다. 서버와 최종 답변 생성은 정상이며, `gpt-5-mini`를 사용한 OpenAI 내장 웹검색이 제한시간 안에 원문을 확보하지 못하는 것이 병목이다.

## 목표

- 웹검색 응답 성공률과 체감 속도를 개선한다.
- 최종 근거 종합 품질은 유지한다.
- 같은 검색 공급자에 대한 순차 재시도로 대기시간이 누적되지 않게 한다.
- 검색 실패를 숨기거나 공공데이터 후보로 바꾸지 않는다.

## 검토한 접근

1. **웹검색은 GPT-4o, 최종 답변은 GPT-5 mini로 역할 분리 — 채택**
   - GPT-4o는 빠른 범용 검색 실행에 사용한다.
   - GPT-5 mini는 수집된 근거의 조건 판단과 최종 설명에 유지한다.
2. **모든 단계를 GPT-4o로 교체**
   - 단순하지만 최종 종합 품질 저하 가능성과 비용 증가가 있다.
3. **GPT-5 mini를 유지하고 timeout만 확대**
   - 성공 가능성은 늘 수 있지만 사용자가 더 오래 기다리게 된다.

## 설계

### 모델 역할

- `WEB_SEARCH_MODEL = "gpt-4o"`
- 최종 Chat Completions 모델은 기존 `llm.MODEL = "gpt-5-mini"` 유지
- 두 모델을 코드에서 명시적으로 분리해 향후 개별 평가와 교체가 가능하게 한다.

### 검색 흐름

- 사용자 질문당 OpenAI 내장 웹검색 요청은 한 번만 실행한다.
- 검색 요청 timeout은 18초로 제한한다.
- `max_tool_calls=1`로 제한한다.
- 사용 가능한 원문 출처가 확보되면 GPT-5 mini가 최종 답변을 조립한다.
- timeout, 공급자 오류, usable source 없음이면 즉시 제한 답변으로 넘어간다.
- 같은 공급자에 확장 검색어를 즉시 재시도하지 않는다.

### 실패 안내

- 단일 검색 흐름에 맞춰 “웹 출처를 확인했지만”으로 표현한다.
- 확인되지 않은 장소나 순위를 생성하지 않는다.
- 공공데이터 후보를 웹검색 결과처럼 반환하지 않는다.

### 로그

- 웹검색 로그에 `model=gpt-4o`, `timeout_seconds=18.0`, `elapsed_ms`를 기록한다.
- 최종 채팅 로그의 `elapsed_ms`와 비교해 검색과 답변 생성 시간을 분리한다.

## 테스트

- 웹검색 Responses 요청 모델이 `gpt-4o`인지 검증한다.
- `max_tool_calls=1`인지 검증한다.
- 검색 실패 뒤 두 번째 호출이 발생하지 않는지 검증한다.
- 반환된 `queries`가 원 질문 하나만 포함하는지 검증한다.
- 실패 안내가 복수 검색을 암시하지 않는지 검증한다.
- GPT-5 mini 최종 답변 모델과 `reasoning_effort="low"` 회귀 테스트를 유지한다.

## 성공 기준

- 검색 실패 시 서버 응답이 약 20~25초 안에 종료된다.
- 검색 성공 시 출처가 포함된 답변을 반환한다.
- 배포 후 동일 질문 세트로 성공률과 p50/p95 지연을 확인한다.
