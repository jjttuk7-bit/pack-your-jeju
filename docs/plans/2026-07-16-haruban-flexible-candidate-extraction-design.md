# 하루방 추천 후보 유연 추출 설계

## 문제와 재현

하루방은 추천 답변 본문을 정상 생성했지만 `place_candidates`가 빈 배열로 반환되어 체크박스가 표시되지 않았다.

운영 API에서 `최근 제주시 맛집들 알려줘`를 재현한 결과 답변은 성공했지만 후보 수는 0개였다. 현재 파서는 `## 추천 장소`, 하이픈 글머리표, 모든 굵은 문자열 탐색을 전제로 한다. 실제 답변은 다음 변형을 포함했다.

- 섹션 제목: `**추천 장소**`
- 필드 글머리표: `• 특징:`, `• 위치:`
- 출처 링크: `[**visitjeju.net**](URL)`

따라서 섹션을 찾지 못하거나, 굵은 출처 라벨을 다음 장소명으로 오인해 장소 블록과 URL을 분리한다.

## 검토한 접근

### 1. 줄 단위 결정론적 Markdown 파서 보강 — 채택

`## 추천 장소`와 `**추천 장소**`를 모두 섹션 시작으로 인정한다. 장소명은 줄 전체가 굵은 고유명인 경우만 허용하고 Markdown 링크 안의 굵은 도메인명은 제외한다. `-`, `*`, `•` 글머리표를 구조화 필드로 인정한다.

장점은 추가 모델 호출과 지연이 없고 기존 안전 규칙을 유지한다는 것이다.

### 2. 검색 프롬프트만 강화

모델에 반드시 `## 추천 장소` 형식을 사용하도록 다시 강조할 수 있다. 하지만 생성 모델은 표, 굵은 제목, 다른 글머리표를 반환할 수 있어 같은 문제가 재발할 수 있다.

### 3. 두 번째 LLM으로 JSON 후보 추출

다양한 답변 형식에 대응하기 쉽지만 호출 비용과 지연이 늘고, 출처에 없는 후보를 모델이 보완할 위험이 있다. 사용자 요구인 빠른 답변과도 맞지 않는다.

## 채택 설계

### 추천 섹션 탐색

- `## 추천 장소`와 줄 전체가 `**추천 장소**`인 형식을 지원한다.
- 다음 Markdown `##` 제목 또는 `**방문 팁**`, `**공공데이터 교차확인**`, `**웹 출처 기준**`, `**확인 필요**` 같은 후속 섹션 전까지만 파싱한다.
- 추천 섹션이 없으면 기존처럼 후보를 만들지 않는다.

### 장소명과 후보 블록

- 장소명은 다음처럼 줄 전체가 굵은 문자열인 경우만 인정한다.
  - `**자매국수**`
  - `- **자매국수**`
  - `1. **자매국수**`
- `[**출처명**](URL)`처럼 링크 내부의 굵은 문자열은 장소명이 아니다.
- 각 장소 블록은 다음 장소명 또는 추천 섹션 끝까지다.
- 가격·시간·거리·수량 형태의 문자열과 일반 섹션 제목은 기존처럼 제외한다.

### 구조화 필드와 출처

- `특징`, `추천 이유`, `이유`, `위치`, `주소` 중 하나 이상이 있어야 한다.
- 필드 앞의 `-`, `*`, `•`, 인용 부호와 공백 변형을 허용한다.
- 후보 블록 안에 실제 URL이 있어야 플랜 후보가 된다.
- 출처 메타데이터와 URL을 연결해 `source_title`, `source_class`, `checked_at`을 유지한다.
- URL이나 구조화 필드가 없는 굵은 문구는 체크박스로 노출하지 않는다.

### 프론트 계약

프론트 변경은 필요하지 않다. 백엔드가 `place_candidates`를 반환하면 현재 `WebPlaceCandidatePicker`가 체크박스를 표시하고, 체크 즉시 공용 `selectedPlanItems`에 추가한다.

## 테스트

- 사용자가 제시한 형식과 동일하게 굵은 섹션 제목, `•` 글머리표, 굵은 출처 링크를 포함한 답변에서 세 식당이 추출되는지 검증한다.
- 굵은 출처 도메인명이 별도 후보로 생성되지 않는지 검증한다.
- 기존 `## 추천 장소` 형식 테스트가 계속 통과하는지 확인한다.
- 구조화 필드나 URL이 없는 굵은 문구는 계속 제외되는지 확인한다.
- 전체 하루방 테스트, API 테스트, 프론트 후보 표시 계약 테스트를 실행한다.

## 성공 기준

- 동일 운영 질문에서 `place_candidates`가 1개 이상 반환된다.
- 장소 후보 이름에 출처 도메인, 가격, 섹션 제목이 포함되지 않는다.
- 프론트에서 추천 후보 체크박스가 다시 표시된다.
- 추가 OpenAI 호출 없이 현재 응답시간 수준을 유지한다.
