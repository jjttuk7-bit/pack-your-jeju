# 사용자 승인형 여행 동선 추천 설계

> 상태: 승인됨
> 작성일: 2026-07-17
> 대상 서비스: 제주를 담다 (Pack Your Jeju)

## 1. 목적

사용자가 선택한 여행 순간을 지도 마커로만 보여주는 현재 기능을 실제 여행 순서를 판단할 수 있는 `내 여행 동선`으로 확장한다. 서비스는 이동수단, 출발·도착점, 날짜·시간대, 날씨, 운영정보 확인 상태와 사용자가 고정한 일정을 함께 고려해 더 나은 순서를 제안한다.

동선 추천은 자동 편집 기능이 아니다. 서비스가 현재 동선과 추천 동선을 비교해 이유와 절감 효과를 보여주고, 사용자가 승인한 경우에만 플랜을 한 번에 변경한다.

## 2. 확정된 제품 원칙

- 기본 목표는 최단거리만 찾는 것이 아니라 **실제 일정형 동선**을 만드는 것이다.
- 이동수단은 자동차·대중교통·도보 중 사용자가 선택하고 기본값은 자동차로 한다.
- 기본 출발·도착은 숙소 왕복으로 하되 공항·렌터카 영업소 등 서로 다른 지점을 직접 지정할 수 있다.
- 실제 경로 서비스를 우선 사용하고 실패하면 좌표 기반 예상 경로로 전환한다.
- `시간 고정` 일정과 사용자가 지정한 출발·도착점은 변경하지 않는다.
- 추천 전후 차이가 의미 있을 때만 변경안을 만든다.
- 날씨 변경안과 마찬가지로 미리보기·명시적 승인·되돌리기를 제공한다.
- 확인하지 못한 이동시간이나 접근성을 실제 경로처럼 표현하지 않는다.

## 3. 범위

### 포함

- 날짜별 방문 순서와 지도 경로선
- 번호가 있는 방문 마커
- 자동차·대중교통·도보 선택
- 숙소 왕복 및 출발·도착 지점 직접 지정
- 현재 순서와 추천 순서의 시간·거리 비교
- 고정 일정 사이의 재배열
- 날씨·시간대·운영정보 확인 상태를 반영한 추천 이유
- 실제 경로와 예상 경로의 구분
- 추천 적용, 기존 일정 유지, 안전한 되돌리기

### 1차 범위에서 제외

- 사용자의 실시간 GPS 추적
- 실시간 교통체증을 보장하는 내비게이션
- 버스표·렌터카·식당 예약의 자동 변경
- 백그라운드에서 플랜을 자동 재배열하는 기능
- 여러 사용자가 동시에 한 플랜을 편집하는 협업 최적화

## 4. 사용자 화면

기존 `이번 여행 지도`를 `내 여행 동선`으로 확장한다.

### 4.1 기본 화면

- 상단에 `Day 1`, `Day 2`와 같은 날짜 탭을 둔다.
- 이동수단 선택기를 자동차·대중교통·도보로 제공한다.
- 숙소 왕복을 기본으로 하고 `출발·도착 직접 지정`을 제공한다.
- 지도에는 `1 → 2 → 3` 번호 마커와 날짜별 경로선을 표시한다.
- 지도 아래에 총 이동거리, 예상 이동시간, 가장 긴 구간을 요약한다.
- 장소가 두 곳 이상이면 `동선 추천받기`를 활성화한다.

### 4.2 추천 비교 화면

추천 요청 후 현재 순서와 추천 순서를 나란히 보여준다.

```text
현재 동선                  추천 동선
숙소 → 오름 → 카페         숙소 → 시장 → 카페
→ 시장 → 숙소              → 오름 → 숙소

2시간 40분 · 96km          1시간 55분 · 71km
                           45분 · 25km 절약
```

추천 이유는 사용자가 판단할 수 있는 언어로 표시한다.

- 오전 날씨가 좋은 시간에 오름 배치
- 시장과 카페가 가까워 연속 방문
- 시간 고정 장소는 기존 시간 유지
- 운영시간을 확인하지 못한 장소는 방문 전 재확인 필요

사용자는 `이 순서로 적용` 또는 `기존 일정 유지`를 선택한다. 적용 직후에는 `되돌리기`를 제공한다.

## 5. 시스템 구조

### 5.1 구성요소

- **Route Planner API**: 요청 검증, 날짜별 분리, 경로 조회, 추천 조립을 담당한다.
- **Route Provider**: 실제 경로 공급자를 감싸는 인터페이스다. 공급자 이름·확인 시각·이동수단·구간별 거리와 시간을 보존한다.
- **Estimated Route Provider**: 외부 경로 실패 시 좌표 거리와 이동수단별 보정값으로 예상 시간과 거리를 계산한다.
- **Route Policy Engine**: 고정 일정, 날씨 영향, 시간대, 운영정보 상태와 이동비용을 평가한다.
- **Route Proposal Guard**: fingerprint와 적용 전 스냅샷을 이용해 오래된 추천 적용과 위험한 되돌리기를 차단한다.
- **My Travel Route UI**: 지도, 날짜 탭, 이동수단, 비교 화면, 적용·유지·되돌리기를 제공한다.

### 5.2 데이터 흐름

```text
선택한 플랜 + 이동수단 + 출발·도착
  → 날짜별 일정과 고정점 검증
  → 현재 순서 실제 경로 조회
  → 고정점 사이 후보 순서 생성
  → 후보별 실제 경로 또는 예상 경로 계산
  → 이동비용·날씨·시간대·운영정보 정책 평가
  → 의미 있는 개선안만 비교 결과로 반환
  → 사용자 미리보기
  → 명시적 승인 시 원자적 적용
  → 직전 플랜 스냅샷으로 안전한 되돌리기
```

## 6. 요청과 응답 계약

### 6.1 요청

`POST /route/plan`

- 여행 기간과 날짜별 일정
- 각 일정의 ID, 이름, 좌표, 지역, Day, 시간대, 시작 시각, 고정 여부
- 이동수단: `driving | transit | walking`
- 출발·도착점 좌표와 표시 이름
- 현재 날씨 리포트 fingerprint
- 기존에 유지하기로 선택한 제안 fingerprint

### 6.2 응답

- 날짜별 현재 경로와 추천 경로
- 방문 순서와 구간별 거리·예상 시간·경로 geometry
- 총 거리·총 시간·절감 거리·절감 시간
- 경로 상태: `verified_route | estimated_route | mixed_route | unavailable`
- 공급자, 조회 시각, 실제/예상 구간 표시
- 추천 이유와 확인 필요 항목
- 원자적으로 적용할 `reorder_day_items` operation
- 제안 fingerprint와 입력 플랜 fingerprint

## 7. 추천 알고리즘

### 7.1 기본 절차

1. 일정을 날짜별로 분리한다.
2. 출발점, 도착점과 `시간 고정` 일정을 움직일 수 없는 anchor로 둔다.
3. anchor 사이의 일정만 후보 순서로 재배열한다.
4. 장소 수가 적으면 가능한 순서를 제한적으로 비교하고, 많으면 결정론적 근접 탐색을 사용한다.
5. 각 후보의 실제 경로를 우선 계산하고 실패 구간은 예상 경로로 보완한다.
6. 이동시간을 기본 비용으로 두고 날씨 부적합, 시간대 불일치, 운영정보 확인 필요와 과도한 권역 왕복에 벌점을 준다.
7. 현재 순서 대비 개선이 최소 임계값 이상일 때만 제안한다.

### 7.2 정책 우선순위

1. 사용자 고정 일정과 출발·도착점 보존
2. 날짜와 시간대 계약 보존
3. 운영 불가능 또는 확인 필요 상태 경고
4. 날씨 영향이 큰 야외 일정의 적합 시간대
5. 총 이동시간과 불필요한 왕복 감소
6. 같은 비용에서는 기존 사용자 순서를 최대한 보존

정책은 경로 공급자의 원본 시간·거리와 분리한다. 서비스 판단 임계값을 실제 교통 통제나 공식 운영정보처럼 표현하지 않는다.

## 8. 실제 경로와 대체 계산

- 자동차는 실제 도로 경로를 제공하는 공급자를 우선 연결한다.
- 대중교통과 도보는 해당 이동수단을 지원하는 공급자가 있을 때 실제 경로를 사용한다.
- 지원하지 않거나 timeout이 발생하면 이동수단별 좌표 기반 예상값으로 전환한다.
- 예상 결과에는 직선 또는 단순 보정 경로임을 명시하고 정확한 도착시간을 약속하지 않는다.
- 공급자 호출에는 timeout, 제한된 재시도, 구간 캐시를 둔다.
- 캐시는 공급자, 이동수단, 출발·도착 좌표, 확인 시각을 보존하며 오래된 값을 최신 결과처럼 표시하지 않는다.

## 9. 실패 처리

- 좌표가 없는 장소는 제외하지 않고 `위치 확인 필요`로 표시한다.
- 장소가 두 곳 미만이면 최적화하지 않고 현재 순서를 유지한다.
- 실제 경로 전체 실패 시 예상 경로로 전환한다.
- 일부 구간만 실패하면 실제 구간과 예상 구간을 각각 표시한다.
- 개선 폭이 작으면 `현재 동선이 적절합니다`를 반환한다.
- 사용자가 플랜을 수정한 뒤에는 이전 proposal fingerprint 적용을 거부한다.
- 날씨 제안 적용으로 플랜이 바뀌면 동선을 다시 계산한다.
- 동선 제안 적용으로 시간대가 바뀌는 경우 날씨 리포트를 다시 계산한다.
- Route Planner 장애가 여행팩, 지도 마커, PDF 저장을 막지 않게 한다.

## 10. 개인정보와 보존

- 숙소와 직접 지정한 출발·도착 주소는 경로 계산 목적에만 사용한다.
- 사용자가 저장을 명시적으로 선택하지 않으면 원문 주소를 장기 보관하지 않는다.
- 로그에는 정밀 주소 대신 좌표를 낮은 정밀도로 처리하거나 식별 불가능한 요청 ID를 사용한다.
- 외부 경로 공급자에 전달하는 데이터 범위와 보존 정책을 운영 문서에 명시한다.

## 11. 테스트와 검증

### 백엔드

- 날짜별 일정만 같은 경로에서 계산하는지 검증
- 고정 일정과 출발·도착점이 이동하지 않는지 검증
- 실제 경로 성공, 전체 실패, 일부 실패 경로 검증
- 좌표 누락과 장소 2곳 미만 처리 검증
- 추천 개선 임계값과 결정론 검증
- 오래된 fingerprint와 중복 operation 거부 검증
- 날씨·운영정보 벌점이 원본 경로값을 변조하지 않는지 검증

### 프런트

- 날짜 탭과 이동수단 전환 검증
- 번호 마커와 날짜별 경로선 검증
- 현재·추천 시간과 거리 비교 검증
- 승인 전 플랜 불변 검증
- 적용·기존 유지·되돌리기 검증
- 사용자 편집 후 오래된 적용과 되돌리기 차단 검증
- 실제·예상·혼합 경로 라벨 검증
- 모바일 지도와 비교 패널의 접근성 검증

### 프로덕션 스모크

- 자동차 숙소 왕복 일정의 실제 경로 응답
- 경로 공급자 키가 없을 때 예상 경로 fallback
- Railway timeout 시 기존 플랜과 지도 마커 유지
- Vercel UI에서 추천 미리보기와 승인 적용
- PWA 재실행 후 저장된 플랜과 제안 상태의 안전한 복원

## 12. 단계적 출시

1. 경로 타입, fingerprint, 예상 경로와 테스트 기반
2. 자동차 실제 경로 공급자와 날짜별 경로 API
3. 고정점 기반 추천과 승인·되돌리기
4. 지도 번호·경로선·비교 패널
5. 대중교통·도보 공급자 연결 또는 명시적 예상 경로
6. 날씨 리포트 재계산과 운영정보 경고 통합
7. 관측 로그, timeout, 캐시, 프로덕션 스모크

## 13. 성공 기준

- 사용자는 지도에서 날짜별 방문 순서와 이동 구간을 이해할 수 있다.
- 서비스는 현재 동선과 추천 동선의 시간·거리 차이를 설명한다.
- 고정 일정과 사용자 선택을 침범하지 않는다.
- 사용자가 승인하기 전에는 플랜이 바뀌지 않는다.
- 실제 경로를 확인하지 못했을 때 예상값임을 숨기지 않는다.
- 외부 경로 장애가 기존 여행 준비 기능을 중단시키지 않는다.
