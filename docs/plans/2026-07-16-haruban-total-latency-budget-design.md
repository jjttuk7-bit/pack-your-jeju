# 하루방 전체 응답 지연 예산 설계

## 배경

운영 환경에서 맛집 질문이 50초 이상 걸렸다. 최신 코드의 웹검색 1회 제한은 20초지만, 첫 검색 실패 뒤 확장 검색어 재시도가 다시 최대 20초 동안 순차 실행된다. 두 검색이 모두 실패하면 웹검색에만 약 40초가 사용되고, 이후 GPT 최종 답변 생성 시간이 더해진다.

## 목표

- 웹검색 재시도 원칙을 유지한다.
- 두 번의 검색을 합친 대기시간이 20초를 넘지 않게 한다.
- GPT-5 mini의 최종 답변 추론 강도를 지연 민감형으로 명시한다.
- 검색 실패를 숨기거나 공공데이터 추천으로 바꾸지 않는다.
- 운영 로그만으로 검색과 최종 답변 중 어느 구간이 느린지 확인할 수 있게 한다.

## 검토한 접근

### 1. 전체 검색 예산 20초를 두 시도에 분배 — 채택

첫 검색에 최대 14초, 확장 검색어 재시도에 최대 6초를 배정한다. 재검색 품질 원칙을 유지하면서 순차 타임아웃 누적을 제거한다.

### 2. 재검색 제거

가장 빠르지만 첫 검색어가 빈약할 때 검색어를 바꿔 확인한다는 `TRUST_ENGINE.md` 원칙을 약화한다.

### 3. GPT-4o로 모델 교체

최종 문장 생성은 빨라질 수 있지만 웹검색 순차 대기 문제를 해결하지 못하고 토큰 비용이 크게 증가한다.

## 설계

### 웹검색 지연 예산

- 전체 목표 예산: 20초
- 1차 검색 제한: 14초
- 2차 확장 검색 제한: 최대 6초
- 1차 검색이 일찍 끝나더라도 2차 검색은 6초를 넘기지 않는다.
- 1차 검색이 성공해 사용 가능한 출처를 확보하면 재검색하지 않는다.
- 1차 검색이 실패하거나 출처가 없을 때만 확장 검색어로 한 번 재시도한다.

### 최종 답변 생성

- 모델은 `gpt-5-mini`를 유지한다.
- GPT-5 계열 최종 Chat Completions 요청에는 `reasoning_effort="low"`를 명시한다.
- 웹검색 Responses 요청은 기존 `reasoning={"effort": "low"}`를 유지한다.
- 출력 상한과 근거 공개 규칙은 유지한다.

### 실패 처리

- 두 검색이 모두 실패하면 `research_status="unavailable"`을 유지한다.
- 내부 공공데이터 결과를 웹검색 성공처럼 포장하지 않는다.
- 사용자에게 확인하지 못한 범위와 재확인 필요를 명시한다.

### 관측성

각 검색 로그에 시도별 제한시간과 실제 `elapsed_ms`를 남긴다. 전체 채팅 로그의 `elapsed_ms`와 함께 보면 웹검색과 최종 답변 생성 병목을 분리할 수 있다.

## 테스트

- 첫 검색 실패 후 재검색이 실행되는지 검증한다.
- 두 시도에 전달된 제한시간의 합이 20초 이하인지 검증한다.
- 첫 검색 성공 시 두 번째 검색이 실행되지 않는지 기존 회귀 테스트를 유지한다.
- GPT-5 mini 최종 답변 요청에 `reasoning_effort="low"`가 전달되는지 검증한다.
- 하루방 및 LLM 전체 관련 테스트를 실행한다.

## 성공 기준

- 검색 공급자 타임아웃 상황에서도 웹검색 대기 누적은 최대 약 20초다.
- 최종 답변을 포함한 운영 목표는 일반적으로 30초 이내다.
- 응답 품질과 실제 지연시간은 배포 후 동일 질문 세트의 p50/p95로 추가 확인한다.
