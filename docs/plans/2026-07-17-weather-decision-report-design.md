# 제주 여행 날씨 의사결정 리포트 설계

> 작성일: 2026-07-17
>
> 상태: 사용자 승인 완료
>
> 핵심 원칙: 날씨에 따른 변경안을 추천하되 선택과 적용은 항상 사용자가 한다.

## 1. 목표

현재의 기상 수치 카드를 **제주 여행 일정 판단 리포트**로 고도화한다. 사용자가 여행 기간의 날씨를 이해하고, 날씨와 충돌하는 일정을 확인하고, 변경 전후를 비교한 뒤 직접 승인할 수 있게 한다.

```text
여행 플랜
→ 날짜·시간대·지역별 기상청 예보
→ 장소 유형별 날씨 영향 판단
→ 일정 충돌 탐지
→ 변경안 미리보기
→ 사용자 승인
→ 적용·되돌리기
```

성공의 기준은 더 많은 기상 수치를 표시하는 것이 아니다. 사용자가 **어떤 일정을 유지하고 무엇을 조정할지 근거를 보고 결정할 수 있는가**가 기준이다.

## 2. 절대 원칙

- 날씨 리포트는 추천 기능이며 자동 일정 변경 기능이 아니다.
- 사용자가 승인하기 전에는 플랜을 수정하지 않는다.
- 적용 전에 변경되는 모든 일정과 영향을 보여준다.
- 적용 후 한 번의 동작으로 되돌릴 수 있다.
- 예약·사용자 고정 일정은 변경 대상에서 제외한다.
- 예보가 없는 날짜·지역·시간대의 위험을 추정해 만들지 않는다.
- 기상청 예보와 서비스의 여행 판단 규칙을 구분해 표시한다.
- AI는 설명을 쉽게 만들 수 있지만 위험 판정과 일정 변경 연산은 결정론적 규칙으로 처리한다.
- 사용자가 유지하기로 한 일정은 같은 예보 조건에서 반복 권유하지 않는다.

## 3. 현재 구현과 한계

### 현재 구현

- 기상청 단기예보와 API Hub 텍스트 예보를 수집한다.
- `SKY`, `PTY`, `POP`, `TMP`, `WSD`, `REH`를 정규화한다.
- 최대 3일의 일별 예보를 반환한다.
- 비·바람·풍랑·안개·더위·눈 신호를 Trust Engine에 전달한다.
- 대시보드에 위험 배지와 일별 기온·강수확률·풍속을 표시한다.

### 한계

- 선택 지역 중 첫 번째 지역의 예보만 조회한다.
- 하루 중 자료가 가장 많이 채워진 대표 시점 하나를 사용한다.
- 오전·오후·저녁의 변화가 보이지 않는다.
- 모든 장소에 같은 날씨 위험을 적용한다.
- 일정 순서, 장소 유형, 예약·고정 여부와 연결되지 않는다.
- 날씨가 나빠도 사용자가 취할 다음 행동을 제안하지 않는다.
- 현재 `TravelPlanItem`에는 시간대와 고정 일정 정보가 없다.

## 4. 사용자 경험

### 4.1 여행 전체 요약

리포트 상단은 여행 기간 전체를 한 문장으로 판단한다.

> 둘째 날 오전에 비와 강한 바람이 예상됩니다. 오름 일정 1건은 시간 변경을 권장하며, 나머지 일정은 유지할 수 있습니다.

함께 표시할 정보:

- 전체 상태: `좋음`, `조정 권장`, `주의`, `판단 보류`
- 적용 지역과 여행 기간
- 예보 발표 시각
- 확인된 날짜·시간대의 범위
- 예보 누락 또는 다음 재확인 안내

### 4.2 날짜·시간대별 리포트

하루를 다음 세 시간대로 구분한다.

- 오전: 06:00~11:59
- 오후: 12:00~17:59
- 저녁: 18:00~23:59

각 시간대에는 다음을 표시한다.

- 하늘 상태와 강수 형태
- 최대 강수확률
- 최저·최고 기온과 체감 참고
- 평균·최대 풍속
- 평균 습도
- 야외활동 적합 상태
- 준비물과 재확인 항목

### 4.3 내 일정 영향

날씨 신호를 장소 유형과 연결한다.

| 일정 유형 | 주요 판단 신호 | 대표 안내 |
|---|---|---|
| 오름·숲길 | 비, 강풍, 안개, 노면, 일몰 | 시간 변경 또는 실내 대안 |
| 해변·해안도로 | 강풍, 풍랑, 비, 일몰 | 체류 단축·시간 변경 |
| 우도·섬 이동 | 바람, 풍랑, 교통 변동 | 배편·운항 공지 재확인 |
| 시장·야외 행사 | 비, 폭염, 강풍, 공식 공지 | 우천 운영과 개최 여부 확인 |
| 실내 전시·카페 | 날씨 영향이 낮음 | 우천 대안 후보 |
| 일반 이동 | 강수, 시야, 바람 | 이동 여유시간 확보 |

### 4.4 변경안 카드

날씨와 충돌하는 일정에만 카드가 나타난다.

```text
2일차 오전 · 성산 오름
조정 권장

강수확률과 바람 때문에 오전 산행이 불편할 수 있습니다.

현재
09:30 성산 오름
13:00 실내 전시

추천
09:30 실내 전시
15:00 성산 오름 또는 다음 날 이동

[변경안 적용] [다른 대안 보기] [기존 일정 유지]
```

적용 전에 이동되는 다른 일정과 예상 영향을 모두 보여준다.

## 5. 시스템 구조

### 5.1 역할 분리

기존 `weather.py`는 기상청 요청과 원본 정규화를 담당한다. 새 모듈은 여행 판단과 변경안 생성을 담당한다.

```text
apps/api/engine/weather.py
  기상청 조회·시간별 예보 정규화

apps/api/engine/weather_report.py
  시간대 집계·장소 유형별 영향·변경안 생성

apps/api/routes/weather_report.py
  POST /weather/report 계약과 오류 응답

apps/web/src/components/WeatherDecisionReport.tsx
  리포트·충돌·변경 미리보기 UI

apps/web/src/weatherProposal.ts
  변경안 검증·적용·되돌리기 순수 함수
```

### 5.2 API 분리

날씨 리포트는 `/pack`과 별도의 `POST /weather/report`로 제공한다.

분리 이유:

- 플랜 전체를 다시 조립하지 않고 최신 예보만 갱신할 수 있다.
- 지역별 기상 조회를 캐시할 수 있다.
- 일정 변경 후 리포트만 다시 계산할 수 있다.
- 기상청 장애가 장소 후보 생성까지 막지 않는다.

기존 `/pack.weather`는 호환성을 위해 간단한 요약으로 유지한다.

## 6. 데이터 계약

### 6.1 플랜 항목 확장

`TravelPlanItem`에 다음 선택 필드를 추가한다.

```typescript
type Daypart = 'morning' | 'afternoon' | 'evening';

interface TravelPlanItem {
  // 기존 필드
  daypart?: Daypart;
  startTime?: string | null;
  durationMinutes?: number | null;
  fixed?: boolean;
  reservationNote?: string | null;
}
```

명시적인 시간이 없으면 사용자가 확인할 수 있는 기본 시간대를 배정한다. 이 값은 위험 판정 전에 리포트 화면에서 수정할 수 있다. 서버가 임의의 정확한 시각을 사실처럼 만들지 않는다.

## 7. 기상 집계와 판단 규칙

### 7.1 시간대 집계

기상청의 시간별 항목을 지역·날짜·시간대별로 묶는다.

- 강수확률: 시간대 최대값
- 강수 형태: 가장 위험도가 높은 값
- 기온: 최저·최고값
- 풍속: 평균과 최대값
- 습도: 평균값
- 하늘 상태: 시간대의 최빈값, 동률이면 더 보수적인 상태

임의의 대표 시점 하나를 고르지 않는다.

### 7.2 위험 판단

기상청의 수치는 사실 데이터다. 일정 영향 등급은 서비스의 규칙이다. UI에 이 구분을 명시한다.

판단 결과:

- `suitable`: 일정 유지 가능
- `prepare`: 준비물 또는 시간 여유 권장
- `adjust`: 시간·순서 변경 검토
- `official_check`: 운항·통제·행사 공지 확인 필요
- `unknown`: 판단할 예보 부족

임계값은 코드 한 곳의 정책표에서 관리하고 테스트로 고정한다. 공식 특보가 아닌 일반적인 비를 빨간 위험으로 과장하지 않는다.

### 7.3 대안 우선순위

1. 같은 플랜 안에서 시간대 교환
2. 같은 플랜 안에서 날짜 이동
3. 사용자가 이미 선택한 실내·저위험 장소 활용
4. 인근 확인 후보 제안
5. 대안이 없으면 재확인 권고만 제공

예약·고정 일정은 이동하지 않는다. 다른 일정을 이동해야 하는 제안은 영향을 함께 표시한다.

## 8. 적용과 되돌리기

### 8.1 적용 전 검증

- 제안이 계산된 플랜 버전과 현재 플랜이 같은지 확인
- 대상 일정이 아직 존재하는지 확인
- 사용자가 그 사이 날짜·시간대를 수정했는지 확인
- 고정 일정으로 바뀌지 않았는지 확인
- 목적지 시간대에 다른 고정 일정이 없는지 확인

조건이 달라졌으면 적용하지 않고 새 리포트를 요청한다.

### 8.2 적용

초기 구현은 로컬 플랜 상태에 원자적으로 적용한다.

- 적용 직전 `selectedPlanItems` 스냅샷 저장
- 모든 operation을 복사본에 검증 적용
- 하나라도 실패하면 전체 적용 취소
- 성공하면 새 상태로 한 번만 교체
- 적용된 항목에 일시적인 강조 표시

### 8.3 되돌리기

- 가장 최근 날씨 변경 1회를 즉시 되돌릴 수 있다.
- 사용자가 적용 후 직접 편집했다면 덮어쓰지 않고 확인을 요구한다.
- 향후 서버 저장 플랜에서는 `plan_version`과 감사 이력을 사용한다.

## 9. 화면과 접근성

### 9.1 화면 순서

1. 여행 전체 판단
2. 날짜 탭과 지역 표시
3. 오전·오후·저녁 예보
4. 내 일정 영향
5. 변경안 미리보기
6. 준비물과 출발 전 확인

### 9.2 상태 표현

- 초록: 일정 유지 가능
- 노랑: 준비 권장
- 주황: 변경 검토
- 빨강: 공식 통제·특보 등 강한 안전 신호
- 회색: 예보 없음·판단 보류

색상만 사용하지 않고 아이콘, 상태명, 이유를 함께 표시한다. 모바일에서는 변경 전후를 위아래로 놓고 주요 버튼의 터치 영역을 최소 44px로 유지한다.

### 9.3 사용자 선택

- `변경안 적용`
- `다른 대안 보기`
- `기존 일정 유지`
- `적용 취소`

`기존 일정 유지`를 선택한 fingerprint는 보존한다. 예보 위험 단계가 더 높아지거나 발표 회차가 의미 있게 달라질 때만 다시 제안한다.

## 10. 하루방 연결

날씨 리포트에서 다음 질문을 바로 시작할 수 있다.

- 비 오는 시간에 갈 실내 장소 찾아줘
- 오름을 어느 날로 옮기는 게 좋아?
- 우도 일정 유지해도 될까?
- 부모님과 다니기 편한 대안으로 바꿔줘

하루방은 리포트의 구조화된 예보와 현재 플랜을 컨텍스트로 받는다. 새 장소를 추천할 때는 기존 웹 리서치·출처 계약을 그대로 적용한다. 하루방 답변만으로 플랜을 수정하지 않고 동일한 변경 미리보기와 승인 단계를 거친다.

## 11. 오류와 부분 성공

- 전체 조회 실패: 플랜은 유지하고 `판단 보류` 표시
- 일부 지역 실패: 확인된 지역만 리포트 제공, `partial=true`
- 일부 시간 누락: 해당 시간대만 `unknown`
- 오래된 발표: 최신 정보처럼 표시하지 않고 갱신 권고
- 예보 범위 밖: 변경안 생성 금지
- 장소 지역 미확인: 날씨 영향 판단 보류
- 장소 유형 미분류: 일반 이동·야외 규칙을 임의 적용하지 않고 확인 필요
- 변경 적용 중 플랜 불일치: 전체 취소 후 재계산 안내

기상청 장애가 장소 검색, 기존 플랜 열기, PDF 다운로드를 막지 않게 한다.

## 12. 캐시와 갱신

- 지역·기상 발표 회차·예보 날짜를 캐시 키로 사용한다.
- 같은 발표 회차의 지역 예보는 여러 플랜에서 재사용한다.
- 실패 응답은 장기 캐시하지 않는다.
- 새 발표 회차가 확인되면 기존 리포트를 오래된 상태로 표시한다.
- 사용자가 플랜을 수정하면 기상 원본은 재사용하되 영향·변경안만 다시 계산한다.

## 13. 알림 원칙

알림은 다음 세 시점으로 제한한다.

- 플랜 생성 직후 첫 판단
- 예보가 바뀌어 기존 일정과 새로 충돌했을 때
- 출발 전 공식 운영·교통·날씨 재확인

단순 기온 변화나 동일 경고를 반복하지 않는다. 사용자가 기존 일정을 유지하면 위험 단계가 높아졌을 때만 다시 알린다.

## 14. 테스트 전략

### 백엔드

- 시간별 예보의 오전·오후·저녁 집계
- 여러 지역의 부분 성공
- 장소 유형별 영향 등급
- 고정 일정 제외
- 플랜 내부 교환 우선
- 예보 범위 밖 변경안 금지
- 같은 입력의 결정론적 proposal fingerprint
- 기상청 장애 시 구조화된 판단 보류

### 프론트

- 리포트의 상태·출처·발표 시각 표시
- 미리보기와 실제 적용 결과 일치
- 사용자 승인 전 상태 불변
- 일부 operation 실패 시 전체 적용 취소
- 적용 후 되돌리기
- 플랜 버전이 바뀐 오래된 제안 거부
- 사용자가 유지한 제안 반복 숨김
- 모바일·키보드·스크린리더 접근성

### 회귀

- 기존 `/pack.weather` 계약 유지
- 기존 Trust Engine의 날씨 점수 회귀 없음
- PDF와 하루방 날씨 요약이 새 리포트와 모순되지 않음
- 기상청 키가 없어도 서버와 플랜 기능 정상 동작

## 15. 단계적 출시

### 1차

- 다중 지역·시간대별 기상 집계
- 일정 영향 리포트
- 기존 플랜 내 순서·날짜 변경안
- 승인·적용·되돌리기

### 2차

- 하루방을 통한 다른 대안 검색
- 공식 특보·배편·통제 정보의 별도 확인 링크
- 로그인 플랜의 서버 버전·감사 이력
- 예보 의미 변화에 따른 선택적 알림

### 3차

- 실제 여행 피드백을 통한 판단 규칙 평가
- 사용자 조건별 준비물과 이동 여유 개인화
- 운영 지표 기반 임계값 조정

## 16. 성공 지표

- 날씨 리포트 조회 대비 변경안 확인률
- 변경안 확인 대비 적용·유지 선택률
- 적용 후 즉시 되돌림 비율
- 날씨로 인한 일정 실패 피드백 비율
- 예보 누락 상태에서 잘못 생성된 변경안 0건
- 사용자 승인 없이 변경된 플랜 0건
- 여러 지역 여행에서 지역별 예보 적용률
- 리포트 API 응답시간과 기상청 부분 실패율

## 17. 최종 결정

날씨 고도화의 중심은 예보 정보량이 아니라 **사용자 플랜과 연결된 판단 가능성**이다. 서비스는 기상 근거와 일정 영향을 설명하고, 실행 가능한 변경안을 미리 보여준다. 최종 선택과 적용 권한은 항상 사용자에게 남긴다.
