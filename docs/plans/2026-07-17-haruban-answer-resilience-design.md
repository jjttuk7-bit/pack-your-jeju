# 하루방 답변 복원력 설계

## 목표

하루방이 웹 조사를 이미 마친 뒤 최종 LLM 호출에 실패하더라도 확보한 근거로 답변하고, 긴 대화가 API의 30개 메시지 제한을 넘지 않도록 한다.

## 범위

이번 구현은 두 가지 장애를 순서대로 해결한다.

1. 사전 검색 결과가 있는 상태에서 최종 LLM 호출이 예외 또는 빈 선택 결과로 끝나면 도구 결과 기반 안전 답변을 반환한다.
2. 프런트가 하루방 API에 전달하는 대화 기록을 최근 24개 사용자·어시스턴트 메시지로 제한한다.

예측하기 어려운 질문 전체를 키워드로 나열하지 않는다. 후속 작업에서 제주 사실 질문의 기본 웹 조사, 짧은 후속 질문의 문맥 복원, 구조화된 장기 대화 상태를 추가할 수 있도록 기존 검색 풀과 `form_state` 경계를 유지한다.

## 설계

### 1. 검색 결과 보존 폴백

`_chat_turn_raw`의 최종 모델 호출에서 예외가 발생하거나 선택 결과가 없을 때 `preloaded_tool`이 있으면 `_fallback_reply_from_tool_messages`로 답변을 조립한다. 반환값은 `available=True`로 두되 `reason`에 최종 모델 장애와 폴백 사용 사실을 기록한다. 기존 `answer_contract`, `place_candidates`, `tool_trace`를 그대로 보존한다.

사전 검색 결과가 없을 때는 현재처럼 `available=False`를 반환한다. 검색되지 않은 사실을 새로 만들지 않기 위해서다.

### 2. 최근 대화 기록 제한

프런트에 순수 함수 `buildHarubanChatHistory`를 두어 화면 엔트리 중 사용자·어시스턴트 메시지만 변환한 후 마지막 24개를 반환한다. `intro` 카드 엔트리는 지금처럼 제외한다.

24개는 API 최대치 30보다 6개 여유가 있어 추후 서버 측 보조 메시지가 추가되어도 즉시 한도에 닿지 않는다. 여행 지역·동행자·목적·기간은 별도의 `form_state`로 계속 전달된다.

## 오류 처리

- 사전 검색 있음 + 최종 LLM 장애: 검색 결과 기반 답변, `available=True`
- 사전 검색 없음 + 최종 LLM 장애: 기존대로 `available=False`
- 24개 초과 대화: 가장 오래된 메시지를 버리고 최신 24개 전송
- 검색 결과 자체가 불충분함: 기존 투명한 “원문을 확보하지 못함” 폴백 유지

## 테스트

- Python 회귀 테스트에서 정상적인 웹 검색 풀을 준비하고 최종 모델 호출 예외를 발생시켜 폴백 답변과 메타데이터 보존을 검증한다.
- 선택 결과가 없는 모델 응답도 같은 폴백을 사용하는지 검증한다.
- TypeScript 단위 테스트에서 30개 이상의 대화 엔트리를 넣고 최신 24개만, 올바른 순서와 역할로 반환되는지 검증한다.
- 집중 테스트 후 API 전체 테스트, 웹 전체 테스트, `tsc --noEmit`, `git diff --check`를 실행한다.
