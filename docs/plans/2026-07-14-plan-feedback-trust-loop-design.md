# 제주를 담다 플랜·방문 피드백 신뢰 순환 고도화 설계

- 작성일: 2026-07-14
- 상태: 승인됨
- 범위: 여행 플랜, 사용자 직접입력, 방문 피드백, 공공데이터 보정, 운영자 검증

## 1. 배경

제주를 담다는 공공데이터 후보, 하루방이 웹에서 조사한 후보, 사용자가 직접 입력한 장소를 여행 플랜에 담을 수 있다. 현재 구현에는 웹검색 후보와 직접입력 장소의 플랜 추가, 방문 후 피드백, 신뢰도 변화 표시의 기반이 있지만, 사용자 경험이 다음 사용자의 추천 근거로 연결되는 운영형 신뢰 순환은 완성되지 않았다.

이번 고도화의 목표는 공공데이터 원본을 사용자 제보로 직접 덮어쓰지 않으면서도 실제 방문 경험을 최신 현장 근거로 축적하는 것이다. 사용자의 플랜과 방문 피드백은 웹 근거 및 공공데이터와 분리해 보존하고, 검증된 보정 정보만 하루방의 다음 조사와 추천에 사용한다.

## 2. 확정된 제품 원칙

1. 사용자 피드백은 현장 신뢰도에 즉시 반영한다.
2. 공공데이터 원본은 수정하지 않는다.
3. 공공데이터 보정 정보는 반복 제보, 웹 재검증, 운영자 승인을 거쳐 별도 계층에 기록한다.
4. 비회원도 플랜을 만들 수 있지만 저장, 공유, 기기 간 동기화, 피드백 제출은 로그인이 필요하다.
5. 사용자가 직접 입력한 장소는 개인 플랜에 즉시 추가한다. 다른 사용자에게는 장소 존재, 주소, 중복 여부를 검증한 뒤 `커뮤니티 확인 후보`로 공개한다.
6. 위치, 사진, 영수증은 선택적 증빙이다. 증빙이 있으면 피드백 가중치를 높이지만 제출 필수 조건으로 삼지 않는다.
7. 시스템은 제보를 자동 분류하고 근거를 모으되, 공공데이터 보정의 최종 결정은 운영자가 내린다.
8. 1차 고도화는 플랜 작성부터 방문 피드백, 운영자 검증, 다음 추천 반영까지 하나의 신뢰 순환을 완성하는 데 집중한다.

## 3. 검토한 접근법

### 3.1 기존 신뢰점수 확장

현재 `trust_score`에 방문 피드백 점수를 더하거나 빼는 방식이다. 구현은 빠르지만 점수의 원인과 변경 이력을 설명하기 어렵고, 중요한 사실 변경과 개인적 경험이 같은 점수에 섞인다. 운영형 신뢰 체계로 채택하지 않는다.

### 3.2 근거 원장 기반 신뢰 순환

공공데이터, 웹 출처, 사용자 입력, 방문 피드백, 운영자 판정을 서로 덮어쓰지 않고 별도 근거로 보존한다. 현재 기능을 재사용할 수 있고 모든 판단의 출처와 이력을 추적할 수 있어 채택한다.

### 3.3 장소 지식 그래프

장소, 주장, 사용자, 방문, 출처 관계를 그래프로 구성하는 장기 확장안이다. 정교하지만 현재 단계에서는 기반 구축 비용이 크다. 근거 원장을 먼저 운영한 뒤 관계 탐색 요구가 충분히 쌓일 때 검토한다.

## 4. 목표 사용자 흐름

```text
비회원 플랜 작성
  -> 공공데이터 후보 담기
  -> 하루방 웹검색 후보 담기
  -> 사용자 장소 직접 추가
  -> 로그인 후 저장·공유
  -> 실제 방문
  -> 구조화 피드백과 선택 증빙 제출
  -> 현장 신뢰도 갱신
  -> 반복 제보·웹 근거 재확인
  -> 운영자 승인·보류·기각
  -> 보정 정보와 하루방 추천 근거 갱신
  -> 다음 사용자의 플랜 품질 향상
```

### 4.1 플랜 장소의 출처 표시

- `public_data`: 내부 공공데이터에서 선택한 장소
- `web_search`: 하루방이 웹 원문을 확인해 제안한 장소
- `user_input`: 사용자가 개인 플랜에 직접 입력한 장소
- `community_verified`: 사용자 입력 후 장소 존재와 주소가 검증된 공용 후보

모든 플랜 장소는 당시의 이름, 주소, 좌표, 출처 URL, 확인 시점, 선택 이유를 스냅샷으로 보존한다. 이후 장소 정보가 바뀌어도 여행 당시 사용자가 어떤 근거로 선택했는지 복원할 수 있어야 한다.

### 4.2 방문 피드백

사용자는 장소별로 다음 정보를 남긴다.

- 방문함, 방문하지 못함, 장소를 찾지 못함
- 정상 영업, 휴무, 폐업 의심, 이전 의심
- 운영시간, 가격, 주소 등 사실 정보의 불일치
- 혼잡도, 접근성, 동행자 적합성 등 경험 평가
- 자유 메모
- 선택적 위치, 사진, 영수증 증빙

피드백 제출 후에는 `현장 신호에 반영되었으며 공공데이터 보정은 추가 확인과 운영자 검토 후 진행된다`는 처리 범위를 명확히 알린다.

## 5. 데이터 모델

### 5.1 핵심 엔터티

| 엔터티 | 책임 |
|---|---|
| `UserProfile` | 계정, 동의, 정상 활동과 제한 상태 |
| `TravelPlan` | 여행 기간, 지역, 동행자, 공개 범위, 소유권 |
| `PlanItem` | 플랜 장소, 출처, 일정, 선택 이유, 당시 정보 스냅샷 |
| `Place` | 여러 출처에서 발견한 동일 장소를 묶는 기준 레코드 |
| `Evidence` | 특정 주장, 출처 역할, 확인 시점, 지원 상태 |
| `VisitFeedback` | 방문 결과, 사실 변경 제보, 경험 평가, 작성 시점 |
| `EvidenceAsset` | 위치, 사진, 영수증 등 비공개 선택 증빙 |
| `ModerationCase` | 신규 장소, 반복 제보, 출처 충돌의 검토 작업 |
| `ModerationDecision` | 승인, 보류, 기각, 취소와 판단 근거의 버전 이력 |
| `PlaceTrustProfile` | 장소 일치, 운영정보, 최신성, 현장 신뢰의 파생 상태 |
| `PublicDataCorrection` | 공공데이터 원본을 보존한 채 제공하는 승인된 보정 정보 |

### 5.2 원본과 파생 상태 분리

```text
공공데이터 원본 스냅샷
+ 공식 웹 근거
+ 플랫폼 확인 정보
+ 사용자 현장 신호
+ 운영자 승인 보정
= 사용자에게 보여줄 현재 정보
```

원본 이벤트와 피드백은 변경 불가능한 기록으로 취급한다. 잘못된 내용을 바로 덮어쓰지 않고 정정 이벤트 또는 운영자 판정의 새 버전을 추가한다. `PlaceTrustProfile`은 원본 근거에서 다시 계산할 수 있는 파생 데이터다.

### 5.3 신뢰 프로필

단일 총점만으로 신뢰를 표현하지 않는다.

- `identity_confidence`: 장소명, 주소, 좌표가 동일 장소를 가리키는 정도
- `operation_confidence`: 영업, 휴무, 가격, 운영시간 정보의 근거 수준
- `freshness_status`: 마지막 확인 시점과 정보 유형별 유효 기간
- `field_confidence`: 최근 방문자 피드백의 반복성과 증빙 수준

내부 계산에 숫자를 사용할 수 있지만 사용자에게는 `최근 공식 확인`, `현장 제보 반복`, `출처 충돌`, `추가 확인 필요`처럼 이유를 함께 표시한다.

### 5.4 사용자 피드백 가중치

가중치는 다음 요소를 사용한다.

- 로그인 계정과 정상 활동 여부
- 실제 플랜에 포함됐던 장소인지
- 선택 증빙의 종류와 유효성
- 독립적인 다른 제보와의 일치
- 공식·플랫폼 출처와의 일치 또는 충돌
- 반복 복사, 대량 제출, 비정상 위치와 같은 이상 신호

가중치는 현장 신뢰도와 검토 우선순위를 정한다. 높은 가중치만으로 보정 정보를 자동 승인하지 않는다.

## 6. 장소 정규화와 사용자 직접입력

사용자 직접입력은 개인 플랜에 즉시 저장한다. 공용 후보 승격 전 다음 순서로 확인한다.

1. 장소명 정규화와 별칭 비교
2. 주소의 도로명·지번 정규화
3. 좌표 거리와 행정구역 비교
4. 기존 공공데이터 및 웹 후보와 유사도 비교
5. 확실한 일치면 기존 `Place`에 연결
6. 불확실하면 새 장소를 만들지 않고 `장소 확인 대기` 검토 건 생성
7. 존재와 주소가 확인되면 `community_verified` 후보로 승격

공공데이터에 없다는 이유만으로 사용자 장소를 제거하지 않는다. 웹에서 존재를 확인할 수 있지만 내부 데이터에 없으면 `공공데이터 미확인` 상태로 유지한다.

## 7. 운영자 검증 흐름

1. 피드백 원문과 플랜 스냅샷 저장
2. 개인정보, 악성 콘텐츠, 이상 제출 점검
3. 피드백을 사실 변경과 경험 의견으로 분류
4. 현장 신뢰도 갱신
5. 폐업, 이전, 주소 변경 등 중요 제보의 웹 재검색
6. 반복 제보 또는 출처 충돌 시 `ModerationCase` 생성
7. 운영자가 공공데이터, 공식, 플랫폼, 사용자 증빙을 함께 검토
8. 승인, 보류, 기각 중 하나로 판정
9. 승인 시 `PublicDataCorrection` 새 버전 생성
10. 하루방의 교차확인과 추천 근거 갱신
11. 제보자에게 처리 결과 알림

관리자 화면은 제보 수만 보여주지 않는다. 주장별 공공데이터 값, 최신 공식·플랫폼 출처, 사용자 제보 분포, 선택 증빙, 이전 판정을 나란히 표시한다.

## 8. 기술 구성

```text
Vercel
  -> React/Vite 웹과 스마트폰 UI

Railway
  -> FastAPI
     - 하루방 웹 리서치
     - 장소 정규화와 중복 판별
     - 근거·신뢰 계산
     - 피드백 검증
     - 운영자 검토 API

Supabase
  -> PostgreSQL + pgvector
  -> Auth
  -> Row Level Security
  -> 비공개 증빙 Storage
```

FastAPI와 장시간 웹 리서치, PDF 생성 작업은 Railway에 유지한다. 회원, 사용자별 권한, 장기 데이터, 비공개 파일은 Supabase를 사용한다. 기존 Railway PostgreSQL은 단번에 폐기하지 않고 복제 환경에서 확장 호환성, 연결 풀, 지연, 백업과 복구를 검증한 뒤 단계적으로 전환한다.

Supabase 도입 전에도 도메인 모델과 API 계약을 먼저 구축해 특정 저장소에 강하게 결합하지 않는다. 전환 실패 시 Railway PostgreSQL로 되돌릴 수 있는 마이그레이션과 롤백 절차를 갖춘다.

## 9. 개인정보와 조작 방지

### 9.1 개인정보

- 위치, 사진, 영수증 제출은 선택 사항이다.
- 증빙 원본은 다른 사용자에게 공개하지 않는다.
- 영수증의 이름, 전화번호, 카드번호 등은 자동 가림 처리한다.
- 사진 위치정보는 검증 후 제거하거나 별도 보호한다.
- 운영자는 필요한 검토 건의 최소 정보만 볼 수 있다.
- 사용자는 피드백과 증빙 삭제를 요청할 수 있다.
- 삭제 후에는 개인 식별이 불가능한 집계만 유지할 수 있다.

### 9.2 조작 방지

- 동일 계정, 기기, 장소에 대한 과도한 반복 제출 제한
- 짧은 시간의 대량 제출과 동일 문구 반복 탐지
- 신규 계정의 초기 가중치 제한
- 이해관계자의 자기 장소 제보 가능성 표시
- 신고 누적만으로 폐업, 이전 등 중요 상태를 자동 확정하지 않음
- 승인과 기각, 승인 취소 이력을 모두 보존

## 10. 오류 처리

| 상황 | 처리 |
|---|---|
| 웹 재검색 실패 | 공공데이터로 조용히 대체하지 않고 `외부 근거 확인 실패` 표시 |
| 위치 검증 실패 | 피드백은 보존하고 증빙 없는 현장 신호로 처리 |
| 사진 업로드 실패 | 텍스트 피드백을 먼저 보존하고 재업로드 제공 |
| 장소 중복 판별 불확실 | 신규 공용 장소를 만들지 않고 `장소 확인 대기`로 전환 |
| 인증·DB 장애 | 비회원 플랜을 기기에 임시 보존하고 복구 후 명시적으로 동기화 |
| 신뢰 계산 실패 | 이전 프로필을 유지하고 재계산 작업을 등록 |
| 운영자 판단 충돌 | 자동 승인하지 않고 추가 근거 요청 상태 유지 |

부분 실패가 사용자 원문을 잃게 해서는 안 된다. 외부 검색, 파일 저장, 신뢰 계산은 독립된 상태로 기록하고 재시도한다.

## 11. 단계별 로드맵

### 11.1 플랜과 출처 표준화

- 세 출처의 장소를 동일한 `PlanItem` 계약으로 통합
- 플랜 스냅샷과 출처 이력 보존
- 비회원 플랜과 로그인 계정 연결
- 저장, 공유, 기기 간 동기화
- 직접입력 장소의 확인 대기 상태

완료 기준: 세 종류의 장소가 한 플랜에서 구분되고 데이터 손실 없이 저장된다.

### 11.2 방문 피드백과 근거 원장

- 사실 변경과 경험 평가를 분리한 피드백
- 선택 증빙과 개인정보 보호
- 변경 불가능한 피드백 원장
- 현장 신뢰도와 이상 제출 탐지

완료 기준: 방문 경험이 공공데이터 원본을 건드리지 않고 별도 근거로 축적된다.

### 11.3 운영자 검증과 하루방 재연결

- 반복 제보와 중요 변경 자동 분류
- 하루방 웹 재검색
- 관리자 근거 비교와 판정
- 승인 취소와 신뢰 상태 재계산
- 승인된 보정의 답변·추천 반영
- 제보자 결과 알림

완료 기준: 사용자 피드백 하나가 검증을 거쳐 다음 사용자의 설명 가능한 추천 근거로 연결된다.

### 11.4 커뮤니티·공공데이터 협력 확장

- 검증된 직접입력 장소의 커뮤니티 공개
- 지역별 오류, 누락, 변경 가능성 리포트
- 공공기관 전달용 보정 후보 내보내기
- 기관 처리 결과의 재수집

1차 출시 범위는 11.1부터 11.3까지다. 11.4는 데이터와 운영 경험이 축적된 뒤 진행한다.

## 12. 성공 지표

- 플랜에서 공공데이터, 하루방 검색, 직접입력 장소가 함께 사용되는 비율
- 저장된 플랜이 실제 방문 피드백으로 이어진 비율
- 직접입력 장소가 검증을 통과하는 비율
- 중요 제보가 운영자 판단에 도달하는 시간
- 승인된 보정의 취소·오판 비율
- 하루방 답변에서 최신 근거가 연결된 장소 비율
- 승인된 보정 정보가 이후 추천에 사용된 횟수
- 피드백 작성 사용자의 재방문율

가입자 수나 피드백 총량만으로 성공을 판단하지 않는다. 검증 가능한 현장 근거가 실제 다음 선택을 개선했는지를 중심으로 본다.

## 13. 테스트 전략

### 13.1 계약과 단위 테스트

- 모든 `PlanItem` 출처의 직렬화와 스냅샷 보존
- 피드백 분류와 신뢰 가중치 계산
- 중요 사실 제보의 자동 승인 금지
- 장소 정규화와 불확실한 중복 처리
- 운영자 판정 버전과 승인 취소 재계산

### 13.2 통합 테스트

- 비회원 플랜의 로그인 계정 연결과 중복 방지
- 공공데이터, 웹검색, 직접입력 장소가 섞인 플랜 저장
- 사진 저장 실패 뒤 텍스트 피드백 보존
- 검색, DB, 파일 저장소의 부분 실패
- 사용자 삭제 요청 후 개인정보와 증빙 제거

### 13.3 신뢰 엔진 회귀 테스트

- 사용자 한 명의 폐업 제보가 확정 사실이 되지 않음
- 반복 제보와 최신 공식 출처가 충돌할 때 `conflicted` 처리
- 승인되지 않은 제보를 하루방이 사실처럼 말하지 않음
- 승인된 보정은 출처, 확인 시점, 운영자 판정과 함께 인용 가능
- 공공데이터 미수록을 장소 부재로 해석하지 않음

## 14. 비목표

- 1차 출시에서 사용자 투표로 공공데이터를 자동 수정하지 않는다.
- 피드백 총량만으로 장소의 절대 순위를 만들지 않는다.
- 증빙 제출을 모든 사용자에게 강제하지 않는다.
- 초기 단계에서 별도 그래프 데이터베이스를 도입하지 않는다.
- 하루방의 FastAPI와 장시간 작업을 Supabase Edge Function으로 재작성하지 않는다.

## 15. 출시 원칙

각 단계는 실패 테스트를 먼저 추가하고, 기존 플랜·신뢰 엔진 계약을 하위 호환으로 확장한다. 운영자 검토가 준비되기 전에는 중요 제보를 수집하더라도 사용자 추천에 확정 사실로 반영하지 않는다. 데이터 마이그레이션에는 검증 리포트, 백업, 롤백 절차가 반드시 포함되어야 한다.
