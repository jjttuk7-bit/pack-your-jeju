# Pack Your Jeju 경진대회 제품·서비스 개발 기획안

> 목적: 아이펠톤 데모를 안정판으로 유지하면서, 2026년 제주 공공데이터·AI 활용 창업경진대회 제품·서비스 개발 부문에 맞는 고도화 방향을 확정한다.
> 기준일: 2026-07-06

## 1. 출품 포지션

Pack Your Jeju는 `제품·서비스 개발` 부문으로 준비한다.

근거:
- 이미 배포된 웹/API 시제품이 있다.
- 비짓제주, 콘텐츠수정요청, 주차장, 정류장 등 제주 공공데이터를 실제 DB에 적재해 사용한다.
- `/pack`, `/verify`, `/agent/intro`, `/agent/chat`, `/pack/pdf` 등 시연 가능한 기능이 있다.
- AI는 자유 생성기가 아니라 공공데이터 조회 결과를 조립하고 검증하는 신뢰형 에이전트로 사용한다.

아이펠톤에서는 "제주 공공데이터 기반 신뢰 RAG 여행팩"으로 발표하고, 경진대회에서는 이를 "계획-공유-방문기록-신뢰데이터 재순환 플랫폼"으로 확장한다.

## 2. 제품 한 줄 정의

공공데이터와 AI 에이전트를 활용해 제주 여행 계획을 만들고, 동행자와 공유하며, 실제 방문 기록을 다시 신뢰 데이터로 축적하는 제주 여행 신뢰 플랫폼.

## 3. 문제 정의

제주 여행자는 블로그, 리뷰, 지도, 공식 관광 포털을 오가며 계획을 세우지만, 여행 정보는 빠르게 낡는다.

현재 프로젝트가 확인한 실증 근거:
- 제주 공식 관광 콘텐츠 수정요청 1,686건
- 정제된 비짓제주 기반 장소 4,422건
- 주차장·정류장 접근성 데이터 5,828건
- 골든셋 게이트 12/12 통과

문제는 "추천 장소가 부족하다"가 아니다. 더 큰 문제는 사용자가 어떤 정보가 확인됐고, 어떤 정보가 오래됐고, 어떤 정보가 공공데이터 기준으로 확인되지 않는지 알기 어렵다는 점이다.

Pack Your Jeju의 핵심 문제 정의:
- 여행 전: 지역·동행자·목적에 맞는 검증 가능한 여행 계획이 필요하다.
- 여행 중: 사용자가 질문할 때마다 근거가 있는 상담이 필요하다.
- 여행 후: 실제 방문 경험이 다음 사용자의 신뢰 신호로 축적되어야 한다.

## 4. 핵심 원칙

### 4.1 LLM은 사실을 만들지 않는다

LLM은 장소명, 주소, 운영시간, 교통 접근성, 위생등급, 날씨 정보를 생성하지 않는다.

사실 정보는 반드시 다음 데이터 계층에서 조회한다:
- `place`
- `transit_point`
- 향후 `plan`
- 향후 `visit_checkin`
- 향후 기상/기후 공공데이터 테이블

LLM은 조회된 근거를 요약, 조립, 비교, 설명하는 역할만 한다.

### 4.2 모르면 모른다고 말한다

결과가 없을 때는 단순히 "정보 없음"이라고 말하지 않는다. 기존 Trust Engine의 fallback 4분기를 유지한다.

| fallback_reason | 의미 | 사용자 문구 원칙 |
|---|---|---|
| `out_of_scope` | 제주 여행 범위 밖 | "제주 여행 정보 범위 밖입니다" |
| `coverage_gap` | 참조 데이터에서 확인 불가 | "저희가 참조하는 공공데이터 기준으로 확인되지 않습니다" |
| `retrieval_miss` | 데이터는 있으나 검색 실패 가능성 | 완화 검색 후에도 실패하면 내부 로그로 남김 |
| `contradicted` | 폐업/이전/변경 반증 존재 | "폐업/변경이 확인됩니다" |

### 4.3 사용자 기록은 공공데이터와 분리한다

방문 기록은 중요한 신뢰 신호지만, 공공데이터 검증과 같은 배지로 섞지 않는다.

예:
- 공공데이터 확인됨
- 최근 사용자 방문 확인 12건
- 위치 기반 방문 확인 3건
- 정보 수정 의심 신고 2건

이렇게 공공데이터 기반 신뢰와 사용자 기반 신뢰를 분리해 보여준다.

## 5. 고도화 사용자 흐름

```text
계획하기
  -> 지역 줌인으로 데이터 커버리지 확인
  -> 하루방 근거 상담으로 정밀 조건 조정
  -> 여행팩 생성
  -> 공유 가능한 플랜 링크 생성
  -> PDF 저널 스냅샷 저장
  -> 여행 중/후 방문 기록
  -> 방문 기록이 다음 추천의 신뢰 신호로 축적
```

## 6. 핵심 기능 4축

### 6.1 지역 정밀 설정: "줌인하는 제주"

현재는 12개 지역 chip을 선택하면 끝난다. 경진대회 버전에서는 지역 선택 직후 해당 지역의 데이터 커버리지를 먼저 보여준다.

예: 사용자가 `구좌`를 선택한 경우
- 구좌 내 세부권역: 월정리, 세화리, 평대리 등
- moment별 커버리지: 오름, 바다 산책, 카페, 로컬푸드, 노을 등
- verified/caution/coverage_gap 비율
- 하루방 브리핑: "구좌는 바다 산책과 카페는 확인된 곳이 많지만, 노을 감상은 저희 공공데이터 기준으로 아직 부족합니다."

가치:
- 사용자가 팩 생성 후에야 빈 결과를 만나는 문제를 줄인다.
- Trust Engine의 정직함을 폼 단계에서 보여준다.
- 지역별 공공데이터 커버리지 자체가 서비스의 차별화 지표가 된다.

첫 API 후보:

```http
GET /region/coverage-preview?region=gujwa
```

응답 초안:

```json
{
  "region": "gujwa",
  "region_label": "구좌",
  "total_places": 412,
  "moments": [
    {
      "moment": "beach_walk",
      "moment_label": "바다 산책",
      "category": "beach",
      "verified": 24,
      "caution": 5,
      "coverage_gap": false
    },
    {
      "moment": "sunset",
      "moment_label": "노을 감상",
      "category": "viewpoint",
      "verified": 0,
      "caution": 0,
      "coverage_gap": true
    }
  ],
  "recommended_moments": ["beach_walk", "quiet_cafe", "local_food"],
  "weak_moments": ["sunset"],
  "briefing": "구좌는 바다 산책과 카페는 확인된 곳이 많지만, 노을 감상은 저희 공공데이터 기준으로 아직 부족합니다."
}
```

구현 원칙:
- LLM 없이 규칙 기반 briefing 템플릿으로 먼저 구현한다.
- DB count query 중심으로 p95 500ms 이내를 목표로 한다.
- `coverage_gap`은 "없다"가 아니라 "저희 데이터 기준으로 확인되지 않습니다" 문구를 사용한다.

### 6.2 하루방 근거 상담

하루방은 자유롭게 사실을 생성하는 챗봇이 아니다. 사용자 발화마다 필요한 도구를 호출해 공공데이터 근거를 가져오고, 그 결과만 자연어로 조립하는 에이전트다.

지원할 상담 예:
- "구좌 카페 말고 서귀포 카페도 보여줘"
- "이 오름 아이랑 가도 괜찮아?"
- "비 오면 대체할 수 있는 실내 후보 있어?"
- "이 장소를 2일차에 넣어줘"

도구 후보:
- `region_coverage_preview(region)`
- `search_places(regions, moments, companion, purpose)`
- `check_transit(place_id)`
- `explain_fallback(region, moment)`
- `add_place_to_plan(plan_id, place_id, day)`

응답 안전장치:
- 응답에 등장하는 place는 반드시 candidates 안에 있어야 한다.
- 후보 밖 장소명은 제거하거나 "확인되지 않습니다"로 바꾼다.
- 운영시간/주소/교통 정보는 DB 필드가 없으면 말하지 않는다.
- 매 턴 `fallback_reason`을 로그에 남긴다.

### 6.3 공유 가능한 플랜

현재 `/pack/pdf`는 정적 파일 생성이다. 경진대회 버전에서는 `/pack` 결과를 저장 가능한 여행 플랜으로 바꾼다.

플랜 상태:

| status | 의미 |
|---|---|
| `draft` | 생성 직후 초안 |
| `shared` | 동행자에게 링크 공유 |
| `confirmed` | 사용자가 확정 |
| `traveled` | 실제 방문 기록이 붙은 상태 |

첫 구현 범위:
- `plan` 테이블
- `POST /plans` 생성
- `GET /plans/{id}` 조회
- `GET /plan/{share_slug}` 공유 조회
- 기존 PDF는 저장된 plan 기준으로 생성

공동편집과 투표는 1차 MVP에서 제외한다. 발표와 사업계획서에는 다음 단계로 둔다.

### 6.4 실제 방문 기록

방문 기록은 Pack Your Jeju의 장기 경쟁력이다. 사용자가 여행 후 기록을 남기면, 그 기록은 다음 사용자에게 "최근 방문 신호"로 축적된다.

MVP:
- "다녀왔어요" 버튼
- 방문 날짜
- 한줄 메모
- 선택적으로 장소별 체크인
- plan 상태를 `traveled`로 변경

향후 확장:
- GPS 근접 확인
- 사진 첨부
- 정보 수정 의심 신고
- 방문 기록 기반 신뢰 신호

중요한 원칙:
- 사용자 체크인은 공공데이터 검증 배지를 대체하지 않는다.
- 체크인은 `user_visit_signal`로 분리해 보여준다.
- 신고/수정 제안은 바로 place를 수정하지 않고 검토 큐에 쌓는다.

## 7. 공공데이터 활용 전략

현재 사용 중:

| 데이터 | 현재 용도 | 경진대회 고도화 용도 |
|---|---|---|
| 비짓제주 | 장소/카페/음식/관광지 후보 | 지역별 coverage-preview, 플랜 생성 |
| 콘텐츠수정요청 | caution/contradicted 신호 | 데이터 신뢰 문제 실증, 개선 우선순위 |
| 주차장 | 교통 접근성 배지 | 장소별 접근성 설명 |
| 버스정류장 | 교통 접근성 배지 | 대중교통 가능성 설명 |
| 위생등급 | 음식점 배지 보강 예정 | local_food 신뢰도 강화 |

추가 검토:

| 데이터 | 기대 효과 | 우선순위 |
|---|---|---|
| 기상청/제주지방기상청 날씨·기후 | 특별상 대응, 여행 준비물/대체 일정 | 높음 |
| 관광약자 접근성 데이터 | 아이/부모님 동행 신뢰도 강화 | 중간 |
| 공공화장실/응급시설 | 가족·아이 여행 안전성 | 낮음 |
| 지역 행사/축제 데이터 | seasonal moment 강화 | 중간 |

기상 데이터는 실시간 경로 추천이 아니라, 여행 계획의 위험도와 준비물 제안에 사용한다.

예:
- 강풍: 오름 일정 caution
- 폭염: 아이/부모님 동행 일정 caution
- 우천: 실내/카페/시장 대체 moment 제안

## 8. AI/RAG 활용 정의

Pack Your Jeju의 AI는 "더 그럴듯하게 말하는 장치"가 아니다. 공공데이터 근거를 찾고, 그 근거만으로 답변을 구성하게 제한하는 인터페이스다.

### 8.1 `/pack`

구조화 검색 + LLM 문구 조립.

폼 입력이 이미 region, days, companion, purpose, moments로 구조화되어 있으므로 Self-Query LLM은 사용하지 않는다. 이는 기능 부족이 아니라 환각을 줄이기 위한 설계 판단이다.

### 8.2 `/verify`

정통 RAG 흐름.

```text
리뷰/문장 입력
  -> claim 분해
  -> place 검색
  -> 수정요청/tombstone/valid_until 근거 확인
  -> verified/coverage_gap/contradicted 판정
```

### 8.3 하루방 상담

대화형 RAG 흐름.

```text
사용자 발화
  -> 의도 파싱
  -> 도구 호출
  -> 공공데이터 후보 조회
  -> LLM 응답 조립
  -> 후보 밖 사실 제거
  -> query_log 기록
```

## 9. 기존 서비스와 차별성

| 기존 서비스 | 강점 | Pack Your Jeju의 차별점 |
|---|---|---|
| 지도 앱 | 장소 검색, 길찾기 | 공공데이터 검증 여부와 미확인 사유를 명시 |
| 블로그/리뷰 | 생생한 경험 | 리뷰를 근거로 쓰지 않고 검증 대상으로 다룸 |
| 일반 AI 여행 챗봇 | 자연어 대화 | LLM이 사실을 만들지 못하게 DB 후보로 제한 |
| 관광 포털 | 공식 정보 | 여행자 조건에 맞게 조립하고 빈틈을 정직하게 표시 |

차별화 문장:

> Pack Your Jeju는 많이 추천하는 서비스가 아니라, 확인된 것과 확인되지 않은 것을 구분해 여행자가 믿고 결정하게 하는 서비스다.

## 10. 사업화 방향

### 10.1 사용자 대상

- 무료: 기본 여행팩 생성, 하루방 상담, PDF 저장
- 프리미엄 후보: 동행자 공유, 여행 저널 아카이브, 방문 기록 앨범, 고급 체크리스트

### 10.2 지역 사업자 대상

- 장소 신뢰 페이지
- 수정요청/방문기록 기반 정보 갱신 알림
- 공공데이터와 사용자 방문 신호를 분리한 신뢰 리포트

### 10.3 지자체/기관 대상

- 지역별 coverage_gap 대시보드
- 자주 미확인되는 카테고리 분석
- 공공데이터 갱신 우선순위 제안

## 11. 아이펠톤과 경진대회 스코프 분리

| 구분 | 아이펠톤 | 경진대회 |
|---|---|---|
| 목표 | 4일 데모 완성도와 발표 임팩트 | 제품·서비스 개발 부문 출품 |
| 핵심 | 제주 데이터, 신뢰 답변, RAG 검증 | 신뢰 플랫폼, 공유 플랜, 방문 기록 순환 |
| 구현 범위 | 현재 배포 안정판 유지 | 안정판 위에 기능 확장 |
| 금지선 | 데모를 깨는 대규모 리팩토링 금지 | 기능 확장 가능하되 Trust Engine 원칙 유지 |

D-04의 "확장안은 다음 단계로만" 결정은 아이펠톤 4일 완주를 위한 결정이다. 경진대회 준비에서는 이 확장안을 별도 제품 로드맵으로 승격하되, 아이펠톤 안정판을 깨지 않는 순서로 진행한다.

## 13. 첫 구현 추천: coverage-preview

가장 먼저 구현할 기능은 `region/coverage-preview`다.

이유:
- 현재 DB와 필터 구조를 그대로 활용할 수 있다.
- 아이펠톤 3축인 제주 데이터, 신뢰 기반 답변, RAG/검색 구조를 모두 강화한다.
- 경진대회용 제품 비전인 "지역 줌인"을 가장 빠르게 화면에 보여준다.
- 하루방 상담과 플랜 저장의 기반 데이터가 된다.

작업 순서:

1. 백엔드 count query 함수 추가
2. `GET /region/coverage-preview` 엔드포인트 추가
3. 단위 테스트 작성
4. 프론트 지역 선택 패널에 preview 카드 추가
5. 하루방 인사에 지역 preview 연결
6. 스모크 데이터 저장
