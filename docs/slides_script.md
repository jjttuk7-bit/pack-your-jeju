## 1막 — 오프닝 · 킥3 "1,686건의 증거" (1.5분)

### 대사 스크립트 (30초)

> "여행 앱들은 다 비슷하게 말합니다. '가성비 좋은 맛집', '숨은 명소', '꼭 가봐야 할 곳'.
> 
> 그런데 그중 폐업한 곳이 얼마나 될까요?
> 
> 저희는 확인해봤습니다. **비짓제주 — 제주 공식 관광 포털**입니다. 여기에 이용자들이 직접 접수한 정보 수정요청이 **1,686건** 쌓여 있습니다.
> 
> 저희가 전부 분석했습니다."

### 화면 액션

- 슬라이드 1장 노출: `docs/kick3_stats.md`의 유형별 카운트 표
- 강조할 숫자:
  - **1,686건** — 총 수정요청
  - **235건** — 운영시간·휴무 변경 (13.9%)
  - **20건** — 폐업/영업종료 (그럼에도 아직 페이지 살아있음)
  - **26건** — 이전/위치 변경

### 다음 문장 (30초)

> "폐업 20건, 이전 26건, 운영시간 변경 235건, 주소 변경 91건 — **372건이 물리적 변화**입니다.
> 원래 있던 곳이 지금은 다른 상태라는 뜻입니다.
> 공식 데이터조차 이 속도로 낡습니다. 개인 블로그는 어떨까요?
> 
> 그래서 저희는 짐 싸기 앱에 **신뢰 엔진**을 심었습니다."

### 슬라이드 전환 시점

"신뢰 엔진을 심었습니다." 문장 마지막에 다음 슬라이드 (본편 데모)로 전환.

### 예상 Q&A

- **"1,686건이 절대적으로 큰 숫자인가?"** → "비율보다 성격이 중요합니다. 폐업·이전·시간 변경만 372건이 이용자 직접 접수. 이건 자동 크롤링이 못 잡는 것들입니다."
- **"블로그·리뷰 데이터를 대안으로 쓰면 안 되나?"** → "그것도 낡습니다. 저희 데모 킥1에서 리뷰가 어떻게 검증되는지 보여드립니다."

### 백업 조치

- 슬라이드 못 뜨면 → 노트북 화면에 표만 노출
- 통계 수치 헷갈리면 → **1,686 → 556 → 372** 세 숫자만 확실히 기억

## 2막 — 본편 · 제품 데모 (5분)

### 2-1. 팩 만들기 (구좌·연인·힐링·3일) — 1분

#### 대사

> "제주 여행 준비를 실제로 해보겠습니다. 지역은 **구좌**, 3일 여행, 연인과, 힐링. 담고 싶은 순간은 **오름·바다 산책·현지 맛집** 세 개."

#### 화면 액션

1. 지역 chip: 구좌 클릭 (하나만)
2. 날짜: 이번 주 이후, 2박 3일
3. 동행자: 연인
4. 목적: 힐링
5. 순간 카드 탭 이동 → 오름·바다 산책·현지 맛집 세 개 선택
6. **"근거 있는 팩 받기"** 클릭

#### 결과 노출

응답 시간 <1초. 각 순간별 섹션에 verified/caution 배지 + 🅿️ 주차 N + 🚌 정류장 근접 배지.

#### 대사 이어서

> "각 장소에 배지가 붙습니다. **초록색 '확인됨'**은 저희가 참조하는 공공데이터로 검증된 항목. **노란색 '수정요청 이력'**은 이용자들이 정보 정정을 요청한 이력이 있는 곳 — 방문 전 재확인 권장. 각 카드에 원본 데이터 링크가 있고, 근처 주차장 수와 정류장 근접 여부도 표시됩니다."

### 2-2. 정직함 장면 1 — 감귤 × 7월 (G05) — 40초

#### 대사

> "이번엔 **감귤 체험**을 넣어보겠습니다. 7월에 감귤을 딸 수 있을까요?"

#### 화면 액션

- 팩 재구성: 순간 카드에 `감귤밭 체험` 추가 → 다시 팩 생성
- 감귤 섹션이 뜨면서 fallback 표시:
  ```
  이 지역에는 지금 시즌에 확인된 감귤 체험 프로그램이 없어요.
  (또는 coverage_gap 문구)
  ```

#### 대사 이어서

> "**저희는 7월에 감귤 체험을 지어내지 않습니다.** 감귤 수확은 10월부터 1월. 지금 시즌엔 데이터가 없다는 사실 그대로를 말씀드립니다. 이게 저희가 다른 여행 앱과 다른 지점입니다."

### 2-3. 정직함 장면 2 — 요일별 일정 뷰 (NEW) — 1분

#### 대사

> "그런데 실제 여행은 지역 하나만 다니지 않죠. 여러 지역을 나눠서. 그래서 저희는 지역 다중 선택도 지원하고, **선택된 것들을 요일별로 자동 조립**해드립니다."

#### 화면 액션

1. 리셋
2. 지역: **애월 · 성산 · 서귀포** 세 개 선택 (다중 선택 UI로)
3. 3일 여행, 연인, 힐링, 오름·바다 산책·현지 맛집
4. "근거 있는 팩 받기" 클릭
5. 결과 화면 상단 뷰 스위처에서 **"요일별로 (Day 1-3)"** 탭 클릭

#### 결과

```
Day 1 (7/8, 수) — 📍 애월
  [오름 - 바굼지오름] · [바다 산책 - 곽지해수욕장] · [현지 맛집 - ...]
Day 2 (7/9, 목) — 📍 성산  
  [오름 - 궁대오름] · ...
Day 3 (7/10, 금) — 📍 서귀포
  ...
```

#### 대사 이어서

> "**규칙 기반 자동 배치**입니다. LLM이 없는 걸 지어내는 게 아니라, 저희가 검증한 조각을 지역별로 요일에 배치한 겁니다. **없는 지역·순간 조합은 정직하게 '확인되지 않았다'고 말합니다.** 이건 저희 시스템이 판단이 아니라 계산으로 조립한다는 증거입니다."

### 2-4. 킥1 라이브 — AI가 AI를 팩트체크 (하이라이트) — 1.5분

#### 대사

> "다른 여행 앱은 GPT에게 물어보면 그럴듯하게 답합니다. 그런데 그중 얼마가 사실일까요? 저희 검증기에 리뷰를 하나 붙여넣겠습니다."

#### 화면 액션

- 상단 메뉴에서 **검증(/verify)** 페이지 이동
- 대본 텍스트 붙여넣기 (`data/verify_kick1_g14.json` 내용):
  ```
  애월오누이 제주 정말 맛있어요, 강추합니다.
  라임오렌지카페앤플라워는 조용해서 좋았어요.
  그 옆 가짜식당은 별로였어요.
  ```
- **검증 시작** 클릭
- 응답 (0.6초):

  1. `애월오누이 제주 정말 맛있어요, 강추합니다.` **× 폐업/변경 확인** — 폐업이 확인됩니다
  2. `라임오렌지카페앤플라워는 조용해서 좋았어요.` **✓ 확인됨** — 공공데이터에서 존재 확인
  3. `그 옆 가짜식당은 별로였어요.` **⚠ 확인되지 않음** — 저희 데이터로 확인되지 않습니다

#### 대사 이어서

> "이 리뷰의 첫 번째 문장 — **이 식당, 폐업이 확인됩니다.**
> 세 번째 문장 — '가짜식당'이라는 곳은 저희 데이터에 없습니다. **없다고 단언하지 않고 '확인되지 않는다'고만 말합니다.** 있을 수도 있으니까요. 저희 데이터 밖일 뿐입니다.
> 
> **저희 LLM은 사실을 만들 권한이 없습니다.** DB가 사실을 대고, LLM은 조립만 합니다."

### 2-5. 아키텍처 1장 — 30초

#### 슬라이드 (다이어그램)

```
사용자 폼 입력 (regions · moments · days · companion · purpose)
           ↓
filters.py (LLM 없음, 매핑 테이블)
           ↓
search.py (strict → relaxed 완화 재시도)
           ↓
trust.py (fallback 4분기 + 배지 4종)
           ↓
assemble.py (템플릿 폴백 or LLM 감성 문구 · dispatch_itinerary)
           ↓
query_log → /admin/metrics 라이브
           
────── 게이트 ──────
packages/eval/run.py : verified 1.0 · fallback 1.0 · badge 1.0 (12/12)
미달 시 exit(1) — 배포 물리적으로 차단
```

#### 대사

> "이게 저희 신뢰 엔진입니다. **골든셋 12문항 게이트가 통과하지 못하면 코드는 배포되지 않습니다.** 12문항 중 4문항은 '답하지 않아야 정답'인 케이스입니다. 방금 보신 감귤 시나리오와 폐업 시나리오가 여기 포함되어 있습니다."

### 예상 Q&A (본편)

- **"챗봇 안 만든 이유?"** → "챗봇은 '뭔가를 지어내는' 압박이 있습니다. 폼은 검증 필터를 이미 제공하므로 LLM이 사실을 만들 권한을 안 가져도 됩니다."
- **"이게 RAG인가요?"** → "네. `/verify` 파이프라인이 정통 RAG입니다: LLM이 문장을 분해 → 저희 DB에서 근거를 찾음 → LLM이 다시 판정. 사실 생성은 없습니다. `/pack`은 구조화 검색 + LLM 조립입니다."
- **"교통 배지는 어떻게 검증하나요?"** → "공공데이터포털 공영주차장 CSV(1,557건, 2026-04-16 갱신)와 국토부 TAGO 버스정류소 API(4,271개)를 지역별로 매칭합니다. 실시간 잔여 대수는 판단하지 않습니다 — 위치만 검증."
- **"카카오/네이버는요?"** → "저희는 근거로 인정하지 않습니다. 대신 검증기(/verify)에 입력값으로 받아 팩트체크는 해드립니다. 방금 보신 그 데모입니다."

### 백업 조치

- 팩 생성 실패 → 로컬 백업 노트북(uvicorn) 전환
- verify 응답 지연 → 사전 준비한 스크린 녹화 재생
- 애월오누이 시나리오가 안 나오면 → 슬라이드에 캡처 이미지 미리 삽입

> - 비짓제주 관광지·음식점 **4,422건**

### 라이브 대시보드 전환 (30초)

- 지금까지 최근 요청 수·배지 분포·fallback 사유·p50/p95 지연 노출

#### 대사

> "**방금 여러분이 만든 팩들입니다.**
> 요청 N건, verified 비율 N%, coverage_gap N건, 지연 p95 약 30ms.
> **저희는 발표하는 동안에도 운영 중이었습니다.**"

### 마지막 슬라이드 — 다음 단계 (30초)

```
다음 단계 (Roadmap)

1. 지역 팩 아키텍처 확장 — 부산 팩 · 도쿄 팩
   데이터 어댑터만 갈아끼우면 됩니다

2. 여행 일정 전체 검증
   개별 조각뿐 아니라 사용자가 만든 일정 전체를 팩트체크

3. 다국어 UI (비짓제주 en/cn/jp locale 활용)
```

#### 대사

> "저희는 **모든 곳을 아는 척하는 대신, 제주부터 완벽하게 하기로 했습니다.**
> 
> 근거가 있는 것만 말합니다. 최신 정보만 답합니다. 지어내지 않습니다.
> 
> Pack Your Jeju입니다. 감사합니다."

### 예상 Q&A (클로징)

- **"왜 제주만?"** → "지역 팩 아키텍처입니다. 데이터 어댑터만 갈아끼우면 부산 팩, 도쿄 팩이 됩니다. 저희는 첫 번째 전문화 지역으로 제주를 선택했습니다."
- **"확장 계획 시점?"** → "이 프로젝트는 아이펠톤 4일 스코프였습니다. 지역 확장은 데이터 소스 어댑터 층만 추가하는 작업입니다."
- **"수익 모델은?"** → "저희 관심은 '지어내지 않는 여행 정보'라는 기술적 자산 자체입니다. 이 자산을 B2B 여행사 검증 API나 지자체 관광포털에 붙일 수 있습니다."

### 백업 조치

- `/admin/metrics` 응답 지연 → JSON 미리 캡처해서 슬라이드 이미지로
- QR 스캔 실패 → URL 직접 노출 (`pack-your-jeju.vercel.app`)
- 심사위원이 실제로 팩을 안 만들어도 → 사전 데이터로 회전 중인 지표 노출
