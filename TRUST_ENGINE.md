# TRUST_ENGINE.md - 하루방 리서치·신뢰 엔진 스펙

이 문서는 `apps/api/engine/`의 검색, 출처 평가, 공공데이터 교차확인, 답변 조립에 대한 권위 스펙이다.

핵심 원칙은 **웹을 충분히 조사하고, 주장마다 근거의 역할과 최신성을 평가하며, 공공데이터는 마지막 교차확인에 사용한다**이다.

## 1. 전체 파이프라인

```text
사용자 질문 + 대화 이력 + 선택 폼
  -> context.py     : 지역·기간·동행자·취향·후보·제외 조건·검색 의도 병합
  -> research.py    : 검색 계획, 검색어 확장, 공식·플랫폼·경험 출처 수집
  -> evidence.py    : 원문 확인, 출처 분류, 주장 추출, 최신성·충돌 평가
  -> ranking.py     : 사용자 조건과 근거 강도로 후보 비교
  -> crosscheck.py  : 선택 지역 내부 공공데이터 후순위 교차확인
  -> assemble.py    : 결론·이유·비교·주의점·인용·교차확인 조립
  -> logging.py     : 검색 과정, 주장 지원, 충돌, 품질과 지연 기록
```

현재 파일 구조가 다르더라도 구현은 이 책임 경계를 지켜야 한다. 작은 서비스 단계에서는 한 모듈에 있을 수 있지만, 검색 결과와 내부 DB 결과를 같은 의미의 후보 풀로 섞지 않는다.

## 2. 질문 분류와 검색 의무

### 웹 검색이 필수인 질문

- 가볼 만한 지역, 관광지, 오름, 해변 추천
- 맛집, 카페, 메뉴, 분위기, 인기 장소 추천
- 운영시간, 휴무, 가격, 예약, 폐업·이전 여부
- 현재 행사, 축제, 날씨 영향, 계절성 정보
- “가장”, “요즘”, “현지인”, “후기가 좋은”처럼 현재 대중 신호가 필요한 질문
- 이전 후보의 비교, 추가 추천, 제외 후 재추천

### 웹 검색 없이 답할 수 있는 질문

- 이미 확인된 대화 근거만 요약하거나 형식을 바꾸는 요청
- 일반적인 여행 계획 방법처럼 현재 장소 사실이 필요 없는 질문
- 사용자가 명시적으로 내부 공공데이터만 요청한 질문

분류는 현재 메시지만 보지 않는다. 직전 웹 검색 질문에 이어지는 “오름은?”, “그중 맛집은?”도 웹 검색 질문으로 유지한다.

## 3. 검색 계획과 재시도

1. 대화에서 지역, 세부 지역, 카테고리, 일정, 동행자, 취향, 제외 후보를 추출한다.
2. 질문을 공식 사실, 플랫폼 정보, 경험 판단에 필요한 하위 검색으로 나눈다.
3. 지역의 한글 표기, 행정구역, 대표 지명과 질문 목적을 조합해 복수 검색어를 만든다.
4. 검색 결과에서 핵심 후보와 원문 URL을 수집한다.
5. 근거가 부족하면 같은 검색을 반복하지 않고 검색어 또는 출처 유형을 바꿔 제한적으로 재시도한다.
6. 충분성 기준을 충족하면 검색을 중단하고 근거 평가로 넘어간다.

재시도에는 횟수, timeout, 전체 지연 예산을 둔다. 한 출처의 장애가 전체 요청을 무기한 막아서는 안 된다.

### 성공 결과의 단기 재사용

- 동일하게 정규화된 질문과 동일한 여행 조건에는 최근 성공한 웹 리서치 결과를 짧은 시간 재사용할 수 있다.
- 현재 프로세스 메모리 캐시는 5분 TTL을 사용하며 최대 128개 항목을 보관한다.
- 캐시에는 출처 URL, 조사 결과, 후보와 최초 확인 시점을 함께 보존한다. 캐시 적중 시 확인 시점을 새로 만든 것처럼 갱신하지 않는다.
- 검색 실패와 출처 없는 결과는 캐시하지 않는다. 일시 장애가 반복 질문에 고착되어서는 안 된다.
- 지역, 동행자, 일정 등 여행 조건이 다르면 같은 문장이라도 별도 조사로 처리한다.

## 4. 출처 분류

| source_class | 예시 | 주 용도 | 단독 근거 제한 |
|---|---|---|---|
| `official` | 지자체, 공공기관, 장소 공식 홈페이지·SNS | 주소, 운영시간, 휴무, 요금, 예약, 행사 일정 | 공식 페이지도 오래됐을 수 있어 확인 시점 기록 |
| `platform` | 네이버·카카오 지도, 예약 서비스, 지역 매체 | 위치, 영업 상태, 메뉴, 예약 가능성, 대중 신호 | 플랫폼 정보가 공식 출처와 충돌하면 공식 최신 정보 우선 |
| `experience` | 블로그, 유튜브, SNS, 사용자 리뷰 | 분위기, 체감 혼잡도, 메뉴 경험, 동행자 적합성 | 운영 사실·폐업 여부의 단독 근거로 사용 금지 |
| `public_data` | 비짓제주, 위생등급, 주차장·정류장 내부 데이터 | 장소·분류·주소·접근성 교차확인 | 미수록을 현실의 부재로 해석 금지 |

### 출처 메타데이터

각 출처는 가능한 범위에서 다음 필드를 가진다.

```json
{
  "url": "https://example.com/place",
  "title": "문서 제목",
  "domain": "example.com",
  "source_class": "official",
  "published_at": "2026-07-01T00:00:00+09:00",
  "checked_at": "2026-07-11T15:00:00+09:00",
  "author_or_owner": "운영 주체",
  "claims": ["영업시간", "예약 필요"],
  "access_status": "ok"
}
```

게시일이 없으면 만들지 않고 `null`로 둔다. 검색 스니펫만 확인한 출처와 원문을 연 출처를 구분한다.

## 5. 주장과 근거 규칙

### 주장 유형

- `identity`: 장소명, 지점, 위치
- `operation`: 운영시간, 휴무, 예약, 가격, 메뉴
- `status`: 영업, 폐업, 이전, 임시 휴무
- `experience`: 분위기, 혼잡도, 맛의 경향, 동행자 적합성
- `popularity`: 반복 언급, 리뷰량, 최근 관심
- `recommendation`: 사용자 조건에 따른 비교와 선택

### 지원 기준

- 운영·상태 주장은 최신 공식 출처를 우선하고, 없으면 서로 독립적인 플랫폼 출처로 보완한다.
- 경험 주장은 복수 후기의 반복 패턴으로 요약하며 개인 경험을 보편적 사실로 확대하지 않는다.
- 인기 주장은 평점 하나가 아니라 출처 다양성, 언급 반복, 최근성 등을 함께 본다.
- 추천 주장은 사실 자체가 아니라 확인된 사실과 경험 신호를 사용자 조건에 적용한 판단이다. 선정 기준을 답변에 밝힌다.
- 주소, 시간, 가격 등 변동 정보는 답변에 확인 시점을 표시한다.

## 6. 근거 상태와 충돌 처리

각 핵심 주장은 다음 상태 중 하나를 가진다.

| status | 의미 | 답변 처리 |
|---|---|---|
| `supported` | 적합한 출처가 주장을 직접 지원 | 사실로 서술하고 인용 연결 |
| `partially_supported` | 일부만 확인되거나 출처 강도가 낮음 | 확인된 범위를 한정해 서술 |
| `conflicted` | 신뢰 가능한 출처끼리 내용이 다름 | 양쪽 내용과 기준 시점을 공개 |
| `inferred` | 여러 근거에서 합리적으로 도출한 판단 | “추천 판단”, “후기 경향”처럼 추론임을 표시 |
| `unsupported` | 근거를 확보하지 못함 | 단정하지 않고 제외하거나 미확인 표시 |

충돌 우선순위는 기계적으로 도메인만 보지 않는다. 기본 순서는 최신 공식 출처, 운영 주체의 최신 공지, 신뢰 가능한 플랫폼 최신 정보, 경험 출처다. 최신성이나 지점 일치 여부가 불명확하면 충돌을 해결한 것으로 처리하지 않는다.

## 7. 웹 리서치 상태

요청 단위 연구 상태는 다음과 같다.

| research_status | 정의 | 처리 |
|---|---|---|
| `sufficient` | 주요 추천과 변동 사실에 적합한 근거 확보 | 정상 답변 생성 |
| `partial` | 일부 후보나 일부 사실만 충분히 확인 | 확인된 범위로 답하고 부족한 항목 표시 |
| `conflicted` | 핵심 출처 충돌이 해결되지 않음 | 충돌을 중심 주의사항으로 표시 |
| `unavailable` | 검색 제공자 장애, 접근 실패 또는 재시도 후 usable source 없음 | 실패 범위와 이유를 밝히고 웹 답변으로 가장하지 않음 |

`unavailable`일 때 내부 공공데이터 후보를 웹 검색 결과인 것처럼 자동 반환하지 않는다. 사용자가 원하면 별도 내부 데이터 모드로 조회할 수 있다.

## 8. 내부 공공데이터 교차확인

웹 리서치와 후보 비교가 끝난 뒤, 선택한 지역과 후보에 관련 데이터가 있을 때만 실행한다.

| crosscheck_status | 의미 | 사용자 표현 |
|---|---|---|
| `matched` | 장소명·주소·분류 등 핵심 필드 일치 | `일치` |
| `mismatched` | 최신 웹 근거와 내부 값이 다름 | `불일치`와 다른 필드 표시 |
| `not_found` | 내부 데이터에서 매칭되지 않음 | `공공데이터 미확인` |
| `needs_review` | fuzzy match 또는 오래된 값으로 판정 불가 | `추가 확인 필요` |

기존 fallback 4분기는 내부 조회 진단에만 유지한다.

- `out_of_scope`: 내부 데이터의 지역·카테고리 범위 밖
- `coverage_gap`: 내부 데이터 커버리지에 없음
- `retrieval_miss`: 내부 데이터에 있을 가능성이 있으나 검색 실패
- `contradicted`: tombstone 또는 명시적 수정 근거와 충돌

`coverage_gap`과 `not_found`는 현실 세계에 장소가 없다는 뜻이 아니다.

## 9. 후보 비교와 추천

후보 순위는 단일 평점이나 내부 데이터 유무로 정하지 않는다. 다음 신호를 질문에 맞게 사용한다.

- 사용자 조건 적합성: 지역, 이동, 일정, 동행자, 음식 취향, 예산
- 근거 강도: 핵심 사실의 지원 상태와 출처 적합성
- 최신성: 운영·상태 정보의 확인 시점
- 경험 일관성: 독립적인 후기에서 반복되는 장단점
- 차별성: 후보끼리 선택 기준이 실제로 달라지는가
- 위험: 예약 필요, 휴무 가능성, 접근 난이도, 정보 충돌

“가장 맛집”은 절대 순위가 아니라 현재 질문 조건에서 먼저 검토할 후보로 해석한다. 답변에 선정 기준과 다른 선택이 더 나은 경우를 함께 설명한다.

### 후보 수와 점진적 탐색

- 최초 화면의 후보 수는 전체 후보 수나 대표 후보 수를 의미하지 않는다.
- 최초 후보를 일부만 보여주면 `전체 N곳 중 M곳`처럼 확인 범위를 명시하고, 중복 없는 다음 후보 탐색을 제공한다.
- cursor는 클라이언트가 내부 정렬 구조에 의존하지 않는 불투명 값으로 전달하며, 잘못된 cursor는 구조화된 오류로 반환한다.
- 단순 무작위 회전으로 다양성을 가장하지 않는다. 같은 조건과 cursor에는 결정론적인 결과를 반환한다.
- 내부 공공데이터의 전체 후보 수는 탐색 상태를 설명할 뿐 추천 품질이나 현실 세계의 장소 수로 표현하지 않는다.

## 10. 답변 계약

추천·탐색 답변은 질문 복잡도에 맞춰 다음 순서를 기본으로 한다.

1. **직접 결론**: 무엇을 먼저 추천하거나 어떤 지역부터 볼지 답한다.
2. **후보별 근거**: 각 후보가 조건에 맞는 이유와 근거를 설명한다.
3. **비교와 선택 기준**: 분위기, 이동, 메뉴, 일정 등 실제 차이를 보여준다.
4. **확인할 점**: 휴무, 예약, 가격 변동, 출처 충돌과 같은 위험을 적는다.
5. **출처**: 주장 가까이에 원문 링크를 연결하고 변동 정보의 확인 시점을 표시한다.
6. **공공데이터 교차확인**: 관련 내부 데이터가 있을 때만 마지막에 짧게 추가한다.

답변 길이를 3~5문장으로 제한하지 않는다. 후보가 여러 개면 스캔 가능한 목록이나 표를 사용할 수 있다. 단순 운영시간 질문은 확인된 답과 출처 중심으로 간결하게 답한다.

## 11. 모델 사용 범위

모델은 다음 작업을 할 수 있다.

- 대화 문맥에서 검색 조건과 후속 의도 해석
- 검색 계획과 검색어 후보 생성
- 확인한 원문에서 주장과 근거 추출
- 후보 비교와 사용자 맞춤 설명 조립
- 출처 충돌과 불확실성의 읽기 쉬운 표현

모델은 다음 작업을 해서는 안 된다.

- 출처에 없는 장소명, 주소, 시간, 가격, 메뉴 생성
- 열지 않은 페이지를 확인한 것처럼 표현
- 후기 한 건을 전체 경향으로 확대
- 검색 실패를 숨기기 위해 내부 후보를 웹 결과로 포장
- 인용이 지원하지 않는 주장을 해당 링크에 연결

모델이 없을 때는 구조화된 근거와 템플릿으로 제한 답변을 만들고 기능 제한을 명시한다.

## 12. 로그와 운영 지표

요청 로그는 개인정보를 최소화하면서 다음을 기록한다.

```json
{
  "intent": "restaurant_recommendation",
  "regions": ["gujwa"],
  "query_variants": ["구좌 맛집 공식", "구좌 현지 맛집 최근 후기"],
  "research_status": "sufficient",
  "source_counts": {"official": 2, "platform": 3, "experience": 4, "public_data": 1},
  "domains": ["visitjeju.net", "example.com"],
  "supported_claims": 12,
  "partial_claims": 2,
  "conflicts": 1,
  "retry_count": 1,
  "crosscheck": {"matched": 2, "not_found": 1},
  "latency_ms": 2400
}
```

핵심 품질 지표는 검색 성공률만이 아니다.

- 핵심 주장 지원률
- 인용-주장 일치율
- 공식·플랫폼·경험 출처 적합성
- 변동 정보 최신성 충족률
- 후속 문맥 유지율
- 검색 실패의 투명한 노출률
- 답변 완결성
- 공공데이터 교차확인 오표현률

## 13. 필수 회귀 시나리오

- “구좌에서 가볼 만한 지역”에 웹 검색 기반 지역 비교와 출처가 나온다.
- 이어서 “구좌의 오름들은?”이라고 물으면 구좌와 웹 의도를 유지해 다시 조사한다.
- 이어서 “가장 맛집은?”이라고 물으면 공공데이터 후보 수가 아니라 선정 기준과 웹 근거가 있는 추천을 제공한다.
- 웹 검색이 실패하면 공공데이터 후보 목록으로 자동 전환하지 않는다.
- 공식 운영시간과 후기의 시간이 다르면 최신 공식 정보와 충돌을 표시한다.
- 내부 데이터에 없는 웹 후보는 `공공데이터 미확인`으로 표시하되 추천에서 자동 제거하지 않는다.
- 사용자가 후보를 제외하면 후속 검색과 추천에서 유지한다.
- 같은 질문과 여행 조건을 짧은 시간 안에 반복하면 최근 성공 결과를 재사용하되, 실패 결과는 재사용하지 않는다.
