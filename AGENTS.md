# AGENTS.md - Pack Your Jeju 작업 헌법

이 문서는 Codex와 모든 개발 에이전트가 이 저장소에서 지켜야 하는 최상위 규칙이다. 세부 문서가 이 헌법과 충돌하면 이 문서를 우선하고, 충돌한 문서를 함께 고친다.

## 프로젝트 정체성

- **Pack Your Jeju**는 실제 사용자가 제주 여행을 조사하고 선택하도록 돕는 운영형 서비스다.
- **하루방**은 지역, 일정, 동행자, 취향과 대화 맥락을 이해하고 웹을 조사해 비교 가능한 답을 만드는 제주 여행 리서치 에이전트다.
- 제품의 목표는 내부 데이터 후보 수를 보여주는 것이 아니라 사용자가 다음 행동을 결정할 수 있게 하는 것이다.
- 핵심 철학은 **충분히 조사하고, 근거를 연결하고, 불확실성을 숨기지 않는다**이다.

## 권위와 우선순위

1. 이 문서가 제품·에이전트 작업 원칙의 최상위 권위다.
2. `DECISIONS.md`는 결정 이력과 현재 유효한 결정을 기록한다.
3. `TRUST_ENGINE.md`는 검색, 출처 평가, 교차확인, 답변 조립의 권위 스펙이다.
4. 기존 문서가 해커톤, 공공데이터 전용 추천, 멀티턴 금지, 짧은 답변을 전제하면 폐기된 정책으로 취급한다.

## 절대 규칙

### 1. 웹 리서치가 기본이다

- 여행지 추천, 맛집, 카페, 오름, 관광지, 일정, 운영시간, 휴무, 가격, 예약, 행사, 혼잡도처럼 조사 가능한 질문은 웹 검색을 기본 경로로 처리한다.
- 한 번의 검색 결과가 빈약하면 검색어, 지역 표현, 출처 유형을 바꿔 재검색한다.
- 검색 결과 제목이나 요약문만 보고 답하지 않는다. 핵심 주장에 필요한 원문을 확인한다.
- 사용자가 명시적으로 내부 공공데이터만 요청한 경우에만 웹 검색보다 내부 조회를 우선할 수 있다.

### 2. 사실을 만들지 않는다

- 장소명, 주소, 운영시간, 휴무, 요금, 메뉴, 행사 일정 등 사실 정보는 확인한 웹 출처 또는 내부 데이터에 근거해야 한다.
- LLM의 기억이나 일반 상식을 최신 사실의 근거로 사용하지 않는다.
- 근거가 부족한 내용을 그럴듯하게 채우지 않는다. 확인한 범위, 확인하지 못한 범위, 추론을 구분한다.
- 검색 결과에 없는 장소를 답변의 완성도를 위해 임의로 추가하지 않는다.

### 3. 출처의 역할을 구분한다

- **공식 출처**: 지자체, 공공기관, 장소 공식 홈페이지와 공식 SNS. 주소, 운영시간, 휴무, 요금, 예약 같은 운영 사실에 우선한다.
- **플랫폼 출처**: 네이버·카카오 지도, 예약 플랫폼, 지역 매체. 위치, 영업 상태, 메뉴, 대중적 반응을 보완한다.
- **경험 출처**: 블로그, 유튜브, SNS, 사용자 리뷰. 분위기, 체감 혼잡도, 메뉴 경험, 동행자 적합성과 같은 경험적 판단에 사용한다.
- **내부 공공데이터**: 웹 리서치 뒤 장소 존재, 분류, 주소, 접근성 등의 일치 여부를 교차확인한다.
- 후기성 출처 하나를 운영시간이나 폐업 여부 같은 변동 사실의 단독 근거로 쓰지 않는다.

### 4. 최신성과 충돌을 다룬다

- 변동 가능한 정보는 가능한 한 최신 출처를 확인하고 답변에 확인 시점을 표시한다.
- 출처가 충돌하면 최신 공식 출처를 우선한다.
- 공식 출처가 없거나 충돌이 해결되지 않으면 서로 다른 내용을 함께 제시하고 추가 확인 필요를 밝힌다.
- “없다”, “폐업했다”, “최고다”와 같은 강한 단정은 그 강도에 맞는 근거가 있을 때만 허용한다.

### 5. 멀티턴 문맥을 유지한다

- 후속 질문은 직전 대화의 지역, 일정, 동행자, 취향, 후보, 제외 조건, 비교 기준과 웹 검색 의도를 이어받는다.
- “오름은?”, “그중 한 곳”, “다른 맛집” 같은 짧은 질문을 독립 질문으로 초기화하지 않는다.
- 사용자가 조건을 바꾸면 새 조건을 우선하고, 충돌하지 않는 이전 조건은 유지한다.
- 이미 제외했거나 거절한 후보를 이유 없이 다시 추천하지 않는다.

### 6. 실패를 숨기거나 다른 데이터로 바꾸지 않는다

- 웹 검색이 실패하거나 근거가 빈약하면 검색을 재시도하고, 그래도 부족하면 무엇을 확인하지 못했는지 구체적으로 밝힌다.
- 웹 검색 실패를 공공데이터 후보 목록으로 조용히 대체하지 않는다.
- 외부 검색 장애, 페이지 접근 실패, 출처 부족은 서로 구분해 기록하고 사용자에게 필요한 범위에서 설명한다.
- 내부 데이터의 `fallback_reason`은 내부 조회 상태를 설명할 뿐, 현실 세계에 정보나 장소가 없다는 증거가 아니다.

### 7. 공공데이터는 마지막 교차확인이다

- 공공데이터는 추천 후보를 제한하는 1차 검색 풀이 아니다.
- 사용자가 선택한 지역과 관련된 내부 데이터가 있을 때만 답변 마지막에 `공공데이터 교차확인`으로 요약한다.
- 교차확인 결과는 `일치`, `불일치`, `공공데이터 미확인`, `추가 확인 필요`로 표현한다.
- 공공데이터에 없다는 이유만으로 웹에서 충분히 확인된 후보를 제거하지 않는다.
- 공공데이터 후보 수를 답변의 핵심 결론이나 품질 지표로 사용하지 않는다.

### 8. 답변은 선택에 충분해야 한다

- 답변 길이는 고정하지 않고 질문의 복잡도와 사용자의 요청에 맞춘다.
- 추천 질문에는 원칙적으로 직접적인 결론, 후보별 이유, 비교 기준, 주의점, 출처를 포함한다.
- 짧은 질문에도 장소명과 후보 수만 던지지 않는다. 사용자가 왜 그 후보를 선택해야 하는지 설명한다.
- 단순 사실 질문에는 불필요한 장문을 붙이지 않는다.
- “가장 맛집”처럼 객관적 정답이 없는 질문은 반복 언급, 최근성, 출처 다양성, 사용자 조건을 기준으로 해석하고 선정 기준을 밝힌다.
- 내부 구현 용어인 DB, RAG, tool, fallback을 사용자 답변의 중심 언어로 쓰지 않는다.

### 9. 인용은 주장과 연결한다

- 출처 링크는 해당 출처가 뒷받침하는 주장 가까이에 배치한다.
- 검색 결과 페이지가 아니라 가능한 한 원문 페이지를 연결한다.
- 출처를 많이 나열하는 것보다 핵심 주장에 직접 맞는 출처를 우선한다.
- 후기나 SNS를 요약할 때 개인 경험을 전체 방문객의 보편적 사실로 확대하지 않는다.

### 10. 장애 상황에서도 정직하게 동작한다

- LLM, 검색 제공자, 특정 웹사이트 또는 내부 DB 하나가 실패해도 서버는 실패 원인을 구조화해 반환해야 한다.
- `OPENAI_API_KEY`가 없을 때는 확인한 검색·내부 데이터만 템플릿으로 조립하고, 모델이 필요한 기능은 제한 상태를 명시한다.
- 캐시를 사용하면 출처 URL과 수집 또는 확인 시점을 함께 보존한다.
- 오래된 캐시를 최신 정보처럼 표시하지 않는다.

## 검색과 답변 기본 흐름

```text
사용자 질문과 대화 문맥
  -> 의도·지역·조건·제외 항목 해석
  -> 검색 계획 및 검색어 확장
  -> 공식·플랫폼·경험 출처 수집
  -> 원문 확인, 최신성·출처 역할·주장 일치 평가
  -> 사용자 조건에 맞춰 후보 비교
  -> 선택 지역 내부 공공데이터 교차확인
  -> 결론·이유·주의점·출처·교차확인 조립
  -> 검색 및 품질 로그 기록
```

## 구현 원칙

- Python 3.11+, FastAPI, SQLAlchemy와 현재 프론트 스택을 유지하되 실제 요구에 필요한 확장은 허용한다.
- 기존 모듈 경계를 먼저 따르고, 필요한 경우에만 검색 계획, 출처 정규화, 주장 근거 연결을 위한 작은 추상화를 추가한다.
- 구조화된 검색 결과와 출처 메타데이터는 구조화된 타입과 파서로 처리한다.
- 외부 호출에는 timeout, 제한된 재시도, 오류 분류를 둔다.
- 모델과 검색 공급자 선택은 현재 유효한 운영 결정과 환경 설정을 따른다. 특정 모델명을 헌법에 영구 고정하지 않는다.
- 원본 Pack Your Moment 자산을 수정할 때는 소유권과 배포 영향을 먼저 확인한다.

## 작업 흐름

- 새 결정을 만들거나 기존 결정을 뒤집을 때 `DECISIONS.md`를 함께 갱신한다.
- 검색, 출처, 신뢰, 답변 계약 변경 전 `TRUST_ENGINE.md`를 읽고 코드와 문서를 함께 맞춘다.
- 행동 변경은 실패 테스트를 먼저 추가하고, 실제 검색 성공·부분 성공·충돌·실패 경로를 검증한다.
- 기존 사용자 변경과 미추적 파일을 보존한다.
- 커밋 메시지는 `[역할] 요약` 형식을 사용한다.

## 운영 품질 체크리스트

- [ ] 추천성 질문에서 실제 웹 검색이 수행되는가
- [ ] 질문 성격에 맞는 공식·플랫폼·경험 출처가 사용되는가
- [ ] 출처 링크가 본문의 주장과 직접 연결되는가
- [ ] 변동 정보에 최신 확인 시점이 있는가
- [ ] 후속 질문에서 지역·후보·제외 조건·웹 검색 의도가 유지되는가
- [ ] 웹 검색 실패가 공공데이터 후보로 자동 대체되지 않는가
- [ ] 답변에 결론·추천 이유·비교·주의점이 충분한가
- [ ] 공공데이터 비교가 마지막 보조 섹션으로만 제공되는가
- [ ] 출처 충돌과 추론이 명확히 표시되는가
- [ ] 외부 서비스 장애 시 오류를 숨기지 않고 제한 상태로 동작하는가
