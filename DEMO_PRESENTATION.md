# DEMO_PRESENTATION.md — 발표·데모·우승 전략

> 목표: 20개 팀 (워싱턴대 팀 포함) 중 **1등**.

## 1. 20팀 경쟁 구도에서의 포지셔닝

예상되는 타 팀 분포: 제주 여행 RAG 챗봇 다수 + 이미지/멀티모달 데모 + 일부 고급 아키텍처(해외 팀). 이 구도에서 우리가 이기는 논리는 세 겹이다:

1. **카테고리 이탈**: 우리는 "챗봇"이 아니라 "여행 준비 프로덕트 + 신뢰 엔진". 첫 화면부터 다르게 보인다.
2. **흉내 불가 요소**: 이미 존재하던 실사용 앱에 이식했다는 사실. 4일 만에 만든 게 아니라 4일 만에 **진화시킨** 것.
3. **엔지니어링 깊이의 증거**: fallback 4분기·배지 체계·eval 게이트 — 말이 아니라 라이브로 보여줌. 기술 심사(해외 팀 대비)에서 밀리지 않는 방어선.

발표에서 반복할 세 단어: **"출처를 붙인다 / 최신만 답한다 / 안 지어낸다."**

## 2. 3막 구성 (발표 시간 기준 배분 — 총 10분 가정, 실제 시간에 비례 조정)

### 오프닝 (1.5분) — 킥3 "1,686건의 증거"
> "비짓제주 — 제주 공식 관광 포털입니다. 여기에 이용자들이 직접 제출한 정보 수정요청이 1,686건 쌓여 있습니다. 저희가 전부 분석했습니다. 폐업, 운영시간 변경, 주소 오류... **공식 데이터조차 이 속도로 낡습니다.** 개인 블로그는 어떨까요?"

- 슬라이드 1장: 수정요청 유형 분포 (Day1~2 PM 분석 결과 숫자 삽입)
- 이어서: "그래서 저희는 짐 싸기 앱에 신뢰 엔진을 심었습니다."

### 본편 (5분) — 제품 데모 (라이브)
1. **팩 만들기**: 애월 · 3일 · 부모님과 · 힐링 → 오름/맛집/노을 카드 선택 → 팩 생성
   - 보여줄 것: 🔵 배지 + 출처, 🚗 주차장 배지, 부모님 필터("난이도 미확인" 명시)
2. **정직함 장면 ①**: 감귤 체험 카드 선택 → "시즌이 아닙니다" (G05) — *"저희는 7월에 감귤 체험을 지어내지 않습니다."*
3. **정직함 장면 ②**: 우도 × 카페 → coverage_gap 문구 (G11) — *"없다고 단언하지 않습니다. '저희 데이터 기준'이라고만 말합니다."*
4. **킥1 — AI가 AI를 팩트체크 (하이라이트)**: 폐업 가게가 낀 리뷰(G14, 사전 확정)를 /verify에 붙여넣기 → 문장별 ○/⚠️/× 판정
   > "이 리뷰의 세 번째 문장 — 이 식당, 폐업이 **확인됩니다.** 근거는 여기 있습니다."
5. **아키텍처 1장**: 파이프라인 그림 + *"근거율 게이트를 통과하지 못하면 배포되지 않습니다"* (eval exit(1) 언급)

### 클로징 (1.5분) — 킥4 "QR + 라이브 대시보드"
> "지금 직접 만들어보세요." (QR) → 발표 마무리 멘트 후 /admin/metrics 화면:
> "방금 여러분이 만든 팩들입니다. 요청 N건, verified 비율 N%, fallback N건. **저희는 발표하는 동안에도 운영 중이었습니다.**"

마지막 슬라이드 — 다음 단계: 지역 팩 확장(부산·도쿄), 여행 일정 전체 검증, 다국어(비짓제주 en/cn/jp locale 활용).

## 3. 예상 Q&A 카드

| 질문 | 답변 골자 |
|---|---|
| "근거없음과 정보부족을 어떻게 구별하나?" | fallback 4분기 설명. "저희는 '모른다'를 네 가지로 구분합니다. 범위 밖 / 데이터에 없음 / 못 찾음(재시도 후) / 반증 있음. 어떤 경우에도 '없다'고 단언하지 않고 '저희 데이터 기준'을 명시합니다. 사유는 전부 로깅되어 데이터 보강 우선순위가 됩니다." |
| "일반 GPT와 뭐가 다른가?" | "GPT는 그럴듯하게 지어냅니다. 저희 LLM은 사실을 생성할 권한이 없습니다 — DB가 사실을 대고, LLM은 조립만 합니다. 방금 데모에서 보셨듯 지어낼 바엔 '확인 불가'라고 말합니다." |
| "데이터가 오래되면?" | "모든 항목에 valid_until이 스키마 레벨로 강제됩니다. 못 채우면 DB에 들어가지도 못합니다. 감귤 데모가 그 증거입니다." |
| "왜 제주만?" (D-16) | "지역 팩 아키텍처입니다. 데이터 어댑터만 갈아끼우면 부산 팩, 도쿄 팩이 됩니다. 저희는 모든 곳을 아는 척하는 대신, 제주부터 완벽하게 하기로 했습니다." |
| "커버리지가 좁은데 실용성이 있나?" | "coverage_gap 로그가 곧 데이터 보강 우선순위입니다. 어디가 부족한지 시스템이 스스로 압니다." + admin no-info 지표 언급 |
| "품질을 어떻게 보장하나?" | "골든셋 15문항 게이트. 그중 4문항은 '답하지 않아야 정답'인 케이스입니다. 미달이면 exit(1) — 배포가 물리적으로 차단됩니다." (Day4 캡처 제시) |
| "게이트가 실패하면 어떻게 되나?" | eval 실패 로그 캡처 1장으로 답변 (킥2 대체) |

## 4. 데모 리스크 관리

- 킥1·팩 생성 전 구간 **스크린 녹화 백업** (Day4 오후 필수)
- 발표장 네트워크 불안 대비: 로컬 실행 환경 노트북 1대 별도 준비
- QR(킥4)은 발표장 Wi-Fi에서 사전 접속 테스트. 실패 시 클로징은 대시보드 화면만으로 진행
- 심사위원이 QR로 이상한 입력을 해도 우리 시스템은 "확인 불가"로 답하는 게 콘셉트 — 어시스트가 된다

## 5. 심사 항목별 자기 점검 (리허설 때 체크)
