# DECISIONS.md - 확정 의사결정 로그

이 문서는 결정 이력과 현재 유효한 제품 결정을 함께 보존한다. `active` 결정만 현재 구현을 구속하며, `superseded` 결정은 역사적 맥락으로만 남는다. 결정을 번복할 때는 기존 항목을 지우지 않고 대체 결정과 연결한다.

## 상태 표기

- `active`: 현재 유효하며 구현과 운영이 따라야 한다.
- `superseded by D-XX`: 새 결정으로 대체되었으며 더 이상 구현 근거로 사용할 수 없다.
- `historical`: 당시 맥락을 설명하지만 현재 제품 범위를 제한하지 않는다.

## 현재 제품 방향

**D-17. `active` Pack Your Jeju는 해커톤 데모가 아니라 실제 운영형 제주 여행 서비스다.**

- 결정: 완성도와 신뢰성을 유지하되 “4일 안에 구현 가능한 기능”을 영구적인 제품 제한으로 사용하지 않는다.
- 근거: 하루방은 실제 사용자의 탐색과 선택을 지속적으로 지원해야 한다.
- 파급: 멀티턴, 웹 검색, 운영 관측, 품질 평가와 검색 공급자 개선을 정상적인 제품 범위로 본다.

**D-18. `active` 하루방의 추천과 최신 정보 질문은 웹 리서치 우선으로 처리한다.**

- 결정: 지역 추천, 맛집, 카페, 오름, 관광지, 일정, 운영시간, 휴무, 가격, 예약, 행사 질문은 웹 검색을 기본 경로로 사용한다.
- 근거: 내부 공공데이터만으로는 최신성, 경험 정보, 비교 맥락과 실제 선택에 필요한 설명을 제공할 수 없다.
- 파급: 검색 실패 시 공공데이터 목록으로 자동 전환하지 않고 재검색과 투명한 실패 처리를 적용한다.

**D-19. `active` 출처는 공식·플랫폼·경험·내부 공공데이터의 네 역할로 평가한다.**

- 공식 출처는 운영 사실의 최우선 근거다.
- 플랫폼 출처는 위치, 영업 상태, 메뉴와 대중적 반응을 보완한다.
- 블로그, 유튜브, SNS, 사용자 리뷰는 분위기, 체감 혼잡도, 메뉴 경험과 동행자 적합성 판단에 사용할 수 있다.
- 내부 공공데이터는 웹 리서치가 끝난 뒤 일치 여부를 확인하는 보조 자료다.
- 파급: 모든 출처를 같은 신뢰도로 취급하지 않고 주장 유형별로 적합성을 평가한다.

**D-20. `active` 공공데이터는 추천 원천이 아니라 마지막 교차확인 자료다.**

- 결정: 선택 지역에 관련 내부 데이터가 있을 때 답변 마지막에 `공공데이터 교차확인`으로 일치, 불일치, 미확인, 추가 확인 필요를 표시한다.
- 근거: 공공데이터의 검증 가치는 유지하면서도 제한된 커버리지가 추천 품질을 지배하지 않게 한다.
- 파급: 공공데이터 후보 수를 핵심 답변이나 품질 지표로 사용하지 않는다.

**D-21. `active` 하루방은 멀티턴 문맥을 제품 기능으로 유지한다.**

- 결정: 지역, 일정, 동행자, 취향, 이전 후보, 제외 조건, 비교 기준과 웹 검색 의도를 후속 질문에 전달한다.
- 근거: “오름은?”, “그중 하나”, “다른 맛집” 같은 실제 대화는 직전 문맥 없이는 올바르게 처리할 수 없다.
- 파급: 현재 메시지만 보는 라우팅을 금지하고 대화 기반 검색 계획을 테스트한다.

**D-22. `active` 답변 길이는 고정하지 않고 의사결정에 필요한 깊이에 맞춘다.**

- 결정: 추천 질문에는 결론, 후보별 이유, 비교, 주의점, 출처를 기본 구성으로 사용한다. 단순 사실 질문은 간결하게 답한다.
- 근거: 3~5문장 강제는 복합 여행 질문의 품질을 떨어뜨리고 후보 나열형 답변을 만든다.
- 파급: 답변 품질 평가는 글자 수보다 완결성, 근거 연결, 사용자 조건 반영을 본다.

**D-23. `active` 웹 검색 실패와 출처 충돌을 숨기지 않는다.**

- 결정: 검색 결과가 빈약하면 검색어와 출처 유형을 바꿔 제한적으로 재시도한다. 그래도 부족하면 실패 범위와 충돌 내용을 밝힌다.
- 근거: 다른 데이터로 조용히 대체하면 사용자는 답변이 어떤 근거로 만들어졌는지 알 수 없다.
- 파급: 웹 연구 상태를 `sufficient`, `partial`, `conflicted`, `unavailable`로 기록한다.

**D-24. `active` 품질 평가는 근거가 연결된 유용성을 중심으로 한다.**

- 평가 항목: 검색 수행, 출처 다양성과 적합성, 최신성, 인용 정확성, 주장 지원률, 후속 문맥 유지, 답변 완결성, 공공데이터 교차확인의 올바른 위치.
- 비평가 항목: 내부 후보 수 자체, 공공데이터 커버리지만으로 계산한 추천 성공률.

**D-25. `active` 사실 생성 금지는 유지하되 근거를 DB로 한정하지 않는다.**

- 결정: 확인한 웹 원문과 내부 데이터를 사실 근거로 허용한다. 모델은 근거를 해석, 비교, 요약할 수 있지만 없는 사실을 추가할 수 없다.
- 파급: 모든 변동 사실은 출처와 확인 시점을 추적하고, 추론은 사실과 구분한다.

## 대체된 제품·해커톤 결정

**D-01. `historical` 기존 Pack Your Moment 연동으로 시작한다.**

- 초기 완주 가능성을 위한 결정이었다. 현재 서비스의 UI 확장이나 신규 화면을 금지하지 않는다.

**D-02. `historical` 제주 특화 에디션으로 시작한다.**

- 제주 전문성은 유지한다. “아이펠톤이 제주 한정”이라는 근거는 현재 제품 범위를 제한하지 않는다.

**D-03. `active` 원본 Pack Your Moment와 Pack Your Jeju의 소유권 경계를 보호한다.**

- 원본 자산을 수정해야 할 때는 영향과 역이식 계획을 명시한다.

**D-04. `superseded by D-17` 확장안은 발표의 다음 단계로만 둔다.**

- 4일 완주를 위한 제한이었으며 운영 제품에는 적용하지 않는다.

## 대체된 신뢰 엔진 결정

**D-05. `superseded by D-25` LLM은 DB 조회 근거만 조립한다.**

- 사실 생성 금지는 유지한다. 근거 범위는 확인한 웹 출처와 내부 데이터로 확장됐다.

**D-06. `active with narrowed scope` 내부 데이터 fallback은 4분기로 구분한다.**

- `out_of_scope`, `coverage_gap`, `retrieval_miss`, `contradicted`는 내부 데이터 조회 상태에만 사용한다.
- 웹 전체의 정보 존재 여부나 추천 가능성을 이 값으로 판단하지 않는다.

**D-07. `superseded by D-19 and D-23` 공공데이터 검증 배지를 중심으로 신뢰를 표시한다.**

- 새 체계는 출처 역할, 최신성, 주장 지원과 충돌 상태를 함께 표시한다. 기존 배지는 내부 교차확인 결과로 제한할 수 있다.

## 대체된 데이터 결정

**D-08. `superseded by D-19` 식당 추천에 카카오·네이버 리뷰를 사용하지 않는다.**

- 리뷰와 SNS는 경험 근거로 사용할 수 있다. 운영 사실의 단독 근거 또는 평점 하나만으로 최고를 단정하는 근거로는 사용할 수 없다.

**D-09. `historical` 교통은 접근성 검증만 구현한다.**

- 당시 구현 범위다. 현재 하루방은 확인 가능한 웹 근거가 있으면 이동·접근 정보를 설명할 수 있으며, 실시간 경로 정확성을 보장하지 않는다.

**D-10. `historical` Google Maps를 연동하지 않는다.**

- 특정 공급자 금지는 해제한다. 공급자 도입은 비용, 약관, 품질과 개인정보 영향을 별도 결정한다.

**D-11. `active` 콘텐츠 수정요청 데이터는 신뢰 하향과 내부 교차확인 신호로 사용한다.**

## 기술 결정

**D-12. `superseded by D-26` LLM을 특정 모델 하나에 영구 고정한다.**

**D-26. `active` 모델과 검색 공급자는 운영 품질과 비용 기준으로 설정한다.**

- 애플리케이션에는 명시적인 기본값을 두되 헌법이 모델 업그레이드를 금지하지 않는다.
- 공급자 변경 시 골든셋과 실제 검색 품질 평가를 통과해야 한다.

**D-27. `active` 최초 후보 수는 탐색 범위를 제한하지 않는다.**

- 결정: 화면의 최초 후보는 비교 가능한 소수로 제한할 수 있지만, 전체 후보 수와 중복 없는 점진적 탐색 경로를 함께 제공한다.
- 근거: 최초 5개는 대표성이나 전체 데이터 접근을 보장하지 않으며, 고정 목록만 반복하면 사용자의 선택권과 후보 비교 품질이 낮아진다.
- 파급: `/pack`의 기존 최초 후보 계약은 유지하고, 순간별 전체 개수·다음 cursor·`/pack/candidates`를 하위 호환 필드와 API로 제공한다. 후보 순위는 별도 결정에서 사용자 조건·근거·최신성·차별성 기준으로 개선한다.

**D-28. `active` 사용자 플랜과 방문 피드백은 공공데이터를 덮어쓰지 않는 근거 원장으로 축적한다.**

- 결정: 공공데이터, 하루방 웹검색, 사용자 직접입력 장소를 출처가 보존된 플랜 항목으로 통합하고, 방문 피드백은 별도 현장 신뢰 신호로 즉시 집계한다.
- 공공데이터 원본은 수정하지 않는다. 중요 사실의 보정은 반복 제보, 웹 재검증, 운영자 승인을 거쳐 버전이 있는 보정 정보로 기록한다.
- 비회원은 플랜을 작성할 수 있지만 저장·공유·피드백 제출은 로그인 후 제공한다.
- 사용자 직접입력 장소는 개인 플랜에 즉시 추가하고, 장소 존재·주소·중복을 검증한 뒤 `커뮤니티 확인 후보`로 공개한다.
- 위치·사진·영수증은 선택 증빙이며, 증빙 수준은 현장 신뢰와 검토 우선순위에만 영향을 준다. 높은 가중치만으로 자동 승인하지 않는다.
- 근거: 실제 방문 경험을 다음 사용자의 선택 근거로 연결하면서도 허위 제보가 공공데이터와 하루방 답변을 오염시키는 것을 막아야 한다.
- 상세 설계: `docs/plans/2026-07-14-plan-feedback-trust-loop-design.md`

**D-29. `active` 플랜 신뢰 순환의 서버 작업은 Railway에 유지하고 회원·권한·증빙 저장은 Supabase로 단계 도입한다.**

- 결정: FastAPI, 하루방 웹 리서치, 장소 판별, 신뢰 계산, PDF와 장시간 작업은 Railway에 유지한다. 회원, 사용자별 권한, 장기 PostgreSQL 데이터, 비공개 증빙 저장은 Supabase Auth·PostgreSQL·Storage를 사용한다.
- 기존 Railway PostgreSQL은 확장 호환성, 연결 풀, 지연, 백업·복구, 롤백을 검증한 뒤 단계적으로 전환한다.
- 파급: 저장소 전환 전 도메인 모델과 API 계약을 먼저 구축하고, 특정 공급자 장애가 피드백 원문 유실로 이어지지 않게 한다.

**D-13. `superseded by D-17` 운영 관측과 배포 인프라를 해커톤 범위 밖으로 둔다.**

- 실제 필요와 운영 규모에 맞춰 단계적으로 도입한다. 불필요한 대규모 인프라는 여전히 피한다.

**D-14. `superseded by D-24` 골든셋을 15문항으로 고정한다.**

- 골든셋은 핵심 시나리오를 지속적으로 확장하고 회귀 유형별로 관리한다.

## 발표 이력

**D-15. `historical` 아이펠톤 발표는 3막으로 구성한다.**

**D-16. `historical` 제주는 첫 번째 지역 팩으로 설명한다.**

- 발표 이력은 남기되 현재 하루방의 검색과 답변 품질을 제한하지 않는다.
